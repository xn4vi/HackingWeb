# 🌐 Web Cache Deception mediante delimitadores en la ruta

## 📄 Descripción del laboratorio

La página:

```
/my-account
```

muestra la **API key del usuario autenticado**.

Este laboratorio presenta una variante más sutil de **Web Cache Deception**, basada en diferencias de interpretación de rutas entre:

* El **servidor de aplicación**
* El **sistema de caché frontal** (Varnish, Akamai, Cloudflare, etc.)

El objetivo del laboratorio es:

* Encontrar **delimitadores que el backend ignore o normalice**
* Forzar que la **caché trate la ruta como contenido estático**
* Cachear la página `/my-account` de **Carlos**
* Recuperar su **API key** desde una URL pública cacheada

Credenciales proporcionadas:

```
wiener:peter
```



## 📚 Teoría

El núcleo del ataque es una **discrepancia en el parsing de la URL** entre el backend y el sistema de caché.

### 📌 Diferencias de interpretación

El **backend**:

* Ignora o normaliza ciertos delimitadores en la ruta.
* Trata rutas como:

```
/my-account;evil.js
```

como si fueran simplemente:

```
/my-account
```

El **sistema de caché frontal**, en cambio:

* Considera esos delimitadores **parte del path**
* Detecta la extensión `.js`
* Almacena la respuesta como **contenido estático**

### 📌 Resultado del engaño

Cuando una víctima autenticada visita una URL manipulada:

1. El backend devuelve la página dinámica `/my-account`.
2. La respuesta contiene la **API key de la víctima**.
3. La caché almacena esa respuesta bajo la **ruta manipulada**.
4. Cualquier usuario puede acceder posteriormente a ese contenido cacheado.

Este patrón es común en aplicaciones que:

* Usan **parámetros matriciales** (`;param=value`)
* Normalizan rutas agresivamente
* Delegan el cacheo en **CDNs que clasifican contenido por extensión**



## 📝 Práctica

### 1️⃣ Análisis inicial

Iniciamos sesión con las credenciales:

```
wiener:peter
```

Accedemos a:

```
/my-account
```

Observamos que la página muestra **nuestra API key**.

### 2️⃣ Interceptar la petición base

Interceptamos la petición:

```http
GET /my-account
```

<img width="1260" height="564" alt="image" src="https://github.com/user-attachments/assets/4bfdfde7-437d-4224-a929-a7daafba2f94" />

Observaciones:

* La respuesta es **dinámica**
* Contiene **información sensible**
* No aparecen cabeceras de caché agresivas

Esto indica que la vulnerabilidad **no está en la página en sí**, sino en **cómo se interpreta la ruta**.



### 3️⃣ Buscar delimitadores válidos

El laboratorio proporciona una lista de posibles delimitadores.

Utilizamos **Burp Intruder** para probarlos.

Posición del payload:

```
/my-account§§
```

Lista de delimitadores probados:

```
!, ", #, $, %, &, ', (, ), *, +, ,, -, ., /, :, ;, <, =, >, ?, @, [, ], ^, _, `, {, |, }, ~
```

También probamos sus versiones **codificadas en URL (%XX)**.

<img width="1520" height="443" alt="image" src="https://github.com/user-attachments/assets/1f7ac60e-0941-43ad-8c23-d5a34b412e8b" />



### 4️⃣ Analizar los resultados

Tras ejecutar el ataque:

* La mayoría de payloads devuelven **errores o 404**.
* Dos delimitadores devuelven **200 OK** y muestran nuestra API key:

```
?
;
```

Esto indica que el backend:

* **Ignora o normaliza esos delimitadores**
* Sigue procesando la ruta como:

```
/my-account
```

### 5️⃣ Construir la URL maliciosa

Elegimos uno de los delimitadores válidos, por ejemplo:

```
/my-account;evil.js
```

<img width="1033" height="225" alt="image" src="https://github.com/user-attachments/assets/c0f731b4-6b52-4680-9143-549a1080f82d" />

Comportamiento observado:

* El servidor devuelve la página real `/my-account`.
* La caché detecta la extensión `.js` y la trata como **contenido estático**.

Creamos un exploit en el **Exploit Server** que redirija automáticamente a la víctima.

```html

```

3. La respuesta contiene **su API key**.
4. La caché almacena esa respuesta bajo esa ruta manipulada.
