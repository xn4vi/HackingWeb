# 1. WCD Manipulación Rutas

## 🌐 Web Cache Deception mediante manipulación de rutas

### 📄 Descripción del laboratorio

Este laboratorio permite a los usuarios autenticados ver su **API key** en la página:

```
/my-account
```

La aplicación utiliza un **sistema de caché frontal** (por ejemplo, Varnish, Akamai o Cloudflare) para almacenar contenido considerado estático.

El problema es una **mala separación entre rutas dinámicas y estáticas**, lo que permite que contenido sensible generado para usuarios autenticados sea **almacenado en caché**.

El objetivo del laboratorio es:

* Engañar a la caché para que almacene una **respuesta dinámica**
* Forzar que se cachee la página `/my-account` de **Carlos**
* Recuperar su **API key** desde una URL pública cacheada

***

### 📚 Teoría

La vulnerabilidad aparece por un **desacoplamiento entre el backend de la aplicación y el sistema de caché**.

#### 📌 Comportamiento del servidor

El servidor de aplicación trata rutas como:

```
/my-account
/my-account/anything.js
```

como **la misma página dinámica**, ignorando la extensión final.

#### 📌 Comportamiento del sistema de caché

El sistema de caché, en cambio:

* Considera que rutas que terminan en `.js`, `.css`, `.png`, etc. son **contenido estático**
* Las **cachea agresivamente**, incluso si contienen información sensible

Esta discrepancia permite que una respuesta dinámica personalizada termine **almacenada en caché pública**.

#### 📌 Web Cache Deception (WCD)

El ataque de **Web Cache Deception** funciona de la siguiente manera:

1. La víctima autenticada visita una **ruta manipulada con extensión estática**
2. El servidor devuelve **contenido dinámico personalizado**
3. La caché almacena esa respuesta creyendo que es un recurso estático
4. El atacante accede a la misma ruta y obtiene el **contenido cacheado**

Ejemplo de ruta manipulada:

```
/my-account/evil.js
```

Este tipo de vulnerabilidad es común cuando:

* El **backend ignora extensiones en la ruta**
* El **sistema de caché confía en la extensión del archivo para decidir qué cachear**

***

### 📝 Práctica

#### 1️⃣ Análisis inicial

Iniciamos sesión con nuestras credenciales.

Accedemos a:

```
/my-account
```

Observamos que la página muestra nuestra **API key personal**.

<figure><img src="../../../.gitbook/assets/image (946).png" alt=""><figcaption></figcaption></figure>

***

#### 2️⃣ Interceptar la petición legítima

Interceptamos la petición:

```http
GET /my-account
```

Observaciones:

* La respuesta es **dinámica**
* Contiene **información sensible**
* No aparecen cabeceras claras de caché como:

```http
Cache-Control
X-Cache
```

Esto indica que la página no debería ser cacheada en condiciones normales.

<figure><img src="../../../.gitbook/assets/image (945).png" alt=""><figcaption></figcaption></figure>

***

#### 3️⃣ Manipular la ruta

Probamos añadir una **extensión estática** al final de la ruta:

```
/my-account/anything.js
```

<figure><img src="../../../.gitbook/assets/image (947).png" alt=""><figcaption></figcaption></figure>

Resultado:

* El servidor devuelve **exactamente la misma página que `/my-account`**
* La página sigue mostrando **nuestra API key**

En peticiones posteriores aparecen cabeceras como:

```http
Cache-Control
Age
```

Esto confirma que:

* El **backend ignora la extensión**
* El **sistema de caché considera la ruta como contenido estático**

***

#### 4️⃣ Construir el exploit

Para explotar la vulnerabilidad contra **Carlos**, necesitamos que él (autenticado) visite una ruta manipulada.

Usamos el **Exploit Server** para forzar una redirección automática.

Contenido del exploit:

```html
<script>
location = "https://ID-LAB.web-security-academy.net/my-account/evil.js";
</script>
```

Guardamos el exploit utilizando **Store** y ejecutamos **Deliver exploit to victim**.

<figure><img src="../../../.gitbook/assets/image (949).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Recuperar el contenido cacheado

Cuando el laboratorio simula que Carlos visita la página maliciosa:

1. Carlos es redirigido a:

```
/my-account/evil.js
```

2. El servidor devuelve su página real `/my-account`.
3. La respuesta contiene **su API key personal**.
4. La caché almacena esa respuesta como si fuera un **recurso estático**.

Ahora accedemos directamente a:

```
https://ID-LAB.web-security-academy.net/my-account/evil.js
```

<figure><img src="../../../.gitbook/assets/image (950).png" alt=""><figcaption></figcaption></figure>

La respuesta cacheada corresponde a la sesión de Carlos y contiene **su API key**.

Copiamos la clave y la enviamos como solución.
