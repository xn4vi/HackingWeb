# 💻 Descubrimiento de endpoint GraphQL oculto

## 📄 Descripción del laboratorio

Las funciones de gestión de usuarios de este laboratorio se ejecutan mediante un **endpoint GraphQL oculto**, que no está accesible desde la interfaz web.

Además, el endpoint implementa **protecciones contra introspection**, bloqueando consultas estándar como `__schema` y `__type`.

🎯 **Objetivo del laboratorio:**

* Descubrir el endpoint GraphQL oculto
* Enumerar el schema de la API
* Eliminar al usuario `carlos`



## 📚 Teoría

Este laboratorio combina dos problemas habituales en implementaciones de GraphQL:

* **Endpoint discovery**
* **Bypass de protecciones anti-introspection**

### 📌 Endpoint GraphQL oculto

El endpoint GraphQL no está enlazado desde la interfaz web, pero sigue utilizando rutas comunes del backend como:

```
/graphql
/api
/api/graphql
/v1/graphql
```

Esto permite descubrir el endpoint mediante **fuzzing de rutas comunes**.

### 📌 Filtro débil contra introspection

El backend intenta bloquear introspection filtrando consultas que contienen literalmente:

```
__schema
__type
```

Sin embargo, este filtro utiliza una **expresión regular simple**, lo que permite evadirlo con pequeñas modificaciones en la query.

Por ejemplo, insertando un salto de línea codificado:

```
%0a
```

Esto rompe la coincidencia literal del patrón y permite ejecutar introspection igualmente.

### 📌 Impacto

Una vez obtenido el schema completo se pueden:

* Enumerar **queries y mutations administrativas**
* Identificar **usuarios por ID**
* Ejecutar **mutaciones destructivas**

Esto demuestra que:

> Un endpoint GraphQL oculto no implica seguridad si el schema sigue siendo accesible.



## 📝 Práctica

### 1️⃣ Descubrimiento del endpoint GraphQL

El laboratorio indica que el endpoint no es accesible desde la interfaz web.

Probamos rutas comunes asociadas a GraphQL:

```
/graphql
/api
/api/graphql
/v1/graphql
```

Al acceder a:

```
/api
```

recibimos una respuesta distinta a un error 404 estándar, lo que sugiere que existe lógica backend activa.

### 2️⃣ Probar introspection

Interceptamos la petición a `/api` y la enviamos a **Burp Repeater**.

Probamos una introspection query estándar:

```graphql
{
  __schema {
    types {
      name
    }
  }
}
```

<img width="768" height="357" alt="image" src="https://github.com/user-attachments/assets/395557d7-ceb3-4753-9529-3aea037d64ec" />
<br>

El servidor devuelve un error indicando que la introspección está bloqueada y menciona específicamente `__schema` y `__type`.

<img width="768" height="327" alt="image" src="https://github.com/user-attachments/assets/71564bc2-ae4b-4b45-b303-3bdbd79082a3" />
<br>

Esto confirma que existe un **filtro por patrón**, no una desactivación real de introspection.



### 3️⃣ Bypass del filtro

Intentamos evadir el filtro introduciendo un salto de línea codificado dentro del campo:

```graphql
{
  __sche%0ama {
    types {
      name
    }
  }
}
```

El `%0a` rompe la coincidencia literal de la expresión regular.

<img width="768" height="343" alt="image" src="https://github.com/user-attachments/assets/2f73b720-d3cd-401b-9bc8-133c01c979c6" />
<br>

El servidor devuelve ahora el **schema completo de GraphQL**.
