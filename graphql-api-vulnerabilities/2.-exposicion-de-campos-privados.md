# 2. Exposición de Campos Privados

## 💻 Exposición accidental de campos privados

### 📄 Descripción del laboratorio

Las funciones de gestión de usuarios de la aplicación se realizan mediante un **endpoint GraphQL**.

El laboratorio contiene un fallo de control de acceso que permite **revelar campos sensibles de credenciales de los usuarios**.

🎯 **Objetivo del laboratorio:**

* Obtener las credenciales del usuario **administrator**
* Iniciar sesión como administrador
* Eliminar al usuario **carlos**

***

### 📚 Teoría

Este laboratorio demuestra un problema común en APIs GraphQL: **exposición de campos internos sin control de acceso adecuado**.

El backend expone una consulta llamada:

```
getUser
```

Esta query:

* Permite consultar usuarios por **ID**
* Devuelve campos sensibles como:

```
username
password
```

El problema es que **no existe validación de permisos ni control de roles**.

Con **introspection habilitada**, un atacante puede:

1. Enumerar el **schema completo**
2. Identificar queries sensibles como `getUser`
3. Probar **IDs secuenciales**
4. Extraer directamente las credenciales del administrador

Este tipo de vulnerabilidad combina:

* **IDOR**
* **exposición de datos sensibles**
* **falta de control de acceso**

***

### 📝 Práctica

#### 1️⃣ Enumeración del endpoint GraphQL

Interceptamos una petición GraphQL desde la aplicación con **Burp Suite** y la enviamos a **Repeater**.

Ejecutamos una consulta de introspection:

```graphql
{
  __schema {
    types {
      name
      fields {
        name
      }
    }
  }
}
```

La respuesta devuelve el **schema completo de la API**.

<figure><img src="../../../.gitbook/assets/image (703).png" alt=""><figcaption></figcaption></figure>

***

#### 2️⃣ Identificar consultas sensibles

Para facilitar el análisis, revisamos la información en **Target → Sitemap** dentro de Burp.

<figure><img src="../../../.gitbook/assets/image (704).png" alt=""><figcaption></figcaption></figure>

Entre las consultas disponibles aparece:

```
getUser
```

Esta query permite consultar información de usuarios a partir de su identificador.

<figure><img src="../../../.gitbook/assets/image (705).png" alt=""><figcaption></figcaption></figure>

***

#### 3️⃣ Abusar de la query getUser

Probamos la query manualmente desde Repeater.

Primero con ID 0:

```graphql
query {
  getUser(id: 0) {
    username
    password
  }
}
```

<figure><img src="../../../.gitbook/assets/image (706).png" alt=""><figcaption></figcaption></figure>

La respuesta es nula.

Probamos con ID 1:

```graphql
query {
  getUser(id: 1) {
    username
    password
  }
}
```

<figure><img src="../../../.gitbook/assets/image (709).png" alt=""><figcaption></figcaption></figure>

La API devuelve directamente:

```
username
password
```

correspondientes al usuario **administrator**.

***

#### 4️⃣ Acceso como administrador

Con las credenciales obtenidas iniciamos sesión en la aplicación web como:

```
administrator
```

Accedemos al panel de gestión de usuarios.

***

#### 5️⃣ Eliminación del usuario carlos

Desde el panel administrativo localizamos al usuario:

```
carlos
```

Seleccionamos la opción **Delete**.

<figure><img src="../../../.gitbook/assets/image (707).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Resultado

Se consigue:

* Extraer las credenciales del usuario **administrator**
* Iniciar sesión como administrador
* Eliminar al usuario **carlos**

**Laboratorio resuelto.**
