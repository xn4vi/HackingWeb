# 🤖 Explotación de vulnerabilidades en APIs accesibles desde un LLM

## 📄 Descripción del laboratorio

Este laboratorio demuestra cómo un **Large Language Model (LLM)** puede convertirse en un vector de ataque cuando tiene acceso directo a **APIs internas vulnerables**.

En este caso, la aplicación permite al LLM interactuar con una **API de suscripción al boletín** que ejecuta comandos del sistema operativo de forma insegura.

El objetivo es:

* Aprovechar una **inyección de comandos del sistema operativo**.
* Ejecutar código en el servidor.
* Eliminar el archivo:

```
/home/carlos/morale.txt
```

utilizando únicamente el **chat con el LLM**.



## 📚 Teoría

Cuando un LLM tiene acceso a **APIs internas que ejecutan comandos del sistema operativo**, el modelo deja de ser solo un generador de texto y pasa a ser un **orquestador de acciones reales**.

Si una API:

* Construye comandos del sistema mediante **concatenación de cadenas**.
* No **escapa correctamente la entrada del usuario**.
* Permite **command substitution** (`$(...)`).

entonces el impacto puede escalar directamente a **Remote Command Execution (RCE)**.

En este laboratorio, la API de suscripción al boletín:

* Recibe el **email proporcionado por el usuario**.
* Lo inserta directamente en un comando similar a:

```
sendmail <email>
```

Si el valor del email incluye una expresión como:

```
$(comando)
```

el **shell evalúa esa expresión antes de ejecutar el comando principal**.

Esto permite ejecutar **comandos arbitrarios en el sistema**.

### 🎯 Impacto

El comando se ejecuta con los **privilegios del proceso backend**, que en este laboratorio corresponden al usuario:

```
carlos
```

Este patrón es especialmente peligroso en arquitecturas modernas donde:

* El **LLM actúa como intermediario inteligente**.
* Las APIs internas asumen que el input ya es seguro.
* No existe separación clara entre **lenguaje natural y ejecución de comandos**.



## 📝 Práctica

### 🎯 Objetivo

Eliminar el archivo:

```
/home/carlos/morale.txt
```

utilizando únicamente el **chat del LLM**.



### 1️⃣ Enumeración de capacidades

Accedemos a la aplicación y abrimos el **live chat con el LLM**.

<img width="1046" height="168" alt="image" src="https://github.com/user-attachments/assets/c30e3187-bc38-4da9-949b-f0d6ec90450e" />
<br>
Mediante preguntas progresivas descubrimos que el modelo tiene acceso a varias herramientas internas, entre ellas:

* Email Client
* Logs del sistema
* API de suscripción al boletín

Esto indica que el LLM puede **interactuar directamente con servicios backend**.



### 2️⃣ Confirmación de ejecución de comandos

Solicitamos al modelo que nos suscriba al boletín con un email controlado.

<img width="1043" height="137" alt="image" src="https://github.com/user-attachments/assets/11b854c8-74f8-4a21-9172-e14a59204835" />
<br>
Observamos que:

* Se genera un correo real.
* Aparece actividad en los **logs del sistema**.
* La API ejecuta comandos en el backend.

Para verificar la existencia de **command injection**, utilizamos el siguiente payload:

```bash
$(whoami)@exploit-server.net
```

<img width="818" height="93" alt="image" src="https://github.com/user-attachments/assets/3f6dad54-ebae-488f-837e-96b8d6625a8c" />
<br>
Después de enviarlo:

* Revisamos los logs del sistema.
* Observamos que el valor devuelto es:

```
carlos
```

<img width="1044" height="127" alt="image" src="https://github.com/user-attachments/assets/42f14690-0cea-45de-aedc-45184d159a85" />
<br>
Esto confirma que:

* El comando se ejecuta en el sistema.
* Se ejecuta con los privilegios del usuario **carlos**.
* Existe una **inyección de comandos del sistema operativo**.



### 3️⃣ Payload final

Una vez confirmado el vector de ataque, enviamos al LLM el siguiente payload:

```bash
$(rm /home/carlos/morale.txt)@exploit-server.net
```

<img width="889" height="103" alt="image" src="https://github.com/user-attachments/assets/54e8d0a9-46f6-4efa-b618-1a67642691fe" />
<br>
El flujo de ejecución es el siguiente:

1. El LLM envía el email proporcionado a la API vulnerable.
2. La API construye el comando de shell.
3. El shell evalúa la expresión `$(rm ...)`.
4. Se ejecuta el comando en el sistema.

El archivo `morale.txt` es eliminado correctamente.



### 4️⃣ Resultado final

El archivo objetivo:

```
/home/carlos/morale.txt
```

es eliminado del sistema.

El laboratorio detecta la acción y se marca como **completado correctamente**.
