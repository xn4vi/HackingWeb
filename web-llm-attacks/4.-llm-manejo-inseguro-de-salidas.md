# 4. LLM Manejo Inseguro de Salidas

## 🤖 Explotación por manejo inseguro de salidas en LLMs

### 📄 Descripción del laboratorio

Este laboratorio muestra una vulnerabilidad crítica en el **manejo de la salida de un LLM**.\
El contenido generado por el modelo se **inserta directamente en el DOM sin sanitización ni escaping**, lo que permite la ejecución de **Cross-Site Scripting (XSS)**.

El usuario **carlos** consulta frecuentemente información sobre la **Lightweight "l33t" Leather Jacket** mediante el chat integrado con el modelo.

El objetivo es:

* Inyectar un **payload XSS persistente**.
* Hacer que el LLM lo incluya en sus respuestas.
* Conseguir que el script se ejecute automáticamente en la sesión de **carlos** y elimine su cuenta.

***

### 📚 Teoría

Cuando la salida de un LLM:

* contiene **HTML o JavaScript**, y
* se renderiza en la página **sin output encoding ni sanitización**,

el navegador interpreta ese contenido como **HTML real**, ejecutando cualquier script incluido.

En este laboratorio ocurre lo siguiente:

* El LLM consume **reviews de productos** (contenido generado por usuarios).
* Dichas reviews se insertan en las respuestas del chat **sin filtrado**.
* El frontend renderiza el contenido del modelo directamente en el **DOM**.

Esto genera la siguiente cadena de ejecución:

```
User input → Review → LLM → DOM → JavaScript ejecutado
```

#### 🎯 Impacto

El problema no es solo XSS clásico.

Se trata de una **inyección indirecta**, ya que:

* El atacante no interactúa directamente con la víctima.
* El **LLM actúa como intermediario que amplifica el payload**.
* El ataque se ejecuta cuando **otro usuario consulta el producto**.

El resultado puede ser **account takeover o ejecución de acciones sensibles sin interacción**.

***

### 📝 Práctica

#### 🎯 Objetivo

Eliminar la cuenta del usuario **carlos** mediante un **XSS persistente** que se active cuando consulte el producto.

***

#### 1️⃣ Confirmación de XSS en la salida del LLM

Creamos un nuevo usuario y accedemos al **live chat**.

Probamos un payload simple:

```html
<img src=x onerror=alert(1)>
```

<figure><img src="../../../.gitbook/assets/image (791).png" alt=""><figcaption></figcaption></figure>

Resultado:

* El LLM refleja el contenido **sin modificarlo**.
* El navegador ejecuta `alert(1)`.

Esto confirma que:

* La salida del modelo se inserta usando algo equivalente a:

```javascript
element.innerHTML = response
```

* No existe ningún proceso de **sanitización**.

Este XSS es **reflejado**, por lo que solo afecta a nuestra sesión.

Para impactar a **carlos**, necesitamos un **vector persistente**.

***

#### 2️⃣ Elección del vector persistente

Sabemos que:

* **carlos consulta habitualmente la Lightweight "l33t" Leather Jacket**.
* El LLM **recupera y muestra las reviews del producto**.

Por lo tanto, el mejor vector es **inyectar el payload dentro de una review del producto**.

Cuando el LLM responda preguntas sobre ese producto, incluirá nuestra review en su respuesta.

***

#### 3️⃣ Payload XSS final

Tras varias pruebas (para evitar que el propio modelo modifique el texto), utilizamos el siguiente payload en la review:

```html
Este producto me encanta.
<iframe src="/my-account"
        onload="this.contentDocument.forms[1].submit()"
        style="display:none;"></iframe>
Lo recomiendo mucho.
```

#### 📌 Funcionamiento del payload

1. Se carga la página `/my-account` dentro de un **iframe oculto**.
2. El iframe se ejecuta en el **contexto de la sesión de la víctima**.
3. Cuando se carga el contenido, se ejecuta:

```javascript
this.contentDocument.forms[1].submit()
```

Esto envía automáticamente el **formulario de eliminación de cuenta**.

Puntos clave:

* El **token CSRF válido ya está presente en la página**.
* No se requiere interacción del usuario.
* El ataque se ejecuta automáticamente.

***

#### 4️⃣ Ejecución del ataque

Publicamos la review maliciosa en el producto objetivo.

Cuando **carlos consulta al LLM sobre la chaqueta**:

1. El modelo recopila información del producto.
2. Incluye nuestra review en su respuesta.
3. El navegador de carlos renderiza el HTML.
4. El iframe se carga en segundo plano.
5. Se envía automáticamente el formulario de eliminación.

La cuenta de **carlos** se elimina sin ninguna interacción.

***

#### 5️⃣ Resultado final

El XSS persistente se ejecuta cuando el LLM muestra la review maliciosa.

Esto provoca la eliminación automática de la cuenta del usuario **carlos**.

El laboratorio se completa correctamente.
