# 📂 Path traversal - bypass con null byte y validación de extensión

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **Path Traversal** en la funcionalidad de visualización de imágenes de productos.

La aplicación valida que el parámetro `filename` termine con la extensión esperada (`.jpg`), pero no tiene en cuenta el **byte nulo (`%00`)**.

El objetivo es recuperar el contenido del archivo:

```
/etc/passwd
```

utilizando un **null byte** para evadir la validación de extensión.

 

## 📚 Teoría

En este escenario, la aplicación aplica una defensa basada en la extensión del archivo, similar a:

```javascript
if (filename.endsWith(".jpg")) {
    // aceptar
}
```

El problema aparece cuando la aplicación o alguna librería subyacente trata el **byte nulo (`\0`) como fin de cadena**, algo común en:

* Código escrito en **C/C++**
* **Wrappers nativos**
* Librerías **legacy** o bindings mal implementados

### 📌 Cómo funciona el bypass

Payload utilizado:

```
../../etc/passwd%00.jpg
```

### 📌 **Interpretación paso a paso**

➡️ **Validación de extensión**

El filtro comprueba que la cadena termina en `.jpg`, por lo que la validación se cumple y la petición es aceptada.

➡️ **Uso real del path**

La función que abre el archivo interpreta el byte nulo como final de cadena y **trunca la ruta en `%00`**.

La ruta efectiva utilizada por el sistema pasa a ser:

```
../../etc/passwd
```

El sistema de archivos resuelve la ruta y devuelve el contenido de:

```
/etc/passwd
```

La vulnerabilidad existe debido a:

* Validar la extensión **antes de normalizar o truncar la cadena**
* No neutralizar el **byte nulo**
* Uso de funciones sensibles a `\0`
* Confianza excesiva en **validaciones basadas en sufijo**

 

## 📝 Práctica

### 🎯 Objetivo

Leer el archivo:

```
/etc/passwd
```

 

### 1️⃣ Identificación de la petición vulnerable

Se navega por la tienda, se selecciona cualquier producto y se observa la petición que carga la imagen:

```http
GET /image?filename=6.jpg
```

La petición se intercepta con **Burp Suite** y se envía al **Repeater**.

 

### 2️⃣ Verificación de la validación por extensión

Se prueban técnicas anteriores como:

* Traversal clásico
* Rutas absolutas
* Doble codificación

Todas las peticiones son bloqueadas porque:

* El parámetro `filename` **no termina en `.jpg`**

Esto confirma que existe una **validación estricta basada en la extensión del archivo**.

 

### 3️⃣ Bypass mediante null byte

Se prueba el siguiente payload:

```
/image?filename=../../../../../../etc/passwd%00.jpg
```

### 4️⃣ Resultado

La aplicación acepta el parámetro porque termina en `.jpg`.

Sin embargo, el **byte nulo trunca la ruta real**, por lo que el sistema accede a:

```
/etc/passwd
```

La respuesta HTTP devuelve directamente el contenido del archivo.

Esto confirma la existencia de una vulnerabilidad de **Path Traversal explotada mediante null byte injection**.
