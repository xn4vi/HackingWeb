# 📂 Path traversal - bypass con doble decodificación en URL

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **Path Traversal** en la funcionalidad de visualización de imágenes de productos.

La aplicación bloquea las secuencias clásicas de recorrido de ruta (`../`), pero comete un error crítico: **decodifica la URL una segunda vez después de aplicar el filtro**.

El objetivo es recuperar el contenido del archivo:

```
/etc/passwd
```

utilizando **doble codificación URL** para evadir el control.

 

## 📚 Teoría

En este escenario, la aplicación sigue un flujo inseguro:

1. Recibe el parámetro `filename`.
2. Aplica un filtro que elimina o bloquea cualquier aparición de `../`.
3. Después del filtrado, realiza una **decodificación URL adicional**.
4. Usa el valor resultante para acceder al sistema de archivos.

El problema está en el **orden incorrecto de las operaciones**: se filtra antes de la decodificación final.

Esto permite aplicar un bypass clásico mediante **double URL encoding**.

### 📌 Cadena clave

```
..%252f
```

### 📌 Paso a paso

1. `%252f` → primera decodificación → `%2f`\
   El filtro no detecta `/` ni `../`.
2. `%2f` → segunda decodificación → `/`\
   La ruta final se convierte en `../`.

De esta forma, la secuencia `../` aparece **solo después de que el filtro haya terminado**, lo que permite escapar del directorio base.

La vulnerabilidad existe debido a:

* Filtrado aplicado **antes de la decodificación final**
* **Decodificación múltiple** sin revalidación
* Falta de **normalización completa de la ruta**
* Confianza excesiva en filtros de entrada simples

 

## 📝 Práctica

### 🎯 Objetivo

Leer el archivo:

```
/etc/passwd
```

 

### 1️⃣ Identificación de la petición vulnerable

Se navega por la tienda, se selecciona cualquier producto y se observa la petición de carga de imagen:

```http
GET /image?filename=6.jpg
```

Se intercepta la petición con **Burp Suite** y se envía al **Repeater**.

 

### 2️⃣ Prueba con traversal clásico

Se prueban primero las técnicas tradicionales:

```
/image?filename=../../../../../../etc/passwd
```

Resultado:

La aplicación bloquea la petición al detectar la secuencia `../`.

Esto confirma que:

* Existe un **filtro activo**.
* El filtro detecta **secuencias directas de traversal**.

 

### 3️⃣ Bypass mediante doble codificación

Se prueba el siguiente payload con **double URL encoding**:

```
/image?filename=..%252f..%252f..%252f..%252f..%252fetc%252fpasswd
```

### 📌 Qué ocurre internamente

1. `%252f` → `%2f` tras la primera decodificación.\
   El filtro no detecta `/`.
2. `%2f` → `/` tras la segunda decodificación.

La ruta final interpretada por el sistema se convierte en:

```
../../../../../etc/passwd
```

### 4️⃣ Resultado

La respuesta HTTP contiene directamente el contenido del archivo:

```
/etc/passwd
```

Esto confirma que la vulnerabilidad de **Path Traversal** ha sido explotada con éxito mediante **doble decodificación de URL**.
