# 📂 Path traversal - secuencias traversal eliminadas sin recursión

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **Path Traversal** en la funcionalidad de visualización de imágenes de productos.

La aplicación intenta mitigar el ataque eliminando las secuencias `../` del parámetro `filename`, pero el filtro se aplica **solo una vez y no de forma recursiva**.

El objetivo es recuperar el contenido del archivo:

```
/etc/passwd
```

evadiendo este filtro defectuoso.

 

## 📚 Teoría

En este caso, la aplicación procesa el parámetro `filename` sustituyendo o eliminando todas las ocurrencias de `../`.

El problema es que este filtrado se realiza en **una sola pasada**, de izquierda a derecha, y no vuelve a comprobar el resultado final.

Esto permite aplicar una técnica clásica de bypass basada en **reconstrucción de secuencias traversal**.

Un ejemplo característico es:

```
....//
```

### 📌 Qué ocurre internamente

1. El filtro elimina la secuencia `../`.
2. La cadena `....//` se transforma en `..//`.
3. El sistema de archivos normaliza `..//` como `../`.

El resultado es que se obtiene un **retroceso de directorio válido después de que el filtro haya terminado**.

Encadenando varias veces esta técnica:

```
....//....//....//
```

es posible reconstruir un `../../../` funcional y escapar del directorio base.

La vulnerabilidad existe debido a:

* Filtro **no recursivo**
* Falta de **normalización posterior**
* Confianza excesiva en **sustituciones de cadenas**
* Uso directo del parámetro `filename` como ruta de archivo

 

## 📝 Práctica

### 🎯 Objetivo

Leer el archivo:

```
/etc/passwd
```

 

### 1️⃣ Identificación de la petición vulnerable

Se navega por la tienda y se selecciona cualquier producto.

Se observa la petición que carga la imagen:

```http
GET /image?filename=6.jpg
```

Se intercepta la petición con **Burp Suite** y se envía al **Repeater**.

 

### 2️⃣ Prueba con el payload clásico

Se prueba primero el ataque tradicional:

```
/image?filename=../../../../../../etc/passwd
```

Resultado:

La aplicación bloquea la petición, confirmando que existe un **filtro que elimina `../`**.

 

### 3️⃣ Reconstrucción de la secuencia traversal

Se prueba una variante diseñada para reconstruir el traversal después del filtrado:

```
/image?filename=....//....//....//....//....//etc/passwd
```

### 📌 Qué sucede internamente

Cada secuencia:

```
....//
```

tras eliminar `../` se convierte en:

```
..//
```

El sistema de archivos interpreta `..//` como `../`, produciendo **múltiples retrocesos efectivos de directorio**.

### 4️⃣ Resultado

La respuesta HTTP contiene directamente el contenido del archivo:

```
/etc/passwd
```

Esto confirma que el filtro ha sido **evadido con éxito** y que existe una vulnerabilidad de **Path Traversal**.
