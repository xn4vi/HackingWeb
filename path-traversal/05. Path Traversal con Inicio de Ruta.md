# 📂 Path traversal - validación del inicio de la ruta

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **Path Traversal** en la funcionalidad de visualización de imágenes de productos.

La aplicación valida que el valor del parámetro `filename` comience con la ruta esperada:

```
/var/www/images/
```

pero no restringe ni valida lo que viene después.

El objetivo es recuperar el contenido del archivo:

```
/etc/passwd
```

evadiendo esta validación superficial basada únicamente en prefijo.

***

## 📚 Teoría

En este escenario, la aplicación implementa una defensa débil basada en una comprobación similar a:

```javascript
if (filename.startsWith("/var/www/images/")) {
    // aceptar
}
```

El problema es que solo se valida el **prefijo de la cadena**, no el resultado final de la ruta tras su resolución en el sistema de archivos.

Esto permite construir rutas que:

* Cumplen el requisito del prefijo `/var/www/images/`
* Incluyen secuencias `../` inmediatamente después
* Escapan del directorio base al resolverse en el sistema de archivos
* Acceden finalmente a archivos arbitrarios como `/etc/passwd`

Un ejemplo de ruta utilizada para explotar este comportamiento sería:

```
/var/www/images/../../../etc/passwd
```

Aunque la ruta empieza correctamente, el sistema de archivos **normaliza las secuencias `../`**, resolviendo finalmente la ruta real como:

```
/etc/passwd
```

La vulnerabilidad existe debido a:

* Validar únicamente el **inicio de la ruta** mediante una comprobación de cadena
* No **normalizar la ruta antes de validarla**
* No comprobar que la **ruta final permanezca dentro del directorio permitido**
* Confiar en que un prefijo considerado seguro es suficiente



## 📝 Práctica

### 🎯 Objetivo

Leer el contenido del archivo:

```
/etc/passwd
```



### 1️⃣ Identificación de la petición vulnerable

Se navega por la tienda, se selecciona cualquier producto y se observa la petición que carga la imagen:

```http
GET /image?filename=6.jpg
```

La petición se intercepta con **Burp Suite** y se envía al **Repeater**.



### 2️⃣ Prueba con técnicas clásicas de traversal

Se prueba primero un payload típico de Path Traversal:

```
/image?filename=../../../../../../etc/passwd
```

Resultado:

La aplicación rechaza la petición, ya que **no comienza con `/var/www/images/`**.

Esto confirma que:

* Existe una validación activa
* La validación solo comprueba **el prefijo de la ruta**


### 3️⃣ Construcción del bypass

Se construye una ruta que:

* Empiece por `/var/www/images/`
* Escale inmediatamente con secuencias `../`

Payload final:

```
/image?filename=/var/www/images/../../../etc/passwd
```

Se envía la petición modificada.

### 4️⃣ Resultado

La aplicación acepta la ruta porque comienza con:

```
/var/www/images/
```

El sistema de archivos resuelve las secuencias `../` y la ruta final se interpreta como:

```
/etc/passwd
```

La respuesta HTTP devuelve el contenido completo del archivo.

Esto confirma la existencia de una vulnerabilidad de **Path Traversal mediante bypass de validación por prefijo**.
