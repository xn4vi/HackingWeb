# 🌐 XSS DOM con `innerHTML` y `location.search`

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **DOM-based XSS** en la funcionalidad de búsqueda del blog.

La aplicación utiliza una asignación directa a `innerHTML` para insertar en el DOM el valor obtenido de `location.search`, **sin ningún tipo de sanitización ni codificación**.

🎯 **Objetivo del laboratorio:**

* Ejecutar la función `alert()` mediante un ataque XSS.



## 📚 Teoría

En este laboratorio analizamos otra causa común de **DOM-based XSS**: el uso inseguro de `innerHTML`.

La propiedad `innerHTML`:

```
inserta HTML directamente en el DOM
interpreta etiquetas HTML
permite atributos y eventos
ejecuta código válido dentro de ese contexto
```

El flujo vulnerable es el siguiente:

```
1. El usuario introduce un término de búsqueda
2. El JavaScript obtiene el valor desde location.search
3. Ese valor se asigna directamente a element.innerHTML
4. El navegador interpreta el contenido como HTML
```

En este caso no existe:

```
validación del input
codificación HTML
sanitización del contenido
```

Por tanto, cualquier contenido controlado por el usuario puede terminar ejecutándose en el navegador.

Un detalle importante es que **no todos los contextos permiten ejecutar `<script>` directamente**.\
Sin embargo, cuando se utiliza `innerHTML`, es posible abusar de **event handlers HTML** como:

```
onerror
onclick
onload
```

Estos eventos permiten ejecutar JavaScript sin necesidad de usar la etiqueta `<script>`.

En este laboratorio aprovechamos un comportamiento típico del navegador:

```
una imagen con una ruta inválida provoca un error
ese error dispara el evento onerror
el código dentro de onerror se ejecuta automáticamente
```



## 📝 Práctica

### 1️⃣ Probar inyección directa de `<script>`

Comenzamos probando el payload clásico:

```html
<script>alert(1)</script>
```

Resultado:

No aparece ninguna alerta.

Esto indica que:

```
el input sí se inserta en el DOM
pero el contexto impide ejecutar <script> directamente
```



### 2️⃣ Probar inyección de HTML simple

Probamos ahora con una etiqueta HTML simple:

```html
<img src="test.png">
```

<img width="732" height="78" alt="image" src="https://github.com/user-attachments/assets/0a527766-6109-4c8c-a8ad-d9179e494c38" />
<br>

Resultado:

El navegador intenta cargar la imagen.

<img width="322" height="57" alt="image" src="https://github.com/user-attachments/assets/84d02321-dcff-472b-b2c2-138aac457fca" />
<br>

Esto confirma que:

```
el HTML se interpreta correctamente
podemos insertar etiquetas HTML arbitrarias
```



### 3️⃣ Aprovechar eventos HTML (`onerror`)

Ahora utilizamos un payload con un manejador de evento.

```html
<img src=0 onerror=alert(1)>
```

<img width="726" height="79" alt="image" src="https://github.com/user-attachments/assets/66e69c28-4b82-4a70-a3ce-fde49a0dc024" />
<br>

Funcionamiento del payload:

```
1. El navegador intenta cargar la imagen src=0
2. La carga falla
3. Se dispara automáticamente el evento onerror
4. Se ejecuta alert(1)
```

Resultado:

Aparece la ventana emergente con `alert(1)`.

Esto confirma que existe **DOM-based XSS explotable**.

En ese momento el laboratorio se marca como completado.

Se consigue:

* Insertar HTML arbitrario mediante `innerHTML`
* Ejecutar JavaScript usando un event handler (`onerror`)
* Explotar un **DOM-based XSS**

**Laboratorio resuelto.**
