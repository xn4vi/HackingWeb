# 🌐 XSS DOM con `document.write` y `location.search`

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **DOM-based XSS** en la funcionalidad de seguimiento de búsquedas.

El código JavaScript de la página utiliza `document.write()` para insertar directamente en el HTML el valor obtenido de `location.search`, que es completamente controlable por el usuario a través de la URL.

🎯 **Objetivo del laboratorio:**

* Realizar un ataque XSS que ejecute la función `alert()`.



## 📚 Teoría

En este laboratorio se explora el **DOM-based XSS**, una variante donde la vulnerabilidad no está en el servidor, sino en el **JavaScript que se ejecuta en el navegador**.

A diferencia del XSS reflejado o almacenado:

```
el servidor puede devolver HTML completamente seguro
la vulnerabilidad ocurre cuando el JavaScript procesa datos no confiables
```

El flujo vulnerable en este caso es:

```
1. El navegador carga la página
2. El script JavaScript lee el valor de location.search
3. Ese valor se inserta en el DOM mediante document.write()
4. No existe validación ni codificación del input
```

El resultado es que cualquier contenido controlado por el usuario puede terminar siendo interpretado como **HTML o JavaScript ejecutable**.

La función `document.write()` es especialmente peligrosa porque:

```
inserta HTML directamente en el DOM
no diferencia entre texto y código
permite ejecutar cualquier etiqueta HTML generada
```

En este laboratorio, el valor controlado por el usuario se inserta dentro del atributo `src` de una etiqueta `<img>`.

Esto introduce un concepto clave: **el contexto HTML**.

Si el payload queda dentro de un atributo, primero será necesario **escapar de ese atributo** antes de poder ejecutar JavaScript.



## 📝 Práctica

### 1️⃣ Probar un payload directo

Accedemos a la funcionalidad de búsqueda y probamos el payload más simple:

```html
<script>alert(1)</script>
```

Resultado:

No aparece ninguna alerta.

Esto indica que el payload sí se está insertando en la página, pero **no en un contexto que permita ejecutar scripts directamente**.



### 2️⃣ Analizar el contexto en el DOM

Inspeccionamos el código de la página usando las herramientas de desarrollo del navegador.

Observamos algo similar a:

```html
<img src="/resources/images/tracker.gif?search=VALOR">
```

<img width="417" height="37" alt="image" src="https://github.com/user-attachments/assets/48411f2e-e771-4532-8758-4e6e7402c8ac" />
<br>

Esto significa que nuestro input se está insertando dentro del atributo `src` de una etiqueta `<img>`.

Si no cerramos correctamente ese atributo, el navegador seguirá interpretando todo como parte de la URL del `src`.

Por lo tanto, debemos **romper el contexto del atributo HTML**.



### 3️⃣ Escapar del atributo HTML

Para comprobar que podemos escapar del atributo, probamos el siguiente payload en la URL:

```html
"><h1>Hola</h1>
```

<br>

Resultado:

El texto **Hola** aparece renderizado como un encabezado en la página.

Esto confirma que:

```
hemos cerrado el atributo src
hemos cerrado la etiqueta <img>
podemos inyectar HTML arbitrario
```

### 4️⃣ Ejecutar JavaScript

Ahora que controlamos el contexto HTML, introducimos el payload final:

```html
"><script>alert(1)</script>
```
