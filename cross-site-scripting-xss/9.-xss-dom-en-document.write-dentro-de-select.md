# 9. XSS DOM en Document.write dentro de Select

## 🌐 XSS DOM en `document.write` dentro de `select`

### 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **DOM-based XSS** en la funcionalidad del verificador de existencias.

El código JavaScript utiliza `document.write()` para insertar directamente en la página un valor obtenido desde `location.search`, que es completamente controlable por el usuario. Ese valor se inserta dentro de un elemento `<select>`.

🎯 **Objetivo del laboratorio:**

* Realizar un ataque XSS que escape del elemento `<select>` y ejecute la función `alert()`.

***

### 📚 Teoría

Este laboratorio muestra otro caso de **DOM-based XSS**, donde el problema está en el JavaScript del lado cliente.

El flujo vulnerable es el siguiente:

```
1. El usuario accede a la página de un producto
2. La URL incluye un parámetro storeId
3. El JavaScript obtiene ese valor desde location.search
4. document.write() inserta el valor directamente en el HTML
```

El HTML generado tiene una estructura similar a:

```html
<select>
  <option>VALOR_USUARIO</option>
</select>
```

En este caso no existe:

```
codificación HTML
validación del input
restricción de caracteres
```

El problema no es solo `document.write()`, sino el **contexto HTML**.

El valor controlado por el usuario se inserta dentro de una etiqueta `<option>`.\
Esto significa que el navegador lo interpretará como parte de la estructura del DOM.

Si el atacante consigue:

```
cerrar la etiqueta <option>
cerrar el elemento <select>
inyectar HTML arbitrario
```

entonces podrá introducir etiquetas ejecutables como `<script>`.

Este laboratorio demuestra que incluso estructuras aparentemente inocuas como **menús desplegables (`<select>`)** pueden convertirse en vectores de ataque.

***

### 📝 Práctica

#### 1️⃣ Identificar el parámetro vulnerable

Accedemos a la página del producto y observamos la URL:

<figure><img src="../../../.gitbook/assets/image (1025).png" alt=""><figcaption></figcaption></figure>

```
https://LAB-ID.web-security-academy.net/product?productId=1&storeId=test
```

Observamos que el parámetro `storeId` aparece reflejado en la página.

Esto indica que el valor se utiliza para rellenar el selector de tiendas.

Por lo tanto, el parámetro vulnerable es:

```
storeId
```

***

#### 2️⃣ Confirmar el contexto HTML

Inspeccionamos el código fuente de la página.

Encontramos una estructura similar a:

```html
<select>
  <option>test</option>
</select>
```

<figure><img src="../../../.gitbook/assets/image (1026).png" alt=""><figcaption></figcaption></figure>

Nuestro input aparece dentro de la etiqueta `<option>`.

Esto implica que:

```
no podemos ejecutar JavaScript directamente
primero debemos romper la estructura HTML
```

***

#### 3️⃣ Romper la estructura del `<select>`

Para escapar del contexto actual debemos:

```
cerrar la etiqueta <option>
cerrar la etiqueta <select>
inyectar nuestro propio HTML
```

Payload utilizado en la URL:

```
https://LAB-ID.web-security-academy.net/product?productId=1&storeId=</option></select><script>alert(1)</script>
```

<figure><img src="../../../.gitbook/assets/image (1027).png" alt=""><figcaption></figcaption></figure>

***

#### 4️⃣ Ejecutar el XSS

Accedemos a la URL con el payload.

El navegador procesa el HTML generado por `document.write()`.

El resultado es:

```
se cierran las etiquetas </option> y </select>
se inserta la etiqueta <script>
se ejecuta alert(1)
```

Esto confirma que el **DOM-based XSS es explotable**.

El laboratorio se marca como completado.

El laboratorio se marca como completado.

<figure><img src="../../../.gitbook/assets/image (1028).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Resultado

Se consigue:

* Identificar el parámetro vulnerable (`storeId`)
* Escapar del contexto del elemento `<select>`
* Inyectar código HTML arbitrario
* Ejecutar JavaScript mediante `<script>`

**Laboratorio resuelto.**
