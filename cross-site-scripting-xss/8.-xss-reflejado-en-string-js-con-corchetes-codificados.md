# 8. XSS Reflejado en String JS con Corchetes Codificados

## 🌐 XSS reflejado en string JS con corchetes codificados

### 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **XSS reflejado** en la funcionalidad de seguimiento de búsquedas.

La aplicación codifica los caracteres `<` y `>`, pero refleja el input del usuario **dentro de una cadena JavaScript**, lo que introduce un vector de ataque distinto al contexto HTML.

🎯 **Objetivo del laboratorio:**

* Realizar un ataque XSS que rompa la cadena JavaScript y ejecute `alert()`.

***

### 📚 Teoría

Este laboratorio muestra un caso típico de **XSS en contexto JavaScript**.

El problema no está en el HTML, sino en cómo el backend construye el código JavaScript que se envía al navegador.

El flujo vulnerable es el siguiente:

```
1. El usuario introduce un término de búsqueda
2. El servidor refleja ese valor en la respuesta
3. El valor se inserta dentro de una cadena JavaScript
```

Ejemplo simplificado:

```javascript
var searchTerm = 'VALOR_USUARIO';
```

La aplicación intenta aplicar una defensa:

```
codifica los caracteres < y >
```

Sin embargo, esta protección **solo es útil para HTML**, no para JavaScript.

En contexto JavaScript, los caracteres más peligrosos son:

```
las comillas
el punto y coma
los delimitadores de cadena
```

Si un atacante puede **cerrar la cadena JavaScript**, puede:

```
ejecutar código arbitrario
insertar nuevas instrucciones
reabrir la cadena para mantener la sintaxis válida
```

Este tipo de XSS es especialmente peligroso porque:

```
no depende del DOM
no requiere interacción del usuario
se ejecuta automáticamente al cargar la página
```

***

### 📝 Práctica

#### 1️⃣ Probar payloads clásicos

Comenzamos probando payloads comunes de XSS:

```html
<script>alert(1)</script>
```

```html
<img src=0 onerror=alert(1)>
```

Resultado:

No se ejecuta ningún código.

Esto confirma que:

```
no estamos en un contexto HTML
el input se está insertando en otro lugar
```

***

#### 2️⃣ Analizar el código JavaScript

Inspeccionamos el código fuente de la página.

Encontramos algo similar a:

```javascript
var search = 'BUSQUEDA_USUARIO';
```

<figure><img src="../../../.gitbook/assets/image (1019).png" alt=""><figcaption></figcaption></figure>

Esto significa que el input del usuario se encuentra dentro de una **cadena delimitada por comillas simples**.

Si conseguimos cerrar esa cadena, podremos ejecutar código JavaScript.

***

#### 3️⃣ Romper la cadena JavaScript

Para explotar el XSS necesitamos:

```
cerrar la cadena original
ejecutar código JavaScript
evitar errores de sintaxis
```

Payload utilizado:

```javascript
probando';alert(1);var probando='hola
```

<figure><img src="../../../.gitbook/assets/image (1020).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../../.gitbook/assets/image (1021).png" alt=""><figcaption></figcaption></figure>

Funcionamiento del payload:

```
probando'           cierra la cadena original
;alert(1);          ejecuta nuestro código
var probando='hola  abre una nueva cadena
```

Esto mantiene la sintaxis del script válida.

***

#### 4️⃣ Ejecutar el XSS

Introducimos el payload en el campo de búsqueda y enviamos la petición.

Resultado:

La página carga normalmente y se ejecuta automáticamente:

```java
alert(1)
```

Esto confirma que el **XSS reflejado es explotable**.

El laboratorio se marca como completado.

<figure><img src="../../../.gitbook/assets/image (1022).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Resultado

Se consigue:

* Identificar que el input se inserta en una cadena JavaScript
* Romper la cadena original
* Ejecutar código JavaScript arbitrario

**Laboratorio resuelto.**
