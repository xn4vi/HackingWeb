# 5. XSS DOM en href con jQuery y location.serch

## 🌐 XSS DOM en `href` con jQuery y `location.search`

### 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **DOM-based XSS** en la página de envío de comentarios.

La aplicación utiliza **jQuery (`$`)** para localizar un enlace `<a>` y modificar su atributo `href` usando datos obtenidos directamente de `location.search`, sin ningún tipo de validación.

🎯 **Objetivo del laboratorio:**

* Conseguir que el enlace **Back** ejecute `alert(document.cookie)`.

***

### 📚 Teoría

En este laboratorio se analiza un patrón muy común de **DOM-based XSS**: modificar atributos HTML con **datos controlados por el usuario**.

El flujo vulnerable es el siguiente:

```
1. El usuario accede a la página con un parámetro en la URL (returnPath)
2. El JavaScript obtiene ese valor desde location.search
3. jQuery localiza el enlace "Back"
4. El valor se asigna directamente al atributo href
```

El código vulnerable es conceptualmente equivalente a:

```
$('a.back').attr('href', returnPath);
```

El problema es que **no existe ningún tipo de validación del esquema de la URL**.

Un atributo `href` no solo acepta URLs HTTP o HTTPS.\
Los navegadores también permiten esquemas como:

```
javascript:
data:
vbscript: (en navegadores antiguos)
```

Si el atributo `href` contiene un esquema `javascript:`, el código:

```
no se ejecuta al cargar la página
se ejecuta cuando el usuario hace clic en el enlace
```

Este tipo de XSS:

```
no requiere <script>
no requiere inyección HTML
ocurre completamente en el cliente
```

Además es especialmente peligroso porque **parece un enlace legítimo**, lo que facilita engañar al usuario para que haga clic.

***

### 📝 Práctica

#### 1️⃣ Localizar el punto vulnerable

Accedemos a la página de envío de comentarios.

Observamos que existe un enlace **Back** que permite volver a la página anterior.

Al interactuar con la página o cambiar ciertos parámetros en la URL, vemos que el destino del enlace cambia dinámicamente.

Probamos a introducir una URL en el campo **Website** del comentario y observamos que el enlace **Back** apunta a esa misma URL.

Esto indica que el valor proviene de un parámetro de la URL llamado:

```
returnPath
```

<figure><img src="../../../.gitbook/assets/image (1006).png" alt=""><figcaption></figcaption></figure>

Por lo tanto, el valor de `location.search` se utiliza para construir el atributo `href`.

***

#### 2️⃣ Entender el sink: `href` + jQuery

El JavaScript realiza una operación equivalente a:

```javascript
$('a.back').attr('href', returnPath);
```

Esto significa que el valor del parámetro **se copia directamente en el atributo `href`**.

No existe:

```
validación del esquema
whitelist de protocolos
sanitización del valor
```

Esto permite usar esquemas peligrosos como `javascript:`.

***

#### 3️⃣ Inyectar un esquema `javascript:`

Construimos una URL maliciosa utilizando el parámetro vulnerable:

```
https://LAB-ID.web-security-academy.net/feedback?returnPath=javascript:alert(document.cookie)
```

<figure><img src="../../../.gitbook/assets/image (1007).png" alt=""><figcaption></figcaption></figure>

Accedemos a esa URL.

La página se carga normalmente, pero ahora el enlace **Back** tiene como destino:

```javascript
javascript:alert(document.cookie)
```

***

#### 4️⃣ Ejecutar el XSS

Hacemos clic en el enlace **Back**.

<figure><img src="../../../.gitbook/assets/image (1008).png" alt=""><figcaption></figcaption></figure>

Resultado:

El navegador ejecuta el código JavaScript y aparece una ventana emergente mostrando `document.cookie`.

Esto confirma que existe un **DOM-based XSS explotable**.

En ese momento el laboratorio se marca como completado.

***

#### 5️⃣ Resultado

Se consigue:

* Controlar el valor del atributo `href`
* Inyectar el esquema `javascript:`
* Ejecutar código JavaScript al hacer clic en el enlace

**Laboratorio resuelto.**
