# 13. XSS Reflejado con Etiquetas Bloqueadas

## 🌐 XSS reflejado en HTML con etiquetas bloqueadas

### 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de XSS reflejado en la funcionalidad de búsqueda. La aplicación está protegida por un WAF (Web Application Firewall) que bloquea la mayoría de etiquetas y atributos XSS comunes.

El objetivo es evadir el WAF y conseguir que se ejecute la función `print()` automáticamente en el navegador de la víctima, sin necesidad de interacción del usuario.

***

### 📚 Teoría

#### 📌 Funcionamiento de la vulnerabilidad

En este laboratorio el flujo vulnerable es el siguiente:

1. El usuario introduce un término de búsqueda.
2. El servidor refleja ese valor directamente en el HTML de la página.
3. Antes de devolver la respuesta, un WAF analiza el contenido.
4. Si detecta etiquetas o atributos considerados peligrosos, bloquea la respuesta.
5. Algunas etiquetas y eventos menos comunes no están filtrados y pueden pasar el WAF.

El WAF bloquea la mayoría de vectores XSS conocidos, por ejemplo:

* Etiquetas como `<script>`, `<img>`, `<svg>`.
* Eventos HTML comunes como `onload`, `onclick`, `onerror`.

Sin embargo, algunos elementos HTML siguen permitidos y algunos eventos menos comunes también.

Además, el laboratorio impone una restricción importante: el código JavaScript debe ejecutarse automáticamente, sin interacción del usuario. No es posible utilizar eventos que dependan de clics, teclado o movimiento del ratón.

Esto obliga a encontrar un evento que se dispare automáticamente y posteriormente forzar su ejecución.

#### 📌 Limitaciones de los WAF

Los WAF suelen utilizar listas negras de etiquetas y atributos considerados peligrosos. Este enfoque tiene dos problemas importantes:

* Es imposible bloquear todas las combinaciones posibles.
* Siempre existen etiquetas o eventos menos comunes que no están incluidos en el filtro.

Por este motivo, un WAF no elimina completamente las vulnerabilidades XSS. Solo dificulta su explotación.

***

### 📝 Práctica

#### 1️⃣ Probar payloads XSS clásicos

Comenzamos probando payloads XSS comunes en el campo de búsqueda:

```html
<script>print()</script>
```

<figure><img src="../../../.gitbook/assets/image (1061).png" alt=""><figcaption></figcaption></figure>

El resultado es un error indicando que la etiqueta no está permitida. El WAF bloquea la petición.

Probamos también otras etiquetas comunes como:

```html
<img>
<svg>
```

Todas ellas son bloqueadas. Esto confirma que el WAF está filtrando activamente etiquetas conocidas.

***

#### 2️⃣ Enumerar etiquetas HTML permitidas

Para descubrir qué etiquetas están permitidas utilizamos Burp Suite Intruder.

1. Interceptamos la petición de búsqueda.
2. La enviamos a Intruder.
3. Marcamos la etiqueta HTML como posición de payload.
4. Utilizamos una lista de etiquetas HTML comunes.

<figure><img src="../../../.gitbook/assets/image (1062).png" alt=""><figcaption></figcaption></figure>

Durante el ataque observamos los códigos de respuesta del servidor. La mayoría de etiquetas generan errores, pero una de ellas devuelve un código 200.

La etiqueta que pasa el filtro es:

```html
<body>
```

Esto indica que `<body>` está permitido por el WAF.

<figure><img src="../../../.gitbook/assets/image (1063).png" alt=""><figcaption></figcaption></figure>

***

#### 3️⃣ Enumerar eventos permitidos

Una vez encontrada una etiqueta permitida, el siguiente paso es identificar eventos HTML que no estén filtrados.

Configuramos Intruder nuevamente utilizando el siguiente patrón:

```html
<body EVENTO=print()>
```

En este caso el payload corresponde al nombre del evento.

<figure><img src="../../../.gitbook/assets/image (1064).png" alt=""><figcaption></figcaption></figure>

Tras lanzar el ataque observamos que muchos eventos son bloqueados, pero uno de ellos devuelve una respuesta válida.

El evento permitido es:

```javascript
onresize
```

<figure><img src="../../../.gitbook/assets/image (1065).png" alt=""><figcaption></figcaption></figure>

***

#### 4️⃣ Construir el payload XSS

Ahora sabemos que:

* La etiqueta `<body>` está permitida.
* El evento `onresize` está permitido.

Podemos construir el siguiente payload:

```html
<body onresize=print()>
```

Sin embargo, este evento solo se ejecuta cuando cambia el tamaño del elemento. Por lo tanto, necesitamos provocar ese cambio automáticamente.

***

#### 5️⃣ Forzar el evento resize

Para forzar el evento utilizamos un `iframe` desde el Exploit Server del laboratorio.

Creamos el siguiente payload:

```html
<iframe 
src="https://LAB-ID.web-security-academy.net/?search=<body onresize=print()>"
onload="this.style.width='100px'">
</iframe>
```

<figure><img src="../../../.gitbook/assets/image (1067).png" alt=""><figcaption></figcaption></figure>

El funcionamiento del exploit es el siguiente:

1. El `iframe` carga la página vulnerable.
2. El parámetro `search` inyecta el elemento `<body onresize=print()>`.
3. Cuando el `iframe` termina de cargar, se ejecuta el evento `onload`.
4. El script modifica el ancho del `iframe`.
5. El navegador detecta un cambio de tamaño y dispara el evento `resize`.
6. El evento `onresize` ejecuta `print()` automáticamente.

***

#### 6️⃣ Resultado

Para completar el laboratorio:

1. Guardamos el payload en el Exploit Server.
2. Seleccionamos la opción **Deliver exploit to victim**.

Cuando la víctima carga la página, el navegador ejecuta `print()` automáticamente sin interacción.

El laboratorio se marca como completado.
