# 🌐 XSS DOM reflejado

## 📄 Descripción del laboratorio

Este laboratorio presenta una vulnerabilidad de **DOM-based XSS reflejado**.

El servidor recibe un parámetro de búsqueda, lo incluye dentro de una respuesta **JSON**, y posteriormente un script del lado cliente procesa ese JSON de forma insegura introduciendo su contenido en un **sumidero peligroso**.

🎯 **Objetivo del laboratorio:**

* Crear una inyección XSS que ejecute la función `alert()`.



## 📚 Teoría

Este laboratorio muestra un patrón muy realista de XSS que combina **backend y frontend**.

El flujo vulnerable es el siguiente:

```
1. El usuario introduce un término de búsqueda
2. El servidor incluye ese valor en una respuesta JSON
3. El navegador recibe el JSON
4. Un script del lado cliente procesa el contenido
5. El script utiliza eval() para interpretar el resultado
```

El problema aparece cuando el contenido del JSON incluye datos controlados por el usuario y se procesa mediante:

```javascript
eval()
```

La función `eval()` interpreta una cadena como **código JavaScript ejecutable**.

Esto significa que cualquier manipulación del contenido evaluado puede convertirse directamente en ejecución de código.

La aplicación intenta aplicar una defensa parcial:

```
escapa las comillas dobles (")
```

Sin embargo, **no escapa la barra invertida (`\`)**.

En JavaScript, la barra invertida se utiliza como carácter de escape.

Por ejemplo:

```
\"  permite insertar una comilla dentro de una cadena
```

Si el atacante puede introducir `\`, puede manipular la forma en que el navegador interpreta la cadena.

Esto permite:

```
romper la estructura del JSON o del código evaluado
insertar instrucciones JavaScript
comentar el resto del código para evitar errores
```

Este tipo de XSS demuestra que:

```
JSON también puede ser un vector de XSS
escapar parcialmente caracteres no es suficiente
eval() convierte datos en código ejecutable
```



## 📝 Práctica

### 1️⃣ Probar un payload clásico

Comenzamos probando un payload típico de XSS:

```html
<script>alert(1)</script>
```

Resultado:

No se ejecuta ningún código.

Esto indica que el contexto **no es HTML**.



### 2️⃣ Analizar el código del lado cliente

Inspeccionamos el código JavaScript de la página.

<img width="446" height="128" alt="image" src="https://github.com/user-attachments/assets/c691df83-5788-4c23-a3c8-be991439b2b9" />
<br>

Observamos que los resultados de búsqueda se devuelven como **JSON**.

Ese JSON se procesa mediante una llamada a:

```javascript
eval()
```

Esto es una señal clara de posible vulnerabilidad.



### 3️⃣ Analizar el escape de caracteres

Probando distintos valores en el parámetro de búsqueda observamos que:

```
las comillas dobles (") están escapadas
la barra invertida (\) no está escapada
```

Esto abre una vía de ataque.

La barra invertida permite manipular el sistema de escapes dentro de JavaScript.



### 4️⃣ Construir el payload

Creamos un payload que:

```
introduce una barra invertida
rompe la estructura original
inyecta una llamada a alert()
comenta el resto del código
```

Payload utilizado:

```java
test\"*alert(1)}//
```

URL final:

```
https://LAB-ID.web-security-academy.net/?search=test\"*alert(1)}//
```

<img width="704" height="62" alt="image" src="https://github.com/user-attachments/assets/74e43ed6-eaa0-4286-a483-0c579ed48456" />
<br>

Funcionamiento conceptual del payload:

```
\" fuerza una comilla dentro de la cadena
se rompe la estructura original del código evaluado
alert(1) se ejecuta
// comenta el resto del script
```



### 5️⃣ Ejecutar el XSS

Accedemos a la URL con el payload.

Resultado:

La página se carga normalmente y aparece la ventana:

```javascript
alert(1)
```

Esto confirma que el **DOM-based XSS reflejado es explotable**.
