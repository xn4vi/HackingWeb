# 🔓 Bypass de autenticación JWT mediante inyección en jku

## 📄 Descripción del laboratorio

Este laboratorio utiliza **JSON Web Tokens (JWT)** con criptografía asimétrica para gestionar las sesiones.

El servidor admite el parámetro:

```
jku
```

en el header del JWT, permitiendo indicar una URL desde la que obtener la **clave pública de verificación**.

El problema es que el servidor **no valida si esa URL pertenece a un dominio de confianza**, lo que permite a un atacante controlar completamente el proceso de verificación.

El objetivo del laboratorio es:

* Falsificar un JWT
* Acceder al panel de administración:

```
/admin
```

* Eliminar al usuario **carlos**

Credenciales proporcionadas:

```
wiener:peter
```



## 📚 Teoría

Este laboratorio explota una mala gestión de claves públicas en JWT con **RS256**.

### 📌 Funcionamiento de RS256

* El servidor firma los tokens con una **clave privada**
* El servidor verifica los tokens con una **clave pública**

En un diseño seguro:

* La clave pública está **predefinida o controlada por el servidor**
* El cliente no puede influir en su origen

### 📌 Parámetro jku

El campo:

```
"jku"
```

indica una URL desde la que el servidor descargará un **JWK Set** (conjunto de claves públicas).

### 📌 Fallo de seguridad

El servidor:

* Acepta `jku` en el header
* No valida el dominio
* Descarga la clave pública sin restricciones
* Confía completamente en un valor controlado por el cliente

### 📌 Vector de ataque

El atacante puede:

1. Generar un par de claves RSA
2. Publicar la clave pública en un servidor controlado
3. Crear un JWT con:

```
"sub": "administrator"
```

4. Firmarlo con su clave privada
5. Indicar en `jku` la URL donde está su clave pública

El servidor descargará la clave desde esa URL y validará la firma correctamente.

No se rompe la criptografía, sino el **modelo de confianza**.



## 📝 Práctica

### 1️⃣ Obtener un JWT válido

Iniciamos sesión con:

```
Username: wiener
Password: peter
```

Interceptamos una petición autenticada y observamos la cookie:

```
session=<JWT>
```



### 2️⃣ Confirmar el algoritmo

Enviamos la petición a **Burp Repeater** y analizamos el header:

```json
{
  "alg": "RS256",
  "typ": "JWT"
}
```

Confirmamos que se utiliza criptografía asimétrica.



### 3️⃣ Generar un par de claves RSA

En la extensión **JWT Editor**:

* Seleccionamos **New RSA Key**
* Pulsamos **Generate**

<img width="535" height="568" alt="image" src="https://github.com/user-attachments/assets/4abbbdef-58d0-472d-bf9d-0e59cb9232fb" />
<br>
Se generan:

* Una clave privada
* Una clave pública



### 4️⃣ Exportar la clave pública como JWK

Sobre la clave generada:

* Click derecho → **Copy public key as JWK**

Obtenemos un objeto con campos como:

<img width="680" height="702" alt="image" src="https://github.com/user-attachments/assets/f11009f8-7fb3-493d-96f1-513c418dd44b" />
<br>
El campo:

```
kid
```

identifica la clave.



### 5️⃣ Publicar el JWK Set

Abrimos el **Exploit Server** y creamos un archivo JSON:

```json
{
  "keys": [
    {
      "kty": "RSA",
      "e": "AQAB",
      "n": "CLAVE_PUBLICA",
      "kid": "ID_CLAVE"
    }
  ]
}
```

Guardamos el contenido con **Store** y copiamos la URL pública.

<br>


### 6️⃣ Modificar el JWT

Editamos el token original:

**Payload**

```
"sub": "administrator"
```

<img width="244" height="129" alt="image" src="https://github.com/user-attachments/assets/519dbc36-e92a-4b86-a90a-e537ed3d70eb" />
<br>
**Header**

Añadimos:

```
"jku": "https://EXPLOIT-SERVER.net/archivo.json",
"kid": "ID_CLAVE"
```

<img width="768" height="124" alt="image" src="https://github.com/user-attachments/assets/15f1ec1e-b2f8-4514-bc67-4fdad2beed3b" />



### 7️⃣ Firmar el token

Firmamos el JWT con nuestra clave privada usando JWT Editor.

El token resultante es coherente con la clave pública publicada en `jku`.



### 8️⃣ Reemplazar la cookie de sesión

Sustituimos la cookie:

```
session=JWT_MODIFICADO
```

Refrescamos la página.

Resultado:

* La sesión sigue siendo válida
* El servidor nos reconoce como **administrator**

<img width="768" height="220" alt="image" src="https://github.com/user-attachments/assets/1b30289d-5473-4005-aa64-f0bac944d114" />



### 9️⃣ Acceder al panel de administración

Accedemos a:

```
/admin
```

El panel carga correctamente.

Buscamos al usuario **carlos** y pulsamos **Delete**.

El usuario se elimina y el laboratorio se completa.
