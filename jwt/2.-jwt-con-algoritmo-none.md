# 2. JWT con Algoritmo None

## 🔓 Bypass de autenticación JWT mediante algoritmo none

### 📄 Descripción del laboratorio

Este laboratorio utiliza **JSON Web Tokens (JWT)** para gestionar la autenticación de usuarios.

El servidor está configurado de forma insegura y **acepta tokens sin firmar**, permitiendo el uso del algoritmo:

```
none
```

Esto permite a un atacante **fabricar tokens válidos sin conocer ninguna clave secreta**.

El objetivo del laboratorio es:

* Modificar el token de sesión
* Acceder al panel de administración:

```
/admin
```

* Eliminar al usuario **carlos**

Credenciales proporcionadas:

```
wiener:peter
```

***

### 📚 Teoría

Un JWT tiene la siguiente estructura:

```
header.payload.signature
```

* **Header**: define el algoritmo (`alg`) y el tipo (`typ`)
* **Payload**: contiene las claims (por ejemplo `sub`)
* **Signature**: garantiza la integridad del token

#### 📌 Algoritmo none

El valor:

```
"alg": "none"
```

indica que el token **no está firmado**.

En una implementación segura:

* Los tokens con `alg: none` deben ser **rechazados**
* Nunca deben aceptarse sin firma

#### 📌 Fallo en el laboratorio

El servidor:

* Acepta tokens con `alg: none`
* No exige firma
* No valida la integridad del token
* Confía directamente en el contenido del payload

Esto permite construir un JWT completamente arbitrario.

#### 📌 Impacto

Podemos crear un token con:

```json
{
  "sub": "administrator"
}
```

y el servidor lo aceptará como válido.

A diferencia del laboratorio anterior:

* Allí la firma existía pero no se verificaba
* Aquí la firma **ni siquiera es necesaria**

Esto convierte el JWT en un **simple contenedor JSON controlado por el cliente**.

***

### 📝 Práctica

#### 1️⃣ Obtener un JWT válido

Iniciamos sesión con:

```
Username: wiener
Password: peter
```

Interceptamos una petición autenticada y observamos la cookie:

```
session=<JWT>
```

<figure><img src="../../../.gitbook/assets/image (523).png" alt=""><figcaption></figcaption></figure>

Copiamos el token completo.

***

#### 2️⃣ Analizar el token original

Dividimos el JWT:

```
header.payload.signature
```

Decodificamos el header:

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

Decodificamos el payload:

```json
{
  "sub": "wiener",
  "iat": 1690000000
}
```

El campo clave es `sub`, que identifica al usuario.

***

#### 3️⃣ Modificar el header

Editamos el header para usar el algoritmo none:

```json
{
  "alg": "none",
  "typ": "JWT"
}
```

Lo codificamos en Base64URL.

***

#### 4️⃣ Modificar el payload

Editamos el payload cambiando:

```
"sub": "wiener"
```

por:

```
"sub": "administrator"
```

Lo codificamos nuevamente en Base64URL.

<figure><img src="../../../.gitbook/assets/image (524).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../../.gitbook/assets/image (525).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Eliminar la firma

Eliminamos completamente la tercera parte del token.

El resultado final debe ser:

```
Base64(header).Base64(payload).
```

Importante:

* El punto final **debe mantenerse**
* No hay firma

<figure><img src="../../../.gitbook/assets/image (527).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Reemplazar el token de sesión

Sustituimos la cookie:

```
session=JWT_MODIFICADO
```

Refrescamos la página.

<figure><img src="../../../.gitbook/assets/image (528).png" alt=""><figcaption></figcaption></figure>

Resultado:

* La sesión sigue siendo válida
* Ahora somos tratados como **administrator**

***

#### 7️⃣ Acceder al panel de administración

Accedemos a:

```
/admin
```

El panel carga correctamente.

Buscamos al usuario **carlos** y pulsamos **Delete**.

El usuario se elimina y el laboratorio se completa.
