# 🧑‍💻 Gestión inconsistente de entradas atípicas

## 📄 Descripción del laboratorio

Este laboratorio presenta una vulnerabilidad en la validación de direcciones de correo electrónico debido a una **gestión incorrecta de entradas largas**.

El sistema restringe el acceso al panel de administración a usuarios con dominio:

```
@dontwannacry.com
```

Sin embargo, las direcciones de correo se recortan automáticamente a **255 caracteres**, sin volver a validar el dominio tras el truncamiento.

El objetivo es:

* Registrar una cuenta que aparenta pertenecer al dominio permitido
* Obtener acceso al panel `/admin`
* Eliminar al usuario carlos



## 📚 Teoría

Las aplicaciones deben validar correctamente los datos tanto antes como después de cualquier transformación.

### 📌 El fallo

El sistema:

* Acepta direcciones de email largas
* Recorta automáticamente a 255 caracteres
* No vuelve a validar el dominio tras el recorte

Esto provoca que:

* El dominio final almacenado puede ser distinto del original
* Se pueda manipular la posición del dominio dentro del límite

### 📌 Vector de ataque

Si construimos un email:

* Con muchos caracteres al inicio
* Y colocamos `@dontwannacry.com` justo antes del carácter 255

El sistema:

* Recorta la cadena
* Mantiene el dominio permitido
* Ignora el resto

### 📌 Impacto

Esto permite:

* Bypass de validaciones de dominio
* Escalada de privilegios
* Acceso no autorizado al panel de administración



## 📝 Práctica

### 1️⃣ Identificar la restricción

Buscamos acceder al panel de administración.

Observamos que:

* Solo está permitido para usuarios con dominio `@dontwannacry.com`

### 2️⃣ Analizar el registro

Accedemos al cliente de correo del laboratorio.

Intentamos registrarnos normalmente, pero observamos que:

* No podemos cambiar el email posteriormente

Esto nos obliga a explotar el registro.



### 3️⃣ Probar límites del campo email

En el registro, introducimos:

* Nuestro email real
* Precedido por una cadena larga (por ejemplo 200 caracteres)

<img width="740" height="453" alt="image" src="https://github.com/user-attachments/assets/3703034f-1596-4a6a-a847-2af9961b077d" />
<br>
El objetivo es comprobar:

* Si existe un límite
* Cómo lo gestiona el sistema

Nos registramos y confirmamos el email.



### 4️⃣ Detectar el truncamiento

Accedemos a nuestra cuenta y observamos:

* El email ha sido recortado

<img width="1221" height="231" alt="image" src="https://github.com/user-attachments/assets/ba7d6adb-054b-44c2-b1a6-1c5f79c24a97" />
<br>
Esto indica que:

* Existe un límite de 255 caracteres
* El sistema trunca el valor automáticamente



### 5️⃣ Construir el payload

Ahora construimos un email con esta estructura:

* Prefijo largo controlado
* Dominio `@dontwannacry.com` justo antes del carácter 255
* Nuestro dominio real después

Ejemplo conceptual:

```
[relleno_controlado]@dontwannacry.com[resto]
```

<img width="1520" height="144" alt="image" src="https://github.com/user-attachments/assets/05dc157a-bfc2-44e6-8bdb-54ef6af21ee0" />
<br>
Al truncarse:

* El sistema conserva el dominio permitido
* El email real queda fuera del límite



### 6️⃣ Registrar la cuenta maliciosa

Nos registramos con el email construido.

Validamos el correo desde el email client.
