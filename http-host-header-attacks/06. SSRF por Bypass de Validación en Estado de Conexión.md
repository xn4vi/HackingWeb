# 📡 SSRF por bypass de validación basada en estado de conexión

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Server-Side Request Forgery (SSRF)** debido a un manejo inseguro del estado de conexión HTTP.

El servidor frontal valida correctamente la cabecera `Host` en la **primera petición de una conexión**, pero asume erróneamente que todas las peticiones posteriores dentro de esa misma conexión son confiables.

El objetivo es aprovechar este comportamiento para acceder al panel administrativo interno ubicado en:

```
192.168.0.1/admin
```

Una vez dentro del panel, debemos eliminar al usuario **carlos**.



## 📚 Teoría

### 📌 SSRF basada en reutilización de conexión

Este laboratorio demuestra una variante avanzada de SSRF basada en **confusión de estado de conexión**.

El comportamiento vulnerable del servidor es el siguiente:

1. El servidor valida la cabecera `Host` en la **primera petición de una conexión TCP**.
2. Si la validación es correcta, la conexión se marca como **confiable**.
3. Las peticiones posteriores dentro de esa misma conexión:
   * no vuelven a validarse
   * se enrutan utilizando directamente los valores de cada petición

Esto permite un ataque de tipo **state confusion**:

* La primera petición establece una conexión válida.
* La segunda petición explota la confianza otorgada a esa conexión.

Este tipo de vulnerabilidad está inspirado en ataques reales investigados por PortSwigger, como:

* **Browser-Powered Desync Attacks**
* SSRF por reutilización de conexión
* errores de implementación en proxies y servidores HTTP

El problema no está en el protocolo HTTP, sino en implementaciones que mezclan:

* estado de conexión
* estado de autenticación o confianza

Una conexión válida no implica que todas las peticiones posteriores deban considerarse confiables.



## 📝 Práctica

### 1️⃣ Comprobación inicial

Intentamos acceder directamente al panel administrativo:

```
/admin
```

Resultado:

El servidor devuelve un mensaje similar a:

```
Only available to local users
```

Esto confirma que el acceso está restringido a hosts internos.



### 2️⃣ Probar manipulación directa del Host

Interceptamos una petición normal:

```http
GET / HTTP/1.1
Host: ID-LAB.web-security-academy.net
```

Intentamos modificar la cabecera `Host`:

```http
Host: 192.168.0.1/admin
```

<img width="768" height="308" alt="image" src="https://github.com/user-attachments/assets/d815ce8c-e1fd-49ce-95d4-eb3d46a423f1" />
<br>
Resultado:

La petición es bloqueada o redirigida.

Esto confirma que el servidor **sí valida la cabecera Host en peticiones aisladas**.



### 3️⃣ Preparar el ataque por reutilización de conexión

El objetivo es enviar **dos peticiones dentro de la misma conexión TCP**.

Usamos **Burp Repeater**.

1. Enviamos la primera petición al Repeater.
2. Creamos un grupo de peticiones:

```
Right click → Add to group
```



### 4️⃣ Enviar la primera petición válida

Primera petición del grupo:

```http
GET / HTTP/1.1
Host: ID-LAB.web-security-academy.net
```

Esta petición:

* pasa la validación del servidor
* marca la conexión como confiable

<img width="768" height="132" alt="image" src="https://github.com/user-attachments/assets/d704c830-7f68-4357-861e-9f7cf9c24846" />
<br>


### 5️⃣ Enviar la segunda petición maliciosa

Segunda petición del grupo:

```http
GET /admin HTTP/1.1
Host: 192.168.0.1
```

En Burp Repeater configuramos:

* **Send group in sequence**
* **Single connection**

Esto garantiza que ambas peticiones se envían dentro de la misma conexión TCP.



### 6️⃣ Acceder al panel administrativo interno

Resultado de la segunda petición:

El servidor devuelve el HTML del panel administrativo interno.

No se requiere autenticación adicional.

Esto confirma que el servidor **solo validó la primera petición de la conexión**.

<img width="768" height="298" alt="image" src="https://github.com/user-attachments/assets/33bfd281-a830-48a3-889f-d49b3251a7e8" />



### 7️⃣ Eliminar al usuario carlos

Dentro del panel identificamos la acción para eliminar usuarios.

Ejemplo de endpoint:

```
/admin/delete?username=carlos&csrf=TOKEN
```

Construimos la segunda petición del grupo:

```http
GET /admin/delete?username=carlos&csrf=CSRF_TOKEN HTTP/1.1
Host: 192.168.0.1
```

Enviamos nuevamente el grupo de peticiones.

<img width="699" height="121" alt="image" src="https://github.com/user-attachments/assets/2e649cdc-ee2d-4938-8bea-3530ef5a65d6" />
<br>
El servidor procesa la solicitud y elimina al usuario **carlos**.

El laboratorio se marca como completado.

<img width="768" height="208" alt="image" src="https://github.com/user-attachments/assets/3389f041-4379-4687-9298-4fbc6a8604ad" />



### 8️⃣ Resultado

Mediante un ataque de **SSRF basado en reutilización de conexión** se ha conseguido:

* establecer una conexión confiable mediante una petición válida
* aprovechar esa conexión para enviar una petición hacia un host interno
* acceder al panel administrativo interno
* eliminar al usuario **carlos**

✅ El laboratorio queda resuelto.
