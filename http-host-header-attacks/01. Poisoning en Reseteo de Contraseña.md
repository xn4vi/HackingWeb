# 📡 Poisoning básico en reseteo de contraseña

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **password reset poisoning** debido a una validación inexistente de la cabecera HTTP `Host`.

La aplicación utiliza el valor de esta cabecera para construir enlaces de recuperación de contraseña enviados por correo electrónico.

El usuario **carlos** hace clic en cualquier enlace recibido por email sin verificar el dominio, lo que permite capturar su token de reseteo y tomar control de su cuenta.

Credenciales disponibles:

```
wiener:peter
```

Los correos enviados a esta cuenta pueden leerse desde el **Email Client del Exploit Server**.

El objetivo es iniciar sesión como **carlos** explotando la manipulación del enlace de recuperación de contraseña.



## 📚 Teoría

### 📌 Password reset poisoning mediante Host Header

El **password reset poisoning** es una variante clásica de **Host Header Attack**.

Ocurre cuando una aplicación:

* Genera enlaces absolutos para emails (reset de contraseña, activaciones, magic links).
* Utiliza directamente el valor de la cabecera `Host` para construir esos enlaces.

Ejemplo conceptual del backend:

```
https://<Host>/forgot-password?token=XYZ
```

Si el atacante controla el valor de `Host`, también controla el dominio del enlace enviado por correo.

En este laboratorio:

* El token de reseteo es válido.
* Carlos hace clic en el enlace recibido sin comprobar el dominio.
* El atacante puede redirigir el enlace hacia un dominio controlado por él y capturar el token.

Es importante destacar que **HTTPS no protege frente a este ataque**, ya que el enlace es válido y contiene un token legítimo, aunque apunte a un dominio malicioso.



## 📝 Práctica

### 1️⃣ Analizar el flujo de reseteo de contraseña

Iniciamos sesión con nuestras credenciales:

```
wiener:peter
```

Accedemos a la funcionalidad **Forgot password**.

Solicitamos un reseteo de contraseña para nuestro usuario y observamos el comportamiento:

* Se envía un email con un enlace de reseteo.
* El enlace contiene un token único.
* Cada nueva solicitud genera un token distinto.

Interceptamos la petición con **Burp Suite**.

### 3️⃣ Utilizar el Exploit Server

Repetimos la solicitud de reseteo de contraseña, pero esta vez cambiamos la cabecera `Host` por el dominio de nuestro **Exploit Server**.

Ejemplo:

```http
Host: exploit-server.exploit-server.net
```

Enviamos la petición indicando el usuario:

```
username=carlos
```

<img width="698" height="666" alt="image" src="https://github.com/user-attachments/assets/0012838a-1105-4d5e-bc0a-2f1020bf9d58" />
<br>
Esto provoca que el sistema envíe a Carlos un email con un enlace de reseteo que apunta a nuestro dominio.

Cuando Carlos hace clic en el enlace, su navegador realiza una petición HTTP hacia nuestro servidor.



### 4️⃣ Capturar el token

Accedemos a la sección **Access log** del Exploit Server.

Observamos una petición similar a:

```http
GET /forgot-password?token=ABC123
```

<img width="768" height="19" alt="image" src="https://github.com/user-attachments/assets/5a0aadbf-0e57-4234-a72d-00d1e8659633" />
<br>
El parámetro `token` contiene el **token de reseteo de Carlos**.



### 5️⃣ Reutilizar el token en el dominio legítimo

El enlace recibido por Carlos apunta a nuestro dominio, por lo que no funciona correctamente para completar el reseteo.

Sin embargo, el token sigue siendo válido.

Utilizamos el token en el dominio legítimo del laboratorio:

```
https://lab-id.web-security-academy.net/forgot-password?token=ABC123
```

Accedemos a esa URL y establecemos una nueva contraseña para el usuario Carlos.

### 6️⃣ Iniciar sesión como carlos

Una vez cambiada la contraseña, accedemos al formulario de login e introducimos:

```
Username: carlos
Password: nueva contraseña
```

La autenticación es exitosa y obtenemos acceso a la cuenta.

### 7️⃣ Resultado

Mediante un ataque de **password reset poisoning** se ha conseguido:

* Manipular la cabecera `Host`.
* Enviar a Carlos un enlace de recuperación controlado por el atacante.
* Capturar su token de reseteo desde el Exploit Server.
* Reutilizar el token en el dominio legítimo.
* Tomar control de la cuenta **carlos**.
