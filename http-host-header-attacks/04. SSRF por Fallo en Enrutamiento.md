# 📡 SSRF por fallo en enrutamiento

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Server-Side Request Forgery (SSRF)** debido a un fallo en el enrutamiento interno basado en la cabecera HTTP `Host`.

La aplicación utiliza esta cabecera para decidir a qué backend interno debe enviar la petición.

El objetivo es aprovechar esta vulnerabilidad para acceder a un panel administrativo interno ubicado dentro de la red:

```
192.168.0.0/24
```

Una vez localizado el panel, debemos eliminar al usuario **carlos**.

Nota: la plataforma bloquea conexiones arbitrarias a sistemas externos. Por este motivo, **Burp Collaborator** debe utilizarse como punto de prueba controlado durante la investigación.



## 📚 Teoría

### 📌 SSRF basada en enrutamiento interno

Este laboratorio explota una variante de **SSRF basada en routing interno**.

A diferencia de otras SSRF donde el atacante controla parámetros de URL, aquí el vector se encuentra en una cabecera HTTP.

El backend realiza algo equivalente a:

```
Enviar petición al backend definido por Host
```

El problema es que la aplicación:

* Utiliza directamente la cabecera `Host` como criterio de enrutamiento.
* No valida si el destino pertenece a una red interna sensible.

Esto permite que el servidor realice peticiones hacia direcciones IP internas.

Un atacante puede modificar la cabecera `Host` para apuntar a servicios internos que normalmente no están expuestos públicamente.

Esto permite:

* Enumerar servicios internos.
* Acceder a paneles administrativos internos.
* Ejecutar acciones privilegiadas dentro de la red.

En este laboratorio, el panel interno devuelve **redirecciones HTTP (302)**, lo que facilita su identificación durante la enumeración.



## 📝 Práctica

### 1️⃣ Reconocimiento inicial

Interceptamos una petición normal a la página principal:

```http
GET / HTTP/1.1
Host: ID-LAB.web-security-academy.net
```

Probamos a modificar la cabecera `Host` para apuntar a una dirección interna:

```http
Host: 192.168.0.1
```

<img width="768" height="294" alt="image" src="https://github.com/user-attachments/assets/5a932b3e-5527-4936-a9ae-0c045a45a397" />
<br>
Resultado:

La respuesta devuelve un error o un **Gateway Timeout**.

Esto indica que el servidor intenta enrutar la petición hacia esa dirección interna.

Esto confirma la existencia de un vector SSRF.



### 2️⃣ Enumerar la red interna con Intruder

Como no conocemos qué IP contiene el panel de administración, utilizamos **Burp Intruder** para escanear el rango interno.

Configuración:

* Enviamos la petición al Intruder.
* Seleccionamos como posición del payload el último octeto de la IP.

Ejemplo de cabecera:

```http
Host: 192.168.0.§X§
```

Es importante desactivar la opción:

```
Update Host header to match target
```

Esto evita que Burp sobrescriba la cabecera `Host`.

Payload:

* Tipo: Numeric
* Rango:

```
1 → 255
```

### 3️⃣ Identificar el panel administrativo

Ejecutamos el ataque y analizamos las respuestas.

La mayoría de direcciones responden con:

* errores
* timeouts

Sin embargo, una IP devuelve una respuesta **302 redirigiendo a `/admin`**.

<img width="604" height="372" alt="image" src="https://github.com/user-attachments/assets/dee46ff7-964e-4eb2-a257-bb0ab62b14ff" />
<br>
Ejemplo:

```
192.168.0.158
```

Esto indica que esa IP aloja el panel administrativo interno.



### 4️⃣ Acceder al panel de administración

Ahora utilizamos **Burp Repeater** para acceder directamente al panel.

Petición:

```http
GET /admin HTTP/1.1
Host: 192.168.0.158
```

<img width="768" height="286" alt="image" src="https://github.com/user-attachments/assets/7a770d94-d21d-42e3-a53b-e9b891985bd9" />
<br>
Respuesta:

El servidor devuelve el HTML del panel administrativo interno.

No se requiere autenticación adicional.

Esto confirma que hemos obtenido acceso administrativo mediante SSRF.



### 5️⃣ Eliminar al usuario carlos

Dentro del panel identificamos la funcionalidad para eliminar usuarios.

El enlace o acción suele tener una estructura similar a:

```
/admin/delete?username=carlos&csrf=TOKEN
```

Construimos la petición final:

```http
GET /admin/delete?username=carlos&csrf=CSRF_TOKEN HTTP/1.1
Host: 192.168.0.158
```

Enviamos la petición desde Repeater.

<img width="768" height="409" alt="image" src="https://github.com/user-attachments/assets/87f4a402-f251-4962-8c1b-fac7917019ad" />
<br>
El servidor elimina correctamente al usuario **carlos**.



### 6️⃣ Resultado

Mediante una vulnerabilidad **SSRF basada en routing interno** se ha conseguido:

* Confirmar que la cabecera `Host` controla el enrutamiento interno.
* Enumerar la red interna `192.168.0.0/24`.
* Identificar el servidor que aloja el panel administrativo.
* Acceder al panel sin autenticación.
* Eliminar al usuario **carlos**.

✅ El laboratorio se marca como completado.
