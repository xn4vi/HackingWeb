# 6. Protección Rota Bloqueo por IP

## 🔓 Protección rota: bloqueo por IP

### 📄Descripción del laboratorio

Este laboratorio es vulnerable debido a un **fallo lógico en su protección contra ataques de fuerza bruta de contraseñas**.

El sistema implementa un **bloqueo por IP tras varios intentos fallidos**, pero la lógica permite **resetear el contador de forma trivial**.

🎯 **Objetivo del laboratorio:**

* Descubrir la contraseña del usuario `carlos`
* Iniciar sesión con su cuenta
* Acceder a su página **My account**

Credenciales propias:

```
wiener:peter
```

Usuario víctima:

```
carlos
```

***

### 📚 Teoría

Este laboratorio demuestra un fallo común en sistemas de protección contra **ataques de fuerza bruta**.

El comportamiento del backend es el siguiente:

* Tras **3 intentos fallidos consecutivos desde la misma IP**, el sistema bloquea temporalmente nuevas peticiones.
* El contador de intentos fallidos **se reinicia automáticamente cuando ocurre un login válido**, independientemente del usuario.

Este comportamiento introduce una debilidad crítica.

Si alternamos:

* Intentos fallidos contra `carlos`
* Logins válidos con nuestra propia cuenta

el contador nunca alcanza el límite, permitiendo realizar **fuerza bruta ilimitada sin activar el bloqueo**.

El fallo aparece porque:

* El rate limit está asociado **solo a la IP**
* El contador **no se vincula al usuario objetivo**
* Un login válido **reinicia el estado global**

***

### 📝 Práctica

#### 1️⃣ Confirmación del comportamiento vulnerable

Desde el formulario de login probamos varias contraseñas incorrectas para `carlos`.

Después de **3 intentos fallidos**, el sistema bloquea temporalmente la IP.

<figure><img src="../../../.gitbook/assets/image (177).png" alt=""><figcaption></figcaption></figure>

Esto confirma que existe un **mecanismo de bloqueo por IP**.

***

#### 2️⃣ Preparar el ataque en Burp Intruder

Interceptamos una petición `POST /login` fallida y la enviamos a **Burp Intruder**.

Configuramos un ataque de tipo:

```
Pitchfork
```

Necesitamos controlar dos parámetros simultáneamente:

```
username
password
```

Marcamos ambos como posiciones.

***

#### 3️⃣ Construcción de los payloads

Configuramos dos sets de payloads.

**Payload set 1 – username**

Creamos un patrón que intercale nuestro usuario con el usuario víctima:

```
wiener
carlos
carlos
wiener
carlos
carlos
...
```

**Payload set 2 – password**

Alineamos nuestra contraseña válida con los intentos de login de nuestro usuario:

```
peter
password1
password2
peter
password3
password4
...
```

De esta forma, después de dos intentos fallidos contra `carlos`, se ejecuta un **login válido con `wiener`**, reiniciando el contador de intentos.

***

#### 4️⃣ Controlar la concurrencia

En **Resource Pool** configuramos un pool personalizado.

Parámetro clave:

```
Maximum concurrent requests = 1
```

Esto asegura que las peticiones se envíen **en orden exacto**, manteniendo la sincronización entre los payloads.

***

#### 5️⃣ Ejecutar el ataque

Lanzamos el ataque con Intruder.

<figure><img src="../../../.gitbook/assets/image (176).png" alt=""><figcaption></figcaption></figure>

Una vez finalizado:

* Filtramos las respuestas
* Buscamos un **status code 302 asociado al usuario `carlos`**

Un código `302` indica que el login fue **exitoso**.

<figure><img src="../../../.gitbook/assets/image (175).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Resultado

Se consigue:

* Identificar la **contraseña válida de `carlos`**
* Iniciar sesión con sus credenciales
* Acceder a la página **My account**

✅ **Laboratorio resuelto.**
