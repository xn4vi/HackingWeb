# 5. Enumeración de Usuarios por Tiempo de Respuesta

## 🔓 Enumeración de usuarios por tiempo de respuesta

### 📄 Descripción del laboratorio

El laboratorio es vulnerable a **user enumeration mediante timing attacks** en el formulario de autenticación.

El backend responde ligeramente **más lento cuando el usuario existe** que cuando no, lo que permite identificar usuarios válidos y posteriormente forzar su contraseña.

🎯 **Objetivo del laboratorio:**

* Enumerar un **usuario válido**
* Descubrir su **contraseña mediante fuerza bruta**
* Acceder a su página de cuenta

Credenciales propias:

```
wiener:peter
```

***

### 📚 Teoría

Este laboratorio muestra un ataque de **enumeración de usuarios basado en tiempos de respuesta**.

El comportamiento vulnerable se debe a que el backend procesa las solicitudes de login de forma distinta según el caso.

Cuando:

* El **usuario no existe**, el servidor responde rápidamente porque no realiza ninguna comprobación real.
* El **usuario existe**, el servidor tarda ligeramente más porque compara la contraseña con el hash almacenado.

Esta diferencia suele ser de **unos pocos milisegundos**, pero es consistente y puede medirse con herramientas de automatización.

Además, el laboratorio incluye un sistema de **bloqueo por IP tras varios intentos fallidos**.\
Sin embargo, el sistema confía en la cabecera:

```
X-Forwarded-For
```

Esto permite simular distintas direcciones IP y evitar el bloqueo simplemente modificando dicha cabecera en cada petición.

Este tipo de vulnerabilidad aparece cuando:

* No se usan **comparaciones constant-time**
* La validación de usuario y contraseña se realiza en flujos distintos
* El rate limiting confía en **cabeceras controladas por el cliente**

***

### 📝 Práctica

#### 1️⃣ Análisis inicial del login

Probamos varios intentos manuales de login.

Observamos dos comportamientos importantes:

Después de **3 intentos fallidos**, la IP queda bloqueada temporalmente.

<figure><img src="../../../.gitbook/assets/image (167).png" alt=""><figcaption></figcaption></figure>

Añadiendo la cabecera:

```http
X-Forwarded-For
```

el bloqueo desaparece, ya que el servidor confía en ese valor como IP de origen.

<figure><img src="../../../.gitbook/assets/image (168).png" alt=""><figcaption></figcaption></figure>

Durante estas pruebas también observamos que:

* Si el usuario **no existe**, la respuesta es casi inmediata.
* Si el usuario **existe**, incluso con contraseña incorrecta, la respuesta tarda ligeramente más.

Cuando se utiliza una contraseña **muy larga**, el retraso es mayor porque el servidor realiza una comparación real del hash.

<figure><img src="../../../.gitbook/assets/image (169).png" alt=""><figcaption></figcaption></figure>

Esto confirma que **el tiempo de respuesta depende de si el usuario existe**.

***

#### 2️⃣ Enumeración de usuarios por timing

Interceptamos la petición:

```http
POST /login
```

y la enviamos a **Burp Intruder**.

Configuración del ataque:

* Probar los **nombres de usuario candidatos**
* Añadir una cabecera para evitar el bloqueo por IP:

```http
X-Forwarded-For: 127.0.0.§1§
```

En Intruder observamos el valor **Response completed**, que permite comparar claramente los tiempos de respuesta.

Lanzamos el ataque.

Lanzamos el ataque.

<figure><img src="../../../.gitbook/assets/image (171).png" alt=""><figcaption></figcaption></figure>

Entre todas las respuestas, una presenta un **tiempo ligeramente superior al resto**.

Ese usuario **sí existe**.

<figure><img src="../../../.gitbook/assets/image (170).png" alt=""><figcaption></figcaption></figure>

En esta instancia, el usuario válido es `apache` (puede variar según el laboratorio).

***

#### 3️⃣ Fuerza bruta de contraseña

Una vez identificado el usuario válido, repetimos el ataque.

Configuración:

* Fijamos el **username** al usuario válido encontrado
* Probamos las **contraseñas candidatas**

Analizamos las respuestas.

<figure><img src="../../../.gitbook/assets/image (172).png" alt=""><figcaption></figcaption></figure>

La mayoría devuelven respuestas normales, pero una petición devuelve una respuesta distinta, por ejemplo:

* **Código 302 (redirect)**
* Tiempo de respuesta diferente

Esto indica que la contraseña es correcta.

***

Iniciamos sesión con:

* La **contraseña descubierta**
