# 🔓 Enumeración de usuarios por respuestas sutilmente distintas

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **enumeración de usuarios y fuerza bruta de contraseñas** debido a **diferencias mínimas en las respuestas del servidor** durante el proceso de autenticación.

Aunque el mensaje mostrado parece genérico, las respuestas del servidor presentan pequeñas variaciones que permiten distinguir entre usuarios existentes y no existentes.

🎯 **Objetivo del laboratorio:**

* Identificar un **nombre de usuario válido**
* Descubrir su **contraseña mediante fuerza bruta**
* Acceder a su página de cuenta



## 📚 Teoría

En este tipo de vulnerabilidad el servidor intenta ocultar información utilizando un mensaje genérico como:

```
Invalid username or password
```

Sin embargo, internamente el comportamiento del servidor no es exactamente el mismo en todos los casos.

Cuando:

* El **usuario no existe**, el servidor responde de una forma.
* El **usuario existe pero la contraseña es incorrecta**, responde de otra ligeramente distinta.

Las diferencias suelen ser muy sutiles, por ejemplo:

* Longitud de respuesta distinta
* Un carácter adicional o faltante
* Cambios mínimos en el texto
* Código de estado ligeramente diferente

Para un usuario normal estas diferencias pasan desapercibidas.

Sin embargo, herramientas como **Burp Intruder** permiten analizar automáticamente las respuestas y detectar estas variaciones.

Esto convierte la aplicación en un **oráculo de enumeración**, permitiendo:

* Identificar **usuarios válidos**
* Realizar **fuerza bruta de contraseñas de forma dirigida**



## 📝 Práctica

### 1️⃣ Interceptar la petición de login

Probamos primero un login manual con credenciales aleatorias.

El mensaje de error parece genérico y no revela si el usuario existe o no.

<img width="820" height="395" alt="image" src="https://github.com/user-attachments/assets/38431808-1d36-4afc-831a-f154c718bc58" />
<br>
Interceptamos la petición de login y la enviamos a **Burp Intruder**.

La petición tiene un formato similar a:

```http
POST /login
username=USER&password=PASS
```



### 2️⃣ Enumeración de usuarios

Configuramos el ataque para probar únicamente el parámetro `username`.

Configuración en Intruder:

* Tipo de ataque: **Sniper**
* Posición: `username`
* Payload: lista de **nombres de usuario candidatos**
* Desactivar **URL encoding de payloads**

Antes de lanzar el ataque configuramos **Grep – Extract** para capturar el mensaje de error de la respuesta.

Ejecutamos el ataque.

<img width="1520" height="497" alt="image" src="https://github.com/user-attachments/assets/e37fddc7-0423-427e-a478-f0a7a0556c77" />
<br>
<img width="800" height="739" alt="image" src="https://github.com/user-attachments/assets/990f6d04-7c25-4b38-85c7-e42b4b9ff546" />
<br>
Al revisar los resultados, casi todas las respuestas son iguales, excepto una.

Esa respuesta presenta una **longitud ligeramente distinta** y el mensaje cambia sutilmente, mostrando algo similar a:

```
Invalid password
```

<img width="1131" height="117" alt="image" src="https://github.com/user-attachments/assets/59cdb235-4bea-434a-b2a8-568b49c77b16" />
<br>
Esto indica que el **usuario sí existe**.



### 3️⃣ Fuerza bruta de contraseña

Una vez identificado el usuario válido, repetimos el ataque sobre el parámetro `password`.

Configuración:

* Fijamos el `username` al usuario válido
* Payload: lista de **contraseñas candidatas**

Lanzamos el ataque y analizamos las respuestas.

<img width="816" height="267" alt="image" src="https://github.com/user-attachments/assets/27476e75-0af7-493d-8846-1aca87b99e51" />
<br>
Una de ellas destaca por presentar:

* Longitud diferente
* Mensaje distinto
* O incluso un código de estado diferente

Esta respuesta corresponde a la **contraseña correcta**.



### 4️⃣ Resultado

Iniciamos sesión utilizando:

* El **usuario válido enumerado**
* La **contraseña descubierta mediante fuerza bruta**

Accedemos correctamente a la página de cuenta.

✅ **Laboratorio resuelto.**
