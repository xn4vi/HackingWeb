# 7. Enumeración de Usuarios por Bloqueo de Cuenta

## 🔓 Enumeración de usuarios por bloqueo de cuenta

### 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **enumeración de nombres de usuario** debido a un fallo lógico en su mecanismo de **bloqueo de cuentas**.

Aunque existe protección frente a ataques de fuerza bruta, esta protección se comporta de forma distinta dependiendo de si el usuario existe o no.

🎯 **Objetivo del laboratorio:**

* Enumerar un **nombre de usuario válido**
* Descubrir su **contraseña mediante fuerza bruta**
* Acceder a la página **My account**

***

### 📚 Teoría

Este laboratorio ilustra un fallo común en mecanismos de **account lockout**.

El comportamiento del backend es el siguiente:

* Tras varios intentos fallidos consecutivos, la cuenta se **bloquea temporalmente**
* El bloqueo **solo se aplica si el usuario existe**

Si el usuario no existe, el sistema:

* No bloquea nada
* O responde siempre con el mismo error genérico

Esto crea una diferencia observable entre:

* **Usuarios válidos** → acaban mostrando un estado de **account locked**
* **Usuarios inválidos** → nunca se bloquean, independientemente del número de intentos

El ataque consiste en **provocar el bloqueo deliberadamente** para usarlo como un **oráculo de enumeración**.

La lógica del ataque es:

1. Enviar múltiples intentos fallidos para cada usuario candidato
2. Identificar qué usuario provoca un **bloqueo de cuenta**
3. Confirmar que ese usuario **existe**
4. Esperar a que expire el bloqueo
5. Realizar **fuerza bruta de contraseña**

Este tipo de vulnerabilidad aparece cuando:

* El bloqueo solo se aplica a **cuentas reales**
* Los mensajes de error **no son completamente uniformes**
* El estado de bloqueo es **observable desde el exterior**

***

### 📝 Práctica

#### 1️⃣ Captura de la petición de login

Accedemos al formulario de login y probamos una combinación aleatoria, por ejemplo:

```
admin : 123
```

Interceptamos la petición con **Burp Suite**.

Ejemplo de petición:

```http
POST /login
username=admin&password=123
```

La enviamos a **Burp Intruder**.

***

#### 2️⃣ Configuración del ataque de enumeración

Seleccionamos el tipo de ataque:

```
Cluster bomb
```

En **Positions** marcamos:

```
username=§administrator§
password=§password123§
```

***

#### 3️⃣ Carga de payloads

Configuramos dos conjuntos de payloads.

**Payload set 1 – usernames**

Cargamos la lista de **nombres de usuario candidatos** proporcionada por el laboratorio.

**Payload set 2 – passwords**

Cargamos la lista de **contraseñas candidatas**.

Esto hace que Intruder pruebe **todas las combinaciones usuario/contraseña**.

***

#### 4️⃣ Detectar el bloqueo

En **Settings → Grep – Extract** añadimos el texto del mensaje de error típico:

```
Invalid username or password
```

Esto permite identificar fácilmente las respuestas que **no contienen ese mensaje**.

***

#### 5️⃣ Identificación del usuario válido

Lanzamos el ataque con Intruder.

<figure><img src="../../../.gitbook/assets/image (178).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../../.gitbook/assets/image (179).png" alt=""><figcaption></figcaption></figure>

Al analizar los resultados observamos que:

* La mayoría de usuarios responden siempre con el mismo mensaje
* Uno de ellos empieza a devolver respuestas diferentes

Por ejemplo:

```
You have made too many incorrect atemps
```

o una longitud de respuesta distinta.

Esto indica que la cuenta **existe y ha sido bloqueada**.

Ese es el **usuario válido**.

<figure><img src="../../../.gitbook/assets/image (180).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Fuerza bruta de la contraseña

Esperamos a que expire el bloqueo temporal.

Creamos un nuevo ataque en Intruder.

Configuración:

Tipo de ataque:

```
Sniper
```

Fijamos el usuario válido:

```
username=usuario_encontrado
```

Marcamos solo el parámetro password:

```
password=§123456§
```

Cargamos nuevamente la lista de **contraseñas candidatas**.

<figure><img src="../../../.gitbook/assets/image (183).png" alt=""><figcaption></figcaption></figure>

***

#### 7️⃣ Login exitoso

Lanzamos el ataque.

<figure><img src="../../../.gitbook/assets/image (182).png" alt=""><figcaption></figcaption></figure>

En los resultados aparece una respuesta que:

* No contiene el mensaje de error
* Devuelve un **302 Redirect**
* O tiene una longitud diferente

Esto indica que la contraseña es correcta.

Iniciamos sesión con:

```
usuario: usuario_encontrado
contraseña: contraseña_descubierta
```

Accedemos a **My account**.

<figure><img src="../../../.gitbook/assets/image (184).png" alt=""><figcaption></figcaption></figure>

***

#### 8️⃣ Resultado

Se consigue:

* Enumerar un **usuario válido**
* Descubrir su **contraseña mediante fuerza bruta**
* Acceder a su cuenta

✅ **Laboratorio resuelto.**
