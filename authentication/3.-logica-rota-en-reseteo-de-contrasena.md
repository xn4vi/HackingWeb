# 3. Lógica Rota en Reseteo de Contraseña

## 🔓 Lógica rota en reseteo de contraseña

### 📄 Descripción del laboratorio

La funcionalidad de restablecimiento de contraseña de este laboratorio contiene un **fallo lógico crítico** que permite cambiar la contraseña de cualquier usuario, incluyendo `carlos`, sin necesidad de poseer su token de recuperación.

🎯 **Objetivo del laboratorio:**

* Restablecer la contraseña de `carlos`
* Iniciar sesión con su cuenta
* Acceder a **Mi cuenta**

Credenciales propias:

```
wiener:peter
```

Usuario víctima:

```
carlos
```

***

### 📚 Teoría

En un flujo correcto de reseteo de contraseña, el backend debería:

* Generar un token único asociado a un usuario concreto
* Enviar ese token por email
* Verificar que el token sea válido
* Verificar que el token pertenezca al usuario cuya contraseña se va a modificar

El problema en este laboratorio es que el servidor **no vincula el token al usuario**.

Solo comprueba que el token existe y es válido, pero confía en el parámetro `username` enviado por el cliente.

Esto permite el siguiente ataque:

* Usar un token válido, por ejemplo el nuestro
* Cambiar el parámetro `username` por `carlos`
* Enviar la petición modificada
* El servidor acepta la operación y cambia la contraseña de `carlos`

***

### 📝 Práctica

#### 1️⃣ Obtener un token válido

Accedemos a la funcionalidad **Forgot password** e introducimos nuestro usuario:

```
wiener
```

Recibimos un email con el enlace de reseteo y hacemos clic en él.

Interceptamos la petición con Burp Suite.

La URL contiene algo similar a:

```
/forgot-password?temp-forgot-password-token=TOKEN
```

<figure><img src="../../../.gitbook/assets/image (155).png" alt=""><figcaption></figcaption></figure>

***

#### 2️⃣ Interceptar la petición de cambio

La página carga un formulario para introducir la nueva contraseña.

Rellenamos el formulario normalmente e interceptamos la petición `POST` que envía el cambio de contraseña.

La enviamos a **Repeater**.

La petición contiene parámetros como:

```
temp-forgot-password-token=TOKEN
username=wiener
new-password1=...
new-password2=...
```

<figure><img src="../../../.gitbook/assets/image (156).png" alt=""><figcaption></figcaption></figure>

***

#### 3️⃣ Modificar el usuario objetivo

Modificamos la petición:

* Cambiamos `username=wiener` por `username=carlos`
* El token se puede dejar tal cual o vacío, ya que ambas variantes funcionan en este laboratorio

Enviamos la petición modificada.

<figure><img src="../../../.gitbook/assets/image (157).png" alt=""><figcaption></figcaption></figure>

El servidor la acepta y restablece la contraseña de `carlos`.

***

#### 4️⃣ Acceso con la cuenta comprometida

Ahora vamos al login normal del sitio e iniciamos sesión con:

```
Usuario: carlos
Contraseña: la que acabamos de establecer
```

Accedemos correctamente a su cuenta y entramos en **Mi cuenta**.

<figure><img src="../../../.gitbook/assets/image (158).png" alt=""><figcaption></figcaption></figure>

***

#### 5. Resultado

Se consigue:

* Cambiar la contraseña de `carlos`
* Iniciar sesión con su cuenta
* Acceder a **Mi cuenta**

✅ **Laboratorio resuelto.**
