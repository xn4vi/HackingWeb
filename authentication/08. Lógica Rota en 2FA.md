# 🔓 Lógica rota en 2FA

## 📄 Descripción del laboratorio

La autenticación de dos factores de este laboratorio es vulnerable debido a una **lógica defectuosa en la verificación del código 2FA**.

Aunque el sistema genera códigos únicos por usuario, el backend **no los vincula correctamente a la sesión que los valida**.

🎯 **Objetivo del laboratorio:**

* Obtener acceso a la cuenta de `carlos`
* Acceder a su página **My account**

Credenciales disponibles:

```
Usuario propio: wiener:peter
Usuario víctima: carlos
```

También se dispone de acceso al **servidor de correo** para recibir códigos 2FA.



## 📚 Teoría

En un sistema correctamente implementado, el flujo de autenticación con **2FA** debería ser:

1. El usuario introduce credenciales válidas
2. El servidor genera un **código 2FA asociado a esa sesión**
3. El endpoint de verificación valida:
   * el código
   * el usuario
   * la sesión que solicitó el código

En este laboratorio esto no ocurre.

El endpoint de verificación acepta un parámetro que indica **para qué usuario se está verificando el código**, por ejemplo:

```
verify=username
```

El servidor **no comprueba** que:

* el código pertenezca al usuario autenticado
* la sesión que valida el código sea la misma que lo solicitó

Esto permite:

* Generar un código 2FA válido para `carlos`
* Utilizar nuestra propia sesión para verificarlo
* Realizar **fuerza bruta del código 2FA** sin restricciones

El problema principal es que:

* El código **no está ligado a la sesión**
* El backend confía en un **parámetro controlado por el cliente**



## 📝 Práctica

### 1️⃣ Análisis del flujo de login

Iniciamos sesión con nuestras credenciales:

```
wiener : peter
```

Después del login, el sistema solicita un **código 2FA**, que recibimos por correo.

Interceptamos la petición de verificación del código con **Burp Suite**.

![](../images/aut35.png)<br>
La petición suele contener algo similar a:

```
mfa-code=XXXX
verify=wiener
```

El parámetro `verify` indica **para qué usuario se está validando el código**.

Este parámetro puede encontrarse en:

* el cuerpo de la petición
* una cookie
* un campo oculto
* un header



### 2️⃣ Preparar el ataque de fuerza bruta

Repetimos el flujo de login como `wiener` y capturamos la petición de verificación del código.

La enviamos a **Burp Intruder**.

En **Positions** marcamos el campo del código:

```
mfa-code=§0000§
```

Generamos una lista de posibles códigos:

```
0000 → 9999
```

![](../images/aut36.png)


### 3️⃣ Modificar el usuario objetivo

En la petición base de Intruder cambiamos el parámetro:

```
verify=carlos
```

Esto hace que estemos verificando el código **para la cuenta de carlos**.



### 4️⃣ Configurar detección de errores

En **Settings → Grep – Match** añadimos el mensaje de error típico:

```
Incorrect security code
```

Esto permite identificar fácilmente cuando una respuesta **no contiene el mensaje de fallo**.

![](../images/aut37.png)


### 5️⃣ Generar el código válido

Antes de lanzar el ataque, enviamos una petición manual con:

```
verify=carlos
```

y cualquier código.

Esto fuerza al servidor a **generar un código 2FA válido para carlos**.



### 6️⃣ Fuerza bruta del código 2FA

Lanzamos el ataque en Intruder.

Entre las respuestas aparecerá una que:

* No contiene el mensaje **Incorrect security code**
* Devuelve un **302 Redirect**
* O carga directamente **My account**

Ese es el **código 2FA correcto de carlos**.

![](../images/aut38.png)


### 7️⃣ Toma de la sesión

Abrimos la respuesta exitosa y copiamos la cookie devuelta por el servidor:

```
Set-Cookie: session=XXXXXXXX
```

![](../images/aut39.png)<br>
En el navegador sustituimos nuestra cookie actual por la nueva.

Accedemos directamente a:

```
/my-account
```

![](../images/aut40.png)


### 8️⃣ Resultado

Se consigue:

* Descubrir el **código 2FA de carlos**
* Secuestrar su **sesión autenticada**
* Acceder completamente a su cuenta

✅ **Laboratorio resuelto.**
