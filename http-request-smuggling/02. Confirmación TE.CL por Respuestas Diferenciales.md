# 📥 Confirmación de TE.CL por respuestas diferenciales

## 📄 Descripción del laboratorio

Este laboratorio presenta una arquitectura con servidor front-end y servidor back-end, donde el **back-end no admite codificación por fragmentos (chunked encoding)**, mientras que el **front-end sí la permite**.

El objetivo del laboratorio es **enviar una petición de forma encubierta al back-end**, de modo que una **petición posterior a `/` (raíz)** provoque una **respuesta 404**, confirmando la existencia de una vulnerabilidad de HTTP Request Smuggling.



## 📚 Teoría

En este laboratorio se trabaja con una variante de **HTTP Request Smuggling de tipo TE.CL**, en la que:

* El **front-end prioriza la cabecera `Transfer-Encoding`**
* El **back-end interpreta la petición basándose en `Content-Length`**

Esta discrepancia permite provocar una **desincronización entre ambos servidores**, haciendo que parte de una petición quede pendiente y se combine con solicitudes posteriores.

Mediante **envíos consecutivos desde Burp Repeater**, se consigue que la segunda petición se vea afectada por los restos de la primera, generando una **respuesta 404 inesperada en la raíz del sitio**.

Este comportamiento diferencial confirma la presencia del fallo y valida el escenario como vulnerable.



## 📝 Práctica

Para resolver el laboratorio, debemos conseguir que una petición a `/` devuelva un **error 404** como consecuencia de una desincronización previa.

Interceptamos la petición a la home (`/`) y realizamos los siguientes cambios:

* Cambiamos el método a **POST**
* Forzamos **HTTP/1.1**
* Desactivamos la actualización automática del **`Content-Length`**
* Eliminamos cabeceras innecesarias

### Forzando la desincronización



A continuación, manipulamos el **`Content-Length`** para indicar un valor mayor al número real de bytes enviados.

En este caso:

* El **back-end espera 15 bytes**
* Solo recibe **13 bytes**

Esto provoca que el back-end **quede esperando los bytes restantes**, generando una desincronización entre ambos servidores:

### Inyección de la petición smuggleada



El siguiente paso consiste en **inyectar una petición POST hacia un recurso inexistente**, con el objetivo de provocar un 404.

La solicitud se construye de la siguiente forma:

El back-end interpreta el **`Content-Length` como válido**, procesa los primeros bytes y deja el inicio de la nueva solicitud **en cola**, a la espera de la siguiente petición.

En algunos casos, el back-end puede interpretar todo como una única solicitud.

Para evitarlo, es necesario **construir una estructura HTTP válida**, asegurando que el back-end pueda separar correctamente ambas peticiones:

<br>

Al **inflar el `Content-Length` de la segunda solicitud**, el back-end queda nuevamente esperando más datos.

Cuando se envía la siguiente petición, parte de esta se concatena con la anterior, provocando finalmente la **respuesta 404** esperada.
