# 📥 Smuggling H2.CL

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **HTTP request smuggling** porque el servidor front-end degrada las solicitudes **HTTP/2 a HTTP/1.1** incluso cuando presentan una longitud ambigua.

El objetivo es realizar un ataque **H2.CL (HTTP/2 → Content-Length)** para provocar que el navegador de la víctima cargue y ejecute un archivo JavaScript malicioso desde el **exploit server**, ejecutando `alert(document.cookie)`.\
El usuario víctima accede a la **home cada 10 segundos**.



## 📚 Teoría

En este laboratorio explotamos un ataque de **request smuggling H2.CL**, manipulando la cabecera **Content-Length** en una petición HTTP/2 degradada.

La técnica consiste en enviar una petición con un **Content-Length malicioso** que permita incrustar parte de la siguiente solicitud del usuario víctima. Aprovechamos que la aplicación realiza redirecciones automáticas al recurso **/resources**, lo que nos permite forzar que el navegador de la víctima cargue contenido desde un **host controlado por el atacante**.

Mediante el **envenenamiento de la cola de respuestas**, conseguimos que la petición de la víctima sea redirigida al exploit server, donde se ejecuta el payload JavaScript `alert(document.cookie)`.

El ataque depende del **timing**, ya que es necesario que la petición smuggleada coincida con el acceso periódico del usuario víctima.



## 📝 Práctica

El objetivo es que a la víctima se le muestre un **alert con su cookie de sesión**. La víctima accede a la página principal cada **10 segundos**.

Analizando el tráfico de la aplicación, observamos que se utiliza con frecuencia el recurso **/resources**.

Interceptamos una petición a la **home**, la enviamos al repeater, cambiamos el método a **POST**, quitamos el **Content-Length automático** y eliminamos lo innecesario:
