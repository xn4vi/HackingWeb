# 📥 Smuggling ocultando el header TE

## 📄 Descripción del laboratorio

Este laboratorio presenta un escenario de **request smuggling** basado en la interpretación distinta de **cabeceras duplicadas** entre el servidor front-end y el back-end.

El objetivo es introducir una solicitud en el back-end de forma que la siguiente petición sea interpretada como si utilizara el método **GPOST**, resolviendo así el laboratorio.



## 📚 Teoría

En este caso explotamos una diferencia de comportamiento entre el front-end y el back-end al procesar **cabeceras HTTP duplicadas**, concretamente el encabezado `Transfer-Encoding`.

El front-end valida las peticiones y solo permite métodos **GET** o **POST**, mientras que el back-end procesa la solicitud real basándose en su propia interpretación de las cabeceras.

Cuando existen dos cabeceras iguales:

* El **front-end** se queda con la **primera**
* El **back-end** se queda con la **última**

Aprovechando esta discrepancia, es posible ocultar el uso de `Transfer-Encoding`, forzar al back-end a usar `Content-Length` y dejar parte de la petición pendiente, lo que permite smugglear una solicitud que será interpretada como **GPOST**.



## 📝 Práctica

El laboratorio nos indica explícitamente que existen problemas con **cabeceras duplicadas**.

Interceptamos la petición a la home y la enviamos al Repeater. Realizamos los siguientes cambios:

* Cambiamos el método a **POST**
* Cambiamos el protocolo a **HTTP/1.1**
* Eliminamos cabeceras innecesarias
* Quitamos el **Content-Length** automático

Para comprobar si el front-end utiliza `Transfer-Encoding`, realizamos una prueba clásica sustituyendo el valor `0` por una **X**, provocando un error. Esto confirma que el front-end interpreta la cabecera `TE` :

A continuación analizamos el comportamiento del back-end. Intentamos provocar una desincronización inflando el `Content-Length`, pero no ocurre nada, lo que indica que el back-end **también interpreta TE:**

<img width="1520" height="281" alt="image" src="https://github.com/user-attachments/assets/1770bc18-0a96-4431-b124-51152b201de9" />
<br>

La clave está en **duplicar la cabecera `Transfer-Encoding`**.

El laboratorio está configurado para que:

* El **front-end** use la **primera cabecera**
* El **back-end** use la **última cabecera**

Al hacer que la última cabecera tenga un valor inválido, el back-end ignora `TE` y pasa a usar `Content-Length`.\
Esto provoca que solo lea **4 bytes**, dejando pendiente el resto de la petición, donde se encuentra el método **GPOST** smuggleado:

<br>
