# 📥 Response queue poisoning con H2-TE

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **HTTP request smuggling** debido a que el servidor front-end degrada las solicitudes **HTTP/2 a HTTP/1.1** incluso cuando presentan una longitud ambigua.

El objetivo es eliminar al usuario **carlos** accediendo al panel **/admin** mediante **response queue poisoning**.\
Un usuario administrador inicia sesión aproximadamente cada **15 segundos**.

La conexión con el back-end se restablece cada **10 solicitudes**, por lo que los errores pueden corregirse enviando varias peticiones normales.



## 📚 Teoría

Este laboratorio demuestra un ataque de **HTTP request smuggling H2.TE (HTTP/2 → Transfer-Encoding)** que permite provocar una **desincronización en la cola de respuestas** entre el front-end y el back-end.

La técnica consiste en inyectar una petición **HTTP/1.1 completa** dentro del cuerpo de una petición **HTTP/2** utilizando la cabecera **Transfer-Encoding: chunked**. Esto provoca que la respuesta asociada a la petición smuggleada se **adelante en la cola de respuestas**.

Cuando el administrador inicia sesión, su respuesta queda desalineada y es entregada al atacante, permitiendo capturar su **cookie de sesión**. Con esta cookie se accede al panel **/admin** y se elimina al usuario carlos.

Este escenario combina **downgrade de protocolo**, **response queue poisoning** y **secuestro de sesión**.



## 📝 Práctica

HTTP/2 implica un **downgrade de protocolo**, donde las peticiones llegan al front-end como HTTP/2 pero se reenvían al back-end como HTTP/1.1.

Sabemos que un administrador inicia sesión cada **15 segundos**, por lo que el objetivo es capturar su respuesta mediante una petición smuggleada.

Interceptamos una petición a la **home**, cambiamos el método a **POST** y dejamos únicamente lo necesario. En HTTP/2 no es obligatorio indicar **Content-Length**.

Añadimos la cabecera **Transfer-Encoding: chunked** e inyectamos una nueva petición **GET** dentro del cuerpo:
