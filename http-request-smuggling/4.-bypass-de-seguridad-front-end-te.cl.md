# 4. Bypass de Seguridad Front-end TE.CL

## 📥 Bypass de seguridad front-end TE.CL

### 📄 Descripción del laboratorio

Este laboratorio presenta una arquitectura con servidor front-end y servidor back-end, donde el **back-end no soporta codificación por fragmentos (chunked encoding)**. Existe un panel de administración accesible en `/admin`, pero el front-end bloquea el acceso directo a dicha ruta.

El objetivo del laboratorio es **enviar una petición directamente al back-end** que permita acceder al panel de administración y **eliminar al usuario `carlos`**, evadiendo los controles de seguridad impuestos por el front-end.

***

### 📚 Teoría

En este laboratorio se explota una vulnerabilidad de tipo **TE.CL (Transfer-Encoding + Content-Length)** para llevar a cabo un ataque de **HTTP Request Smuggling**.

Este tipo de vulnerabilidad ocurre cuando:

* El front-end prioriza la cabecera `Transfer-Encoding`
* El back-end interpreta la petición basándose en `Content-Length`

Aprovechando esta discrepancia, es posible **inyectar una segunda petición HTTP** dentro del cuerpo de la primera, de forma que:

* El front-end solo valida la petición externa
* El back-end procesa una petición interna adicional que el front-end nunca inspecciona

Esto permite **evadir controles de acceso front-end** y acceder a rutas restringidas como `/admin` o `/admin/delete`.

Durante la explotación:

* Se construye una petición con codificación `chunked` malformada
* Se ajusta el **`Content-Length`** para que el back-end lea bytes adicionales
* Se incluye una petición HTTP embebida con la cabecera **`Host: localhost`**, requisito necesario para que el back-end acepte acciones administrativas

Finalmente, se modifica la ruta de la petición smuggled para ejecutar la acción de **eliminación del usuario `carlos`**, completando así el laboratorio con éxito.

***

### 📝 Práctica

Para resolver el laboratorio, es necesario **acceder al panel de administración** y eliminar al usuario Carlos.

Como en laboratorios anteriores, interceptamos la petición a la raíz (`/`) y realizamos las siguientes modificaciones:

* Cambiamos el método a POST
* Forzamos HTTP/1.1
* Eliminamos el **`Content-Length` inicial** para evitar que Burp lo recalculé automáticamente

El objetivo es dejar una **petición mínima y completamente controlada** para poder manipular manualmente las cabeceras.

#### Verificación del uso de Transfer-Encoding



Primero, comprobamos si el front-end utiliza `Transfer-Encoding` :

<figure><img src="../../../.gitbook/assets/image (59).png" alt=""><figcaption></figcaption></figure>

Al modificar el valor final del chunk (`0`) por un carácter inválido (`X`), el servidor responde indicando que la petición es incorrecta. Esto confirma que el front-end está procesando la petición como chunked, lo que valida que el escenario es vulnerable a **TE.CL**.

#### Construcción de la petición smuggled



El siguiente paso consiste en **inyectar una segunda petición HTTP** hacia `/admin` dentro del cuerpo de la petición principal.

Para ello:

* Se mantiene `Transfer-Encoding: chunked`
* Se añade un **`Content-Length` ligeramente inflado**
* El tamaño inflado permite que el back-end interprete parte de la siguiente petición como datos válido

<figure><img src="../../../.gitbook/assets/image (60).png" alt=""><figcaption></figcaption></figure>

La petición se envía dos veces.

En la segunda solicitud, el back-end ya ha desincronizado el flujo HTTP y responde con el panel de administración, confirmando que el **request smuggling** ha sido exitoso.

#### Eliminación del usuario Carlos



Una vez confirmado el acceso al panel de administración, se ajusta la petición smuggled para apuntar directamente a la ruta de borrado del usuario.

<figure><img src="../../../.gitbook/assets/image (61).png" alt=""><figcaption></figcaption></figure>

La petición interna ahora ejecuta la acción administrativa `/admin/delete?username=carlos`, la cual es procesada correctamente por el back-end.
