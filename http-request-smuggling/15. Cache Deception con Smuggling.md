# 📥 Decepción de caché web con smuggling

## 📄 Descripción del laboratorio

Este laboratorio combina **HTTP request smuggling** con **web cache deception**.

El servidor front-end no soporta `Transfer-Encoding` y almacena en caché recursos estáticos, lo que nos permite forzar que contenido sensible de un usuario se guarde en caché y sea accesible por terceros.

El objetivo es conseguir que la **API Key del usuario víctima** quede cacheada y posteriormente recuperarla para completar el laboratorio.



## 📚 Teoría

En este escenario explotamos una desincronización entre front-end y back-end para engañar al sistema de caché.

El front-end interpreta `Content-Length` y cachea recursos considerados estáticos, mientras que el back-end interpreta `Transfer-Encoding`.

Aprovechando esta diferencia, smuggleamos una petición hacia `/my-account` de forma que, cuando el usuario víctima navega por la aplicación, su solicitud autenticada se acopla a nuestra petición.

Como resultado, la respuesta de `/my-account`, que contiene la **API Key**, se almacena en caché bajo un recurso estático.

Posteriormente, accedemos a ese recurso cacheado y extraemos la clave de la víctima.



## 📝 Práctica

El objetivo es lograr que la **API Key de la víctima** quede cacheada y sea accesible.

Iniciamos sesión con las credenciales proporcionadas (`wiener:peter`) y observamos el panel de usuario, donde se muestra nuestra propia API Key.

A continuación, interceptamos una petición normal a la home y la enviamos al Repeater. Realizamos los siguientes cambios:

* Cambiamos el método a **POST**
* Cambiamos a **HTTP/1.1**
* Eliminamos cabeceras innecesarias
* Quitamos el `Content-Length` automático

Sabemos que el front-end interpreta `Content-Length` y el back-end `Transfer-Encoding`. Para confirmarlo, smuggleamos una petición `GET /error` y observamos si se produce un error de desincronización:
