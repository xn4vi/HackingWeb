# 💉 Inyección ciega con redirección de salida

## 📄 Descripción del laboratorio

La funcionalidad **Submit feedback** es vulnerable a **inyección de comandos del sistema operativo ciega**: el comando se ejecuta en el servidor, pero la salida no se refleja en la respuesta HTTP.

Sin embargo, existe un directorio escribible y accesible públicamente:

```
/var/www/images/
```

La aplicación sirve el contenido de este directorio mediante una URL pública. Esto permite **redirigir la salida de comandos a un archivo y posteriormente recuperarlo vía HTTP**.

🎯 **Objetivo del laboratorio:**

* Ejecutar el comando `whoami`
* Redirigir su salida a un archivo en `/var/www/images/`
* Recuperar la salida accediendo al archivo por la URL pública

 

## 📚 Teoría

En **OS Command Injection ciega**, la salida del comando no se muestra en la respuesta HTTP. Para obtener información es necesario utilizar **canales alternativos de exfiltración**.

Métodos comunes:

* ⏱️ **Time-based** (`sleep`, `ping`)
* 🌐 **Out-of-band** (DNS/HTTP)
* 📝 **Redirección de salida a archivos**

En este laboratorio se utiliza **redirección de salida** hacia un archivo dentro de un directorio público.

Ejemplo:

```bash
whoami > /var/www/images/output.txt
```

Si:

* El proceso del servidor tiene **permisos de escritura**
* El directorio se **sirve públicamente**

entonces es posible **recuperar el resultado del comando accediendo al archivo mediante HTTP**.

 

## 📝 Práctica

### 1️⃣ Interceptar el feedback

Accedemos a **Submit feedback**.

Enviamos un comentario de prueba e interceptamos la petición **POST** con **Burp Suite**.

<img width="768" height="446" alt="image" src="https://github.com/user-attachments/assets/9d917b9f-1b8f-4a09-9bf4-37e9cb680977" />
<br>

La petición contiene varios parámetros, por ejemplo:

* `name`
* `email`
* `subject`
* `message`

 

### 2️⃣ Identificar el parámetro vulnerable

Probamos cada parámetro para determinar cuál se concatena al comando del sistema.

Inyectamos una prueba en el parámetro `email`:

```
email=test@test.com;whoami
```

<img width="768" height="286" alt="image" src="https://github.com/user-attachments/assets/2499388c-6e98-415c-aa86-aea13921d700" />
<br>

No se observa salida en la respuesta HTTP, lo que confirma que se trata de una **inyección ciega**, aunque el comando se ejecuta.

 

### 3️⃣ Redirección de salida a archivo

Para recuperar la salida del comando, redirigimos el resultado a un archivo dentro del directorio público:

```bash
email=test;whoami>/var/www/images/test.txt
```

Enviamos la petición modificada.

<img width="768" height="430" alt="image" src="https://github.com/user-attachments/assets/2b070816-bba6-4578-944a-3530f5f78306" />

 

### 4️⃣ Recuperar el archivo por HTTP

Accedemos al archivo generado utilizando la ruta pública de imágenes:

```
https://ID-DEL-LAB.web-security-academy.net/image?filename=test.txt
```

 

### 5️⃣ Resultado final

La respuesta HTTP muestra el contenido del archivo:

```
peter-EGEsp
```

<br>

Esto confirma:

* Ejecución del comando `whoami`
* Escritura exitosa en `/var/www/images/`
* Exfiltración de la salida mediante HTTP
