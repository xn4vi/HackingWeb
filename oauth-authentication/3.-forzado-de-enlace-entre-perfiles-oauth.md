# 3. Forzado de Enlace entre Perfiles OAuth

## 🔁 **Forzado de enlace entre perfiles OAuth**

### 📄 **Descripción del laboratorio**

Este laboratorio permite a los usuarios vincular un perfil de redes sociales a su cuenta local para iniciar sesión mediante OAuth.

Debido a una implementación insegura del flujo OAuth por parte de la aplicación cliente, es posible **forzar el enlace de un perfil social controlado por el atacante a la cuenta de otro usuario**, en este caso el **administrador**.

🎯 **Objetivo:**\
Vincular nuestro perfil social a la cuenta del administrador mediante un ataque CSRF, iniciar sesión como administrador y eliminar al usuario **carlos**.

Credenciales disponibles:

* Cuenta del blog:

```
wiener : peter
```

* Perfil de redes sociales:

```
peter.wiener : hotdog
```

El administrador:

* Siempre tiene una sesión activa en el blog
* Abrirá cualquier contenido servido desde el Exploit Server

***

### 📚 Teoría

Este laboratorio ilustra un ataque conocido como **Forced OAuth Profile Linking**, una variante muy peligrosa de los fallos OAuth relacionados con CSRF.

La aplicación permite que un usuario ya autenticado **vincule un perfil social** a su cuenta local. Para ello, tras completar la autenticación en el proveedor OAuth, el usuario es redirigido a un endpoint del tipo:

```
/oauth-linking?code=AUTHORIZATION_CODE
```

Ese `code` es un **código de autorización OAuth válido**, que el servidor intercambia por un token de acceso y usa para asociar el perfil social a la cuenta que tenga la sesión activa.

El fallo crítico es el siguiente:

* El endpoint de vinculación **no valida el parámetro `state`**
* No existe ninguna protección CSRF
* El servidor **no comprueba que el código OAuth haya sido generado por el mismo usuario que va a vincular la cuenta**

Esto permite que un atacante:

1. Obtenga un código OAuth válido usando **su propio perfil social**
2. Haga que **otro usuario con sesión activa** consuma ese código
3. Vincule así el perfil social del atacante a la cuenta de la víctima

Si la víctima es el administrador, el impacto es **toma completa de cuenta**.

***

### 📝 Práctica

#### 1️⃣ Acceso inicial y análisis de la funcionalidad

Nos logueamos normalmente en el blog con:

```
wiener : peter
```

Accedemos a nuestro panel de cuenta y vemos la opción **Attach social profile**, que permite vincular un perfil de redes sociales.

<figure><img src="../../../.gitbook/assets/image (241).png" alt=""><figcaption></figcaption></figure>

#### 2️⃣ Interceptar el flujo de enlace OAuth

Pulsamos **Attach social profile** y usamos las credenciales del perfil social:

```
peter.wiener : hotdog
```

Interceptamos todo el flujo con Burp Suite.\
Tras autenticarnos correctamente en el proveedor OAuth, observamos una petición clave:

```
GET /oauth-linking?code=XXXXXXXX
```

📌 Este endpoint es el que **realiza la vinculación** del perfil social con la cuenta local que tenga la sesión activa.

<figure><img src="../../../.gitbook/assets/image (243).png" alt=""><figcaption></figcaption></figure>

***

#### 3️⃣ Entender el vector de ataque

Observaciones importantes:

* El parámetro `code`:
  * Es válido
  * Es de un solo uso
* El endpoint **no verifica**:
  * Quién inició el flujo OAuth
  * Si el navegador que consume el `code` es el mismo que lo solicitó

Como el administrador:

* Tiene una sesión activa
* Abrirá cualquier URL que le enviemos

Podemos forzar que **él consuma nuestro `code` OAuth**.

***

#### 4️⃣ Preparar el código OAuth del atacante

Repetimos el flujo **Attach social profile** una segunda vez, pero esta vez:

* Interceptamos la petición final:

```
/oauth-linking?code=...
```

* **NO dejamos que se envíe**
* Clic derecho → **Copy URL**
* Guardamos la URL completa con el `code` válido

***

#### 5️⃣ Construir el exploit CSRF

Nos vamos al **Exploit Server** y creamos un exploit sencillo usando un iframe:

```html
<iframe src="https://ID-LAB.web-security-academy.net/oauth-linking?code=XXXXXXXX"></iframe>
```

Este iframe forzará al navegador de la víctima a cargar automáticamente el endpoint vulnerable.

Pulsamos:

* **Store**
* **Deliver exploit to victim**

<figure><img src="../../../.gitbook/assets/image (244).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Forzar el enlace en la cuenta del administrador

El laboratorio simula que el **administrador** visita nuestro exploit:

* Su navegador carga el iframe
* Se envía la petición a `/oauth-linking?code=...`
* La petición incluye **las cookies de sesión del administrador**
* El servidor vincula **nuestro perfil social** a la cuenta del administrador

✔️ Perfil social del atacante → cuenta admin

***

#### 7️⃣ Acceder como administrador

Ahora:

1. Cerramos sesión
2. Vamos a la página de login
3. Elegimos **Login with social media**
4. Introducimos:

```
peter.wiener : hotdog
```

El servidor reconoce el perfil social y lo asocia a la cuenta **administrator**.

📌 Accedemos directamente como administrador.

<figure><img src="../../../.gitbook/assets/image (245).png" alt=""><figcaption></figcaption></figure>

***

#### 8️⃣ Eliminar a carlos

Accedemos al panel de administración:

```
/admin
```

Buscamos al usuario **carlos** y lo eliminamos.

El laboratorio detecta la acción y se resuelve.

<figure><img src="../../../.gitbook/assets/image (246).png" alt=""><figcaption></figcaption></figure>
