# 8. WCP mediante Fat GET Request

## 🧪 Web Cache Poisoning mediante Fat GET request

### 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning** porque el servidor acepta **peticiones GET con cuerpo (body)**, pero el sistema de caché **no incluye ese body en la cache key**.

Esto crea una discrepancia entre:

* Lo que el **backend utiliza para generar la respuesta**
* Lo que la **caché utiliza para identificar una respuesta almacenada**

Un usuario visita regularmente la página de inicio del sitio utilizando **Chrome**.

El objetivo del laboratorio es **envenenar la caché con una respuesta que ejecute `alert(1)` en el navegador de la víctima**.

***

### 📚 Teoría

En este laboratorio se explota una técnica conocida como **Fat GET request**.

Esta técnica se basa en una inconsistencia frecuente en algunas arquitecturas web:

* El **backend procesa el cuerpo de una petición GET**.
* La **caché ignora completamente ese cuerpo al construir la cache key**.

Según el estándar HTTP, las peticiones GET normalmente **no deberían tener body**, pero algunos servidores lo aceptan igualmente. Si el backend utiliza esos datos para generar la respuesta y la caché no los tiene en cuenta, se crea una situación vulnerable a **Web Cache Poisoning**.

#### 📌 Recurso vulnerable

El recurso vulnerable en este laboratorio es:

```
/js/geolocate.js
```

Este script utiliza un parámetro llamado:

```
callback
```

El valor de `callback` determina **qué función JavaScript se ejecutará en el navegador**.

La caché construye su clave utilizando el valor del parámetro en la URL. Sin embargo, el backend también acepta un **segundo valor del parámetro `callback` enviado en el body del GET**.

Esto produce el siguiente comportamiento:

* La caché cree que la respuesta corresponde a la URL legítima.
* El backend genera el script usando el valor malicioso del body.
* La respuesta manipulada queda **almacenada en caché**.

Cuando otros usuarios cargan el recurso, reciben el **JavaScript envenenado**.

***

### 📝 Práctica

#### 1️⃣ Identificar el recurso vulnerable

Interceptamos la carga de la página principal y observamos que se solicita el recurso:

```
/js/geolocate.js
```

Enviamos la petición a **Burp Repeater** para analizar su comportamiento.

***

#### 2️⃣ Confirmar el uso del parámetro callback

Probamos primero una petición con un valor controlado para confirmar que el servidor procesa el parámetro.

Petición utilizada:

```http
GET /js/geolocate.js?callback=setCountryCookie HTTP/1.1
Host: laboratorio

callback=sivenga
```

<figure><img src="../../../.gitbook/assets/image (1450).png" alt=""><figcaption></figcaption></figure>

Observamos que la respuesta cambia, lo que indica que el servidor **procesa el valor enviado en el body**.

***

#### 3️⃣ Construir una Fat GET request

Ahora construimos una **petición GET con body**, algo que la caché no tiene en cuenta.

Petición utilizada:

```http
GET /js/geolocate.js?callback=setCountryCookie HTTP/1.1
Host: laboratorio
Content-Type: application/x-www-form-urlencoded
Content-Length: 18

callback=alert(1)
```

<figure><img src="../../../.gitbook/assets/image (1451).png" alt=""><figcaption></figcaption></figure>

En esta petición ocurre lo siguiente:

* La caché utiliza únicamente la URL (`callback=setCountryCookie`) para generar la clave.
* El servidor procesa el valor `callback=alert(1)` enviado en el body.
* El script generado incluye la llamada `alert(1)`.

***

#### 4️⃣ Confirmar el envenenamiento de la caché

La respuesta generada queda almacenada en caché.

Podemos confirmarlo observando la cabecera:

```http
X-Cache: hit
```

Esto indica que el contenido malicioso ha sido **almacenado en caché**.

***

#### 5️⃣ Impacto en la víctima

A partir de ese momento, cualquier usuario que cargue la página principal realizará una petición normal al recurso:

```
/js/geolocate.js
```

Aunque esa petición no incluya body, el servidor devolverá la **respuesta envenenada almacenada en caché**.

El navegador ejecutará automáticamente:

```javascript
alert(1)
```

Cuando la víctima accede a la página, el laboratorio detecta la ejecución del payload.
