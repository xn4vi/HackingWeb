# 🧪 Web Cache Poisoning con cookie no indexada

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning** porque ciertas cookies enviadas por el cliente **no se incluyen en la clave de caché (cache key)**.

La aplicación utiliza una cookie que influye en la respuesta generada por el servidor. Sin embargo, la infraestructura de caché **no tiene en cuenta esta cookie al almacenar la respuesta**.

Un usuario legítimo visita regularmente la página principal del sitio, por lo que si conseguimos **envenenar la caché**, podremos hacer que ese usuario reciba una respuesta manipulada.

El objetivo del laboratorio es **envenenar la caché con una respuesta que ejecute `alert(1)` en el navegador de la víctima**.



## 📚 Teoría

En este laboratorio se explora un caso de **Web Cache Poisoning provocado por una cookie no indexada (unkeyed cookie)**.

En muchos sistemas con caché (CDN, reverse proxy o cache server), las respuestas se almacenan usando una **cache key**, que normalmente incluye:

* URL
* parámetros de la query
* algunos headers

Si un valor enviado por el usuario **afecta a la respuesta pero no forma parte de la cache key**, se puede producir un escenario de **cache poisoning**.

### 📌 Concepto clave

El flujo vulnerable es el siguiente:

1. El servidor **lee una cookie enviada por el cliente**.
2. El valor de esa cookie **se refleja directamente en la respuesta**.
3. La caché **no incluye esa cookie en la cache key**.

El resultado es que un atacante puede:

* Modificar el contenido de la respuesta.
* Hacer que esa respuesta maliciosa se **almacene en caché**.
* Provocar que **otros usuarios reciban la respuesta manipulada** sin haber enviado la cookie.

### 📌 El fallo concreto

La aplicación utiliza una cookie llamada:

```
fehost
```

Esta cookie presenta tres problemas:

* Se inserta directamente en una respuesta **JavaScript**.
* No se valida ni se escapa correctamente.
* No forma parte de la **cache key**.

Esto permite **inyectar código JavaScript**, hacer que la respuesta sea cacheada y provocar que ese código se ejecute en el navegador de otros usuarios.



## 📝 Práctica

### 1️⃣ Analizar la petición

Interceptamos la petición a la página principal (`/`) y la enviamos a **Burp Repeater**.

<br>
Observamos que:

* La petición incluye una cookie llamada `fehost`.
* La respuesta contiene **código JavaScript**.
* El valor de la cookie aparece reflejado en ese JavaScript.

Ejemplo conceptual del comportamiento del backend:

```javascript
var host = "VALOR_DE_FEHOST";
```

Esto indica que el valor de la cookie se inserta directamente en el código JavaScript.



### 2️⃣ Confirmar la reflexión

Modificamos la cookie en la petición:

```http
Cookie: fehost=test
```

Enviamos la petición.

<img width="1399" height="439" alt="image" src="https://github.com/user-attachments/assets/d2e551f2-b562-4cdd-9d18-e20acc520a10" />
<br>
En la respuesta observamos que el valor `test` aparece reflejado dentro del código JavaScript.

Esto confirma:

* La cookie es **controlable por el atacante**.
* Existe **reflexión directa** en la respuesta.
* La reflexión ocurre en **contexto JavaScript**.



### 3️⃣ Construir el payload XSS

Como el valor de la cookie se inserta dentro de una **cadena JavaScript**, podemos:

* Cerrar la cadena
* Inyectar código JavaScript
* Reabrir la cadena para evitar errores de sintaxis

Payload utilizado:

```
test"-alert(1)-"asd
```

La cookie completa queda así:

```http
Cookie: fehost=test"-alert(1)-"asd
```

Este payload rompe la estructura original del script y provoca la ejecución de:

```javascript
alert(1)
```

<img width="1398" height="432" alt="image" src="https://github.com/user-attachments/assets/c9fef8ae-b705-4843-aa5d-17396caac1b1" />



### 4️⃣ Envenenar la caché

Enviamos varias veces la petición con la cookie maliciosa hasta observar en la respuesta:

```http
X-Cache: hit
```

Esto indica que:

* La respuesta con el payload ha sido **almacenada en caché**.
* La cookie `fehost` **no forma parte de la cache key**.



### 5️⃣ Impacto en la víctima

A partir de este momento, cualquier usuario que visite la página principal (`/`):

* Aunque **no envíe la cookie `fehost`**
* Recibirá la **respuesta cacheada con el payload**

El navegador del visitante ejecutará automáticamente:

```
alert(1)
```

El laboratorio detecta la ejecución del payload en el navegador de la víctima.
