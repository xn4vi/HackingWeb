# 🧪 Web Cache Poisoning con query string no indexado

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning** porque los parámetros de la **query string** no forman parte de la **cache key**.

La aplicación almacena en caché las respuestas de la página principal (`/`), pero ignora los parámetros añadidos en la URL. Esto significa que múltiples URLs con diferentes query strings pueden compartir **la misma entrada de caché**.

Un usuario víctima visita regularmente la página principal usando **Chrome**, por lo que si conseguimos almacenar una respuesta maliciosa en caché, su navegador recibirá esa versión manipulada.

El objetivo del laboratorio es **envenenar la caché con una respuesta que ejecute `alert(1)` en el navegador de la víctima**.



## 📚 Teoría

En este laboratorio se explora un caso típico de **caché mal configurada**, donde los parámetros de la URL no se incluyen en la clave de caché.

### 📌 Conceptos clave

La **cache key** es el identificador que utiliza el sistema de caché para decidir si una respuesta almacenada puede reutilizarse.

En este caso:

* El **path** (`/`) sí forma parte de la cache key.
* La **query string** (`?test=...`) **no** se tiene en cuenta.

Esto provoca que URLs como:

```
/
/?test=1
/?test=evil
```

compartan exactamente **la misma entrada de caché**.

### 📌 El fallo

El problema se vuelve explotable cuando se cumplen tres condiciones:

1. El parámetro **no forma parte de la cache key**.
2. El valor del parámetro **se refleja en la respuesta**.
3. Es posible **romper el contexto HTML o JavaScript** donde aparece.

Cuando estas condiciones se cumplen, un atacante puede:

* Inyectar contenido malicioso usando la query string.
* Conseguir que la respuesta manipulada se **almacene en caché**.
* Hacer que otros usuarios reciban esa respuesta aunque **no envíen el parámetro**.

Este es el núcleo del **Web Cache Poisoning basado en query strings no indexadas**.



## 📝 Práctica

### 1️⃣ Analizar el comportamiento de la caché

Interceptamos la petición a la página principal:

```http
GET /
```

La enviamos a **Burp Repeater** y observamos cabeceras relacionadas con la caché, como:

```http
X-Cache: hit
Age: XX
```

Probamos a añadir un parámetro arbitrario:

```http
GET /?test=1
```

Observamos que:

* La caché **no se refresca**.
* El valor de **Age continúa aumentando**.

Esto confirma que la **query string no forma parte de la cache key**.



### 2️⃣ Comprobar si el parámetro se refleja

Probamos valores controlados en la query string:

```http
GET /?test=1
```

En la respuesta observamos que el valor del parámetro aparece reflejado dentro del **HTML** de la página.

Esto indica que el parámetro influye en el contenido generado por el servidor.



### 3️⃣ Romper el contexto HTML

Probamos a modificar el valor del parámetro para intentar salir del contexto donde se refleja.

Petición utilizada:

```http
GET /?test=1'/>
```

El resultado es que:

* El HTML queda **parcialmente roto**.
* Algunos atributos o etiquetas quedan **abiertos**.

Esto confirma que es posible **inyectar contenido arbitrario** en la respuesta.



### 4️⃣ Inyectar el payload XSS

Construimos un payload que rompa el contexto e inserte un script.

Petición final:

```http
GET /?test=1'/><script>alert(1)</script>
```

En la respuesta observamos que:

* Nuestro `<script>` aparece dentro del HTML.
* El contenido se procesa correctamente por el navegador.

Además, vemos en la respuesta:

```http
X-Cache: hit
```

Esto indica que la **respuesta maliciosa ha sido almacenada en caché**.



### 5️⃣ Impacto en la víctima

Durante el tiempo de vida de la caché (aproximadamente **30–35 segundos**):

* La víctima accede a la página principal:

```
/
```

* No envía ningún parámetro en la URL.
* Recibe la **respuesta cacheada envenenada**.

El navegador ejecuta automáticamente:

```html
alert(1)
```

El laboratorio detecta la ejecución del payload y se completa.
