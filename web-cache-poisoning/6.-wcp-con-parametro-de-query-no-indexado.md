# 6. WCP con Parámetro de Query No Indexado

## 🧪 Web Cache Poisoning con parámetro de query no indexado

### 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **Web Cache Poisoning** porque excluye un parámetro concreto de la **clave de caché**, a pesar de que dicho parámetro sí es procesado por el backend y reflejado en la respuesta.

La aplicación genera la respuesta utilizando valores enviados en la URL, pero la infraestructura de caché **no tiene en cuenta todos los parámetros al construir la cache key**.

Un usuario víctima visita regularmente la página de inicio del sitio utilizando **Chrome**. Si logramos almacenar una respuesta manipulada en caché, ese usuario recibirá la versión envenenada de la página.

El objetivo del laboratorio es **envenenar la caché de la página principal para que se ejecute `alert(1)` en el navegador de la víctima**.

***

### 📚 Teoría

Este laboratorio muestra un caso clásico de **Web Cache Poisoning causado por un parámetro de query no indexado**.

La vulnerabilidad aparece cuando existe una diferencia entre:

* Lo que el **backend utiliza para generar la respuesta**
* Lo que el **sistema de caché utiliza para identificar una respuesta almacenada**

En este escenario:

* El backend **procesa ciertos parámetros de la URL** y los refleja en el HTML.
* La caché **ignora algunos de esos parámetros al generar la cache key**.

Esto provoca que una respuesta generada con un valor específico del parámetro pueda **almacenarse en caché y reutilizarse para otras peticiones**, incluso si esas peticiones no incluyen el parámetro.

#### 📌 Identificación del parámetro vulnerable

Herramientas como **Param Miner** permiten descubrir parámetros que:

* No afectan a la cache key.
* Siguen siendo procesados por el backend.

En este laboratorio, el parámetro vulnerable es:

```
utm_content
```

Este parámetro:

* Se refleja en la respuesta HTML.
* No forma parte de la cache key.

Como consecuencia, podemos **inyectar código JavaScript en la respuesta**, hacer que se almacene en caché y provocar su ejecución en navegadores de otros usuarios.

***

### 📝 Práctica

#### 1️⃣ Identificar el parámetro vulnerable

Interceptamos la petición a la página principal y la enviamos a **Burp Repeater**.

El enunciado indica que existe un parámetro que no forma parte de la cache key, por lo que utilizamos **Param Miner** para identificarlo.

<figure><img src="../../../.gitbook/assets/image (1187).png" alt=""><figcaption></figcaption></figure>

Tras ejecutar el análisis, encontramos el parámetro:

```
utm_content
```

<figure><img src="../../../.gitbook/assets/image (1441).png" alt=""><figcaption></figcaption></figure>

***

#### 2️⃣ Confirmar que el parámetro no forma parte de la cache key

Probamos una petición simple utilizando ese parámetro:

```http
GET /?utm_content=1
```

<figure><img src="../../../.gitbook/assets/image (1442).png" alt=""><figcaption></figcaption></figure>

Observamos que:

* El valor `1` aparece reflejado en la respuesta.
* La cabecera `Age` continúa aumentando.
* La respuesta se devuelve con:

```http
X-Cache: hit
```

Esto confirma que `utm_content` **no forma parte de la cache key**.

***

#### 3️⃣ Inyectar el payload XSS

Como el valor del parámetro se refleja en el HTML, intentamos romper el contexto e insertar código JavaScript.

Utilizamos el siguiente payload:

```http
GET /?utm_content=as'/><script>alert(1)</script>
```

<figure><img src="../../../.gitbook/assets/image (1443).png" alt=""><figcaption></figcaption></figure>

Al enviar la petición observamos que:

* El script aparece dentro de la respuesta HTML.
* La página sigue renderizándose correctamente.

Además, la respuesta queda **almacenada en caché**.

***

#### 4️⃣ Envenenar la caché

La respuesta generada con el payload malicioso se guarda en caché para la página principal.

Durante el tiempo de vida de la caché:

* Cualquier usuario que visite la página principal:

```
/
```

* Incluso sin enviar el parámetro `utm_content`

recibirá la **versión cacheada de la página con el payload inyectado**.

***

#### 5️⃣ Ejecución en la víctima

Cuando la víctima visita la página principal:

* Recibe la respuesta cacheada manipulada.
* El navegador ejecuta automáticamente:

```html
alert(1)
```

El laboratorio detecta la ejecución del payload y se completa.
