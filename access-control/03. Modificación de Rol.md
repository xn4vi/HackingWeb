# 🎯 Modificación del rol en el perfil de usuario

## 📄 Descripción del laboratorio

Este laboratorio contiene un **panel de administración** disponible en:

```
/admin
```

Sin embargo, solo los usuarios cuyo campo **`roleid = 2`** pueden acceder a él.

El objetivo es:

* Escalar privilegios hasta **administrador**.
* Acceder al panel `/admin`.
* Eliminar al usuario **carlos**.

Credenciales proporcionadas:

```
wiener:peter
```

 

## 📚 Teoría

La aplicación permite a los usuarios **modificar su perfil**, por ejemplo actualizando el email.\
Esta operación se realiza mediante una **petición JSON enviada al backend**.

El problema es que el servidor **no valida qué campos pueden modificarse**, y acepta **cualquier propiedad incluida en el JSON**.

Durante el análisis se observa un campo sensible:

```
"roleid": 1
```

Este campo define el rol del usuario:

```
1 → usuario normal
2 → administrador
```

Debido a que el backend **no restringe los campos modificables**, el usuario puede incluir propiedades adicionales en la petición y sobrescribir su propio rol.

### 📌 Problemas de seguridad presentes

La vulnerabilidad corresponde a un caso de **Mass Assignment** combinado con **Privilege Escalation**.

La aplicación presenta los siguientes fallos:

* No existe una **lista blanca (whitelist)** de campos permitidos.
* No se validan **permisos en el servidor**.
* No hay separación entre **datos de usuario y atributos de seguridad**.

Esto permite modificar directamente atributos críticos como:

```
roleid
```

 

## 📝 Práctica

### 🎯 Objetivo

Escalar privilegios a **administrador** y eliminar al usuario **carlos**.

 

### 1️⃣ Autenticación inicial

Iniciamos sesión con las credenciales:

```
Usuario: wiener
Contraseña: peter
```

Accedemos a la sección donde se puede **actualizar el perfil o cambiar el email**.

 

### 2️⃣ Interceptar la petición de actualización

Interceptamos la petición con **Burp Suite** al actualizar el email.

El cuerpo de la petición contiene un JSON similar a:

```json
{
  "email": "test@test.com"
}
```

<img width="768" height="294" alt="image" src="https://github.com/user-attachments/assets/e370b5a7-ffdb-49b3-81e4-354919ad71c4" />
<br>

En la respuesta del servidor se observa que el **rol del usuario se incluye explícitamente**, lo que indica que forma parte del modelo interno.

 

### 3️⃣ Manipulación del rol

Modificamos manualmente el JSON añadiendo el campo `roleid`:

```json
{
  "email": "test@test.com",
  "roleid": 2
}
```

Enviamos la petición modificada.

<img width="768" height="301" alt="image" src="https://github.com/user-attachments/assets/d1f1be6b-68ee-4109-b8cc-c23f163b1659" />
<br>

Resultado:

El servidor acepta la modificación **sin ningún error ni validación adicional**.

Nuestro usuario ahora tiene privilegios de administrador.

 

### 4️⃣ Acceso al panel de administración

Accedemos directamente a:

```
/admin
```

Resultado:

* El panel administrativo carga correctamente.
* Nuestro usuario tiene ahora **acceso completo**.

 

### 5️⃣ Eliminación del usuario

Dentro del panel administrativo:

1. Localizamos al usuario **carlos**.
2. Pulsamos **Delete**.

La acción se ejecuta inmediatamente.
