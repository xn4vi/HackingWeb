# 10. Bypass de Control por HTTP

## 🎯 Bypass de control por método HTTP

### 📄 Descripción del laboratorio

Este laboratorio implementa controles de acceso que dependen parcialmente del **método HTTP utilizado en la petición**.

Aunque ciertas acciones administrativas están protegidas, el backend no valida correctamente **todos los métodos HTTP**, lo que permite realizar un bypass del control de acceso.

El objetivo es:

* Iniciar sesión como **wiener:peter**.
* Escalar privilegios hasta administrador.
* Acceder al panel `/admin`.
* Eliminar al usuario **carlos**.

***

### 📚 Teoría

La aplicación intenta restringir acciones administrativas basándose en dos factores:

* **Rol del usuario**
* **Método HTTP utilizado**

La acción sensible es la promoción de usuarios a administrador mediante el endpoint:

```
/admin/upgrade
```

En teoría:

* **POST** → permitido únicamente para administradores.
* **Usuarios normales** → bloqueados.

#### 📌 Problema de seguridad

El fallo ocurre porque el backend **solo valida los permisos cuando la petición utiliza el método POST**.

Si la misma acción se ejecuta mediante **GET**, el control de acceso **no se aplica correctamente**.

Esto provoca que:

* El método HTTP se convierta en la única barrera de seguridad.
* El endpoint procese la petición aunque el usuario no tenga privilegios.

#### 📌 Vulnerabilidad

Este comportamiento introduce un caso de **Broken Access Control** causado por:

* Validación de permisos solo para un método HTTP específico.
* Falta de rechazo de métodos alternativos.
* Lógica de autorización incompleta en el backend.

Como resultado, cualquier usuario autenticado puede escalar privilegios.

***

### 📝 Práctica

#### 🎯 Objetivo

Promocionar a **wiener** a administrador y eliminar al usuario **carlos**.

***

#### 1️⃣ Reconocimiento con cuenta administradora

Iniciamos sesión con una cuenta administradora:

```
administrator:admin
```

Accedemos al panel de administración y realizamos la acción **Upgrade user**.

<figure><img src="../../../.gitbook/assets/image (895).png" alt=""><figcaption></figcaption></figure>

Interceptamos la petición con **Burp Suite** y la enviamos al **Repeater**.

<figure><img src="../../../.gitbook/assets/image (896).png" alt=""><figcaption></figcaption></figure>

Esta acción funciona correctamente cuando se realiza como administrador.

***

#### 2️⃣ Prueba con usuario normal

Cerramos sesión e iniciamos sesión con:

```
wiener:peter
```

Tomamos la **cookie de sesión** de este usuario y la utilizamos en la misma petición interceptada.

Enviamos la petición.

<figure><img src="../../../.gitbook/assets/image (897).png" alt=""><figcaption></figcaption></figure>

Resultado:

```
Unauthorized
```

El backend bloquea correctamente la acción cuando se usa **POST**.

***

#### 3️⃣ Cambio del método HTTP

Ahora modificamos únicamente el método de la petición:

```
POST → GET
```

<figure><img src="../../../.gitbook/assets/image (898).png" alt=""><figcaption></figcaption></figure>

Enviamos la petición.

Resultado:

* La petición es aceptada.
* El servidor ejecuta la acción.
* No se valida el rol del usuario.

El usuario **wiener** ahora tiene privilegios de administrador.

***

#### 4️⃣ Eliminación del usuario carlos

Refrescamos la aplicación y accedemos a:

```
/admin
```

Ahora tenemos acceso al panel administrativo.

Dentro del panel:

1. Localizamos al usuario **carlos**.
2. Pulsamos **Delete**.

El usuario se elimina correctamente.

***

#### 5️⃣ Resultado final

Se consigue:

* Escalar privilegios desde usuario normal a administrador.
* Acceder al panel administrativo.
* Eliminar al usuario **carlos**.

El laboratorio se completa correctamente.
