# 6. ID de Usuario Impredecible

## 🎯 ID del usuario impredecible controlado por parámetro

### 📄 Descripción del laboratorio

Este laboratorio presenta una **escalada de privilegios horizontal** en la página de cuenta de usuario.

A diferencia de casos anteriores, la aplicación identifica a los usuarios mediante **GUIDs (Global Unique Identifiers)** en lugar de nombres de usuario.

El objetivo es:

* Obtener el **GUID del usuario carlos**.
* Usarlo para acceder a su cuenta.
* Recuperar su **API key**.

Credenciales proporcionadas:

```
wiener:peter
```

***

### 📚 Teoría

En este laboratorio, la aplicación identifica a los usuarios mediante un parámetro en la URL:

```
/my-account?id=GUID
```

Los **GUIDs** son identificadores largos que aparentan ser aleatorios, lo que puede dar la impresión de que no pueden ser adivinados.

Sin embargo, el problema no es la aleatoriedad del identificador, sino que **el GUID se filtra en otras partes de la aplicación**.

En este caso:

* Los **posts del blog** muestran el nombre del autor.
* El nombre del autor enlaza a su perfil utilizando directamente su **GUID**.
* Cualquier usuario puede ver y copiar ese identificador.

Una vez que un atacante obtiene el GUID de otro usuario, puede usarlo directamente en la página de cuenta.

El backend:

* No verifica si el usuario autenticado es el propietario del GUID solicitado.
* Devuelve toda la información asociada al usuario solicitado.

Entre los datos expuestos se incluye la **API key**.

#### 📌 Vulnerabilidad

Este comportamiento corresponde a un caso de:

**IDOR (Insecure Direct Object Reference)**\
o **Horizontal Privilege Escalation**.

Los problemas de seguridad presentes son:

* Exposición de identificadores sensibles en contenido público.
* Falta de verificación de permisos por recurso.
* Uso de identificadores controlados por el cliente para recuperar datos privados.

Una lección importante de este laboratorio es que:

**Los identificadores impredecibles no son un mecanismo de control de acceso.**

***

### 📝 Práctica

#### 🎯 Objetivo

Obtener la **API key del usuario carlos**.

***

#### 1️⃣ Iniciar sesión

Accedemos a la aplicación utilizando las credenciales:

```
Usuario: wiener
Contraseña: peter
```

Una vez autenticados, accedemos a **My account**.

***

#### 2️⃣ Identificar el uso de GUID

Observamos la URL de nuestra página de cuenta:

```
/my-account?id=GUID-DE-WIENER
```

Esto confirma que el backend utiliza el parámetro `id` para identificar al usuario.

***

#### 3️⃣ Obtener el GUID de carlos

Navegamos a la sección **Blog** de la aplicación.

Encontramos un post escrito por **carlos**.

Al inspeccionar el enlace asociado a su nombre o perfil, observamos una URL similar a:

```
/blog?userId=GUID-DE-CARLOS
```

o un enlace que incluye directamente:

```
id=GUID-DE-CARLOS
```

<figure><img src="../../../.gitbook/assets/image (880).png" alt=""><figcaption></figcaption></figure>

Copiamos el **GUID completo de carlos**.

***

#### 4️⃣ Manipular el parámetro

Volvemos a nuestra página de cuenta y sustituimos el identificador en la URL:

```
/my-account?id=GUID-DE-CARLOS
```

Recargamos la página.

***

#### 5️⃣ Acceso no autorizado

El servidor responde mostrando:

* Información personal del usuario **carlos**.
* Su **API key**.

Esto confirma que el backend **no verifica si el usuario autenticado tiene permiso para acceder a ese recurso**.

<figure><img src="../../../.gitbook/assets/image (881).png" alt=""><figcaption></figcaption></figure>

***

#### 6️⃣ Resolver el laboratorio

Copiamos la **API key de carlos** y la introducimos en el formulario de solución del laboratorio.

El laboratorio se marca como completado.

***

#### 7️⃣ Resultado final

Se consigue:

* Acceder a la cuenta de otro usuario.
* Explotar un identificador aparentemente impredecible.
* Obtener la **API key de carlos**.

El laboratorio queda resuelto correctamente.
