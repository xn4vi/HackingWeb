# 5. Explotación de Tokens de Reset

## 🧨 Explotación de tokens de reset sensibles al tiempo

### 📄 Descripción del laboratorio

El laboratorio implementa un mecanismo de **restablecimiento de contraseña vulnerable**, donde los tokens se generan de forma predecible.

Aunque no se trata de una race condition clásica de concurrencia en lógica de negocio, el fallo es **temporal y sincronizable**.

El sistema genera tokens que dependen únicamente del tiempo, sin:

* Entropía real
* Dependencia del usuario
* Dependencia del CSRF
* Ningún valor secreto adicional

El objetivo es:

* Obtener un token válido para **carlos**
* Cambiar su contraseña
* Acceder a su cuenta
* Entrar en `/admin`
* Eliminar al usuario carlos

Credenciales:

```
wiener:peter
```

***

### 📚 Teoría

Este laboratorio explota una debilidad en la generación de tokens basada exclusivamente en tiempo.

#### 📌 Flujo normal del reset

1. El usuario solicita un reset de contraseña
2. El servidor genera un token
3. El token se almacena como válido
4. Se envía un email con el enlace
5. El token se invalida tras su uso

#### 📌 El fallo

El token:

* No depende del usuario
* No usa aleatoriedad segura
* No incorpora CSRF
* Depende únicamente del timestamp

Esto implica que:

* Dos peticiones generadas en el mismo instante producen el mismo token
* El token es reutilizable entre usuarios

Resultado:

Un token generado para un usuario puede ser válido para otro si coincide el tiempo de generación.

#### 📌 Impacto

Permite:

* Predecir o colisionar tokens
* Resetear contraseñas de otros usuarios
* Escalar privilegios

***

### 📝 Práctica

#### 1️⃣ Reconocimiento inicial

Iniciamos sesión con:

```
wiener:peter
```

Exploramos la funcionalidad de reset.

Solicitamos un reset para nuestro usuario y observamos:

* Recibimos un email con un token
* Si solicitamos otro, el token cambia
* El anterior deja de ser válido

<figure><img src="../../../.gitbook/assets/image (874).png" alt=""><figcaption></figcaption></figure>

Comportamiento esperado.

***

#### 2️⃣ Interceptar la petición de reset

Interceptamos la petición:

```http
POST /forgot-password
username=wiener
```

La enviamos a Repeater.

Probamos enviarla dos veces seguidas:

* Los tokens son distintos

No hay comportamiento anómalo aún.

***

#### 3️⃣ Intento de concurrencia básico

Duplicamos la petición y creamos un tab group.

Ejecutamos:

```
Send group → in parallel
```

Observamos:

* Las peticiones no llegan exactamente al mismo tiempo
* Los tokens siguen siendo distintos

Esto indica:

* Posible serialización por sesión
* Limitación por cookie o CSRF

***

#### 4️⃣ Romper la serialización

Para conseguir concurrencia real:

* Refrescamos la sesión
* Usamos cookies distintas
* Usamos CSRF distintos

<figure><img src="../../../.gitbook/assets/image (875).png" alt=""><figcaption></figcaption></figure>

Ahora enviamos de nuevo en paralelo.

Resultado:

* Ambas peticiones llegan en el mismo instante
* Los tiempos son prácticamente idénticos

Observamos:

* Ambos correos contienen el mismo token

Confirmamos que el token depende únicamente del tiempo.

***

#### 5️⃣ Explotación contra carlos

Preparamos dos peticiones:

Petición A:

```
username=wiener
```

Petición B:

```
username=carlos
```

Ambas:

* Con cookies distintas
* Con CSRF distintos
* Enviadas en paralelo

<figure><img src="../../../.gitbook/assets/image (876).png" alt=""><figcaption></figcaption></figure>

Resultado:

* Se genera el mismo token para ambas solicitudes
* El token recibido es válido para carlos

Puede requerir:

* Repetir varias veces
* Ajustar timing

***

#### 6️⃣ Reset de contraseña

Con el token obtenido:

* Abrimos el enlace de reset
* Cambiamos la contraseña de carlos

El servidor acepta el token sin validaciones adicionales.

<figure><img src="../../../.gitbook/assets/image (877).png" alt=""><figcaption></figcaption></figure>

***

#### 7️⃣ Acceso y explotación final

Iniciamos sesión como:

```
carlos:nueva_password
```

<figure><img src="../../../.gitbook/assets/image (878).png" alt=""><figcaption></figcaption></figure>

Accedemos a:

```
/admin
```

Eliminamos al usuario carlos.
