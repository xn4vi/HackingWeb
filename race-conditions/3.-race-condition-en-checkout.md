# 3. Race Condition en Checkout

## 🧨 Condición de carrera en múltiples endpoints del checkout

### 📄 Descripción del laboratorio

El laboratorio presenta una **race condition distribuida entre varios endpoints** del flujo de compra.

No existe un único punto vulnerable, sino una falta de sincronización entre distintas operaciones del proceso de checkout.

El objetivo es:

* Comprar la **Lightweight “l33t” Leather Jacket** (≈1300€)
* Teniendo únicamente **100€ de crédito**

Credenciales:

```
wiener:peter
```

***

### 📚 Teoría

En este laboratorio, la vulnerabilidad no reside en un único endpoint, sino en la interacción entre varios.

#### 📌 Flujo normal de compra

1. El usuario añade productos al carrito
2. El servidor valida el contenido del carrito
3. Se comprueba si el usuario tiene crédito suficiente
4. Si la validación es correcta, se ejecuta el pago
5. Se actualiza el saldo del usuario

#### 📌 El fallo

Las operaciones de:

* Añadir productos al carrito
* Realizar el checkout

no están correctamente sincronizadas.

Esto permite una situación en la que:

* El servidor valida el crédito con un carrito “barato”
* El carrito cambia antes de finalizar la compra
* Se completa la compra con un carrito más caro

Resultado:

El servidor cree que el carrito es válido, pero termina procesando uno distinto.

#### 📌 Impacto

Este tipo de vulnerabilidad aparece cuando:

* Las validaciones no son atómicas
* Los endpoints se procesan en paralelo
* No existen locks ni transacciones globales

***

### 📝 Práctica

#### 1️⃣ Reconocimiento inicial

Iniciamos sesión con:

```
wiener:peter
```

Accedemos a nuestra cuenta y comprobamos:

* Crédito disponible: 100€

Observamos que existen:

* Gift cards

Estas pueden añadirse al carrito y afectan al total.

Esto indica que el carrito tiene lógica relevante para explotar.

***

#### 2️⃣ Identificación de endpoints críticos

Interceptamos y enviamos al Repeater las peticiones clave:

* Petición para añadir una gift card al carrito
* Petición de checkout:

```
/cart/checkout
```

Ahora tenemos dos endpoints que interactúan con:

* El contenido del carrito
* El crédito disponible

***

#### 3️⃣ Entender la validación del servidor

Intentamos comprar directamente la chaqueta.

El servidor responde:

```
Not enough store credit for this purchase
```

<figure><img src="../../../.gitbook/assets/image (586).png" alt="" width="284"><figcaption></figcaption></figure>

Esto confirma:

* Existe validación de crédito
* Se ejecuta antes de completar la compra

La clave es:

Qué ocurre si el carrito cambia después de esa validación.

***

#### 4️⃣ Análisis de tiempos

Creamos un **tab group en Burp Repeater**.

Añadimos ambas peticiones:

* add to cart
* checkout

Las enviamos en secuencia dentro del mismo grupo.

<figure><img src="../../../.gitbook/assets/image (587).png" alt="" width="194"><figcaption></figcaption></figure>

Observamos:

* El endpoint de checkout responde más rápido
* El endpoint de añadir al carrito tarda más

Esto indica un desfase temporal explotable.

<figure><img src="../../../.gitbook/assets/image (588).png" alt="" width="65"><figcaption></figcaption></figure>

***

#### 5️⃣ Preparar la condición de carrera

En el carrito dejamos inicialmente:

* Solo una gift card

Modificamos la petición de add to cart para añadir:

* La chaqueta (product ID = 1)

El objetivo es:

* Checkout valida el carrito “barato”
* En paralelo se añade la chaqueta
* El checkout finaliza con el carrito modificado

<figure><img src="../../../.gitbook/assets/image (589).png" alt="" width="266"><figcaption></figcaption></figure>

***

#### 6️⃣ Explotación de la race condition

Creamos un tab group con:

* Petición de checkout
* Petición de add to cart (chaqueta)

Ejecutamos:

```
Send group → in parallel
```

<figure><img src="../../../.gitbook/assets/image (590).png" alt="" width="155"><figcaption></figcaption></figure>

Resultado esperado:

* El servidor valida el crédito cuando el carrito aún es barato
* La chaqueta se añade después
* La compra se completa igualmente

Puede ser necesario:

* Repetir varias veces
* Ajustar el orden de envío
* Lanzar múltiples ejecuciones

Esto es normal en race conditions.

***

#### 7️⃣ Resultado final

Tras varios intentos:

* La chaqueta aparece como comprada
* El saldo no corresponde al precio real

El laboratorio se completa correctamente.
