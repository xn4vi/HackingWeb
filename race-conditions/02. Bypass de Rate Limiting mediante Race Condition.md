# 🧨 Bypass de rate limiting mediante condición de carrera

## 📄 Descripción del laboratorio

El laboratorio implementa un mecanismo de autenticación con **rate limiting** para prevenir ataques de fuerza bruta.

Tras **3 intentos fallidos**, la cuenta se bloquea temporalmente.

Sin embargo, esta protección es vulnerable a una **condición de carrera**, permitiendo enviar múltiples intentos simultáneamente antes de que el contador se actualice.

El objetivo es:

* Evitar el rate limit
* Descubrir la contraseña de **carlos** mediante fuerza bruta
* Iniciar sesión como administrador
* Eliminar la cuenta de **carlos**

Credenciales de prueba:

```
wiener:peter
```



## 📚 Teoría

Los mecanismos de rate limiting basados en contadores simples son vulnerables cuando la lectura y actualización del contador **no son operaciones atómicas**.

### 📌 Flujo normal del login

1. El servidor recibe un intento de autenticación
2. Lee el contador de intentos fallidos
3. Si el contador es < 3, permite el intento
4. Si la contraseña es incorrecta, incrementa el contador
5. Al llegar a 3, bloquea la cuenta

### 📌 El fallo

Si enviamos múltiples peticiones en paralelo:

* Varias leen el mismo valor del contador (por ejemplo, 0)
* Todas pasan la validación
* Ninguna ha incrementado aún el contador

Resultado:

👉 Se procesan múltiples intentos sin activar el bloqueo

### 📌 Impacto

Esto permite:

* Bypassear el rate limit
* Ejecutar fuerza bruta efectiva

Especialmente potente en:

* HTTP/2 (multiplexing)
* Turbo Intruder
* Envío concurrente desde Burp



## 📝 Práctica

### 1️⃣ Confirmar el rate limiting

Iniciamos sesión con:

```
wiener:peter
```

Intentamos autenticarnos como:

```
carlos
```

con contraseñas incorrectas.

Observamos que:

* Tras 3 intentos
* La cuenta se bloquea temporalmente

Confirmado: existe rate limit



### 2️⃣ Detectar la condición de carrera

Interceptamos una petición de login:

```http
POST /login
username=carlos
password=incorrecta
```

La enviamos a **Repeater**.

Creamos un grupo:

```
Add group
```

Duplicamos la petición unas **10 veces**.

Ejecutamos:

```
Send group → in parallel
```
