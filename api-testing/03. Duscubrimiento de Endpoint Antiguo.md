# 🌐 Descubrimiento y explotación de un endpoint no usado

## 📄 Descripción del laboratorio

El laboratorio oculta un **endpoint API no utilizado por la interfaz web**, pero activo y accesible en el backend.

Este endpoint permite **modificar atributos de productos** mediante el método **PATCH**, incluyendo el precio, sin controles de autorización adecuados.

El objetivo es:

* Descubrir el **endpoint oculto**.
* Modificar el precio de la **Lightweight "l33t" Leather Jacket** a **0 €**.
* Completar la compra gratuitamente.

Credenciales de prueba:

* **wiener : peter**

 

## 📚 Teoría

Muchas aplicaciones mantienen **endpoints API internos o heredados** que:

* No están documentados.
* No se utilizan desde la interfaz pública.
* Carecen de controles de autorización robustos.

Estos endpoints suelen:

* Permanecer **accesibles directamente**.
* Aceptar **métodos HTTP adicionales** como `PUT` o `PATCH`.
* Ser detectables mediante el método **OPTIONS**.

### 📌 Método OPTIONS

El método **OPTIONS** permite consultar qué métodos HTTP están permitidos para un recurso. Esta información se devuelve normalmente en la cabecera:

```
Allow
```

Si un endpoint acepta **PATCH** o **PUT**, es una señal clara de que permite modificaciones.

En este laboratorio:

```
/api/products/{id}
```

acepta **PATCH**, permite modificar el campo **price** y no valida si el usuario tiene permiso para hacerlo.

Esto permite **manipulación de precios y compras fraudulentas**.

 

## 📝 Práctica

### 🎯 Objetivo

Reducir el precio de la **Lightweight "l33t" Leather Jacket** a **0 €** y completar la compra.

 

### 1️⃣ Análisis inicial

Se inicia sesión con **wiener:peter**.

Se navega por la tienda y se añade la **Lightweight "l33t" Leather Jacket** al carrito.

Mientras tanto, se observan las peticiones en **Burp Suite**.

 

### 2️⃣ Identificación del endpoint interesante

Se detecta la siguiente petición:

```http
GET /api/products/1/price
```

<img width="768" height="266" alt="image" src="https://github.com/user-attachments/assets/a462301b-0c91-4ef1-937c-b6e267f2e04a" />
<br>

Esto indica que existe un **endpoint API para consultar el precio** del producto.

Es probable que el endpoint admita **más operaciones además de GET**.

 

### 3️⃣ Descubrimiento de métodos permitidos

Se envía la petición al **Repeater** y se cambia el método a **OPTIONS**.

> El método HTTP **OPTIONS** se utiliza para consultar qué métodos y opciones de comunicación están permitidos para un recurso o servidor específico.

<img width="768" height="297" alt="image" src="https://github.com/user-attachments/assets/26bb4b41-d52c-4b8b-87d4-3633f26dc066" />
<br>

Resultado:

La respuesta incluye la cabecera:

```http
Allow: GET, PATCH
```

Esto confirma que el endpoint acepta **PATCH**.

 

### 4️⃣ Modificación del precio

Se prueba una petición **PATCH** con los siguientes datos.

Header:

```http
Content-Type: application/json
```

Body:

```json
{
  "price": 0
}
```

<br>

Resultado:

El servidor responde con **200 OK** y no aparecen errores de autorización.

Al refrescar la página del producto se observa que el **precio ahora es 0 €**.

### 5️⃣ Compra del producto

Se añade la chaqueta al carrito.

Se procede al **checkout** y se finaliza la compra sin problemas.

La compra se completa **gratuitamente**.

### 6️⃣ Resultado final

Se descubre un **endpoint API oculto**.

Se explota la **falta de controles de autorización en el método PATCH**, lo que permite modificar el precio del producto.

El precio se cambia a **0 €** y se completa la compra sin coste.

El laboratorio se marca como **resuelto correctamente**.
