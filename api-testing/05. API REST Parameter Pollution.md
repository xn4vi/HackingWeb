# 🌐 Parameter pollution en URL REST del servidor

## 📄 Descripción del laboratorio

El laboratorio explota una vulnerabilidad de **Server-Side Parameter Pollution (SSPP)** combinada con **path traversal** en un parámetro utilizado para construir rutas REST internas.

La aplicación:

* Concatena un parámetro controlado por el usuario (por ejemplo `username`).
* Lo utiliza directamente para formar **rutas internas del backend**.
* No aplica sanitización ni validación adecuada.

El objetivo es:

* Obtener el **token de reseteo de contraseña** del usuario **administrator**.
* Cambiar su contraseña.
* Eliminar la cuenta de **carlos**.

 

## 📚 Teoría

En APIs REST mal diseñadas es común encontrar patrones como:

```
/api/.../users/{username}/...
```

Si el valor de `{username}`:

* Proviene directamente del usuario.
* Se concatena a una ruta base.
* No se valida adecuadamente.

Se abren dos vectores críticos.

### 📌 Path Traversal

Utilizando secuencias como:

```
../
```

o sus variantes codificadas, un atacante puede:

* Escapar del path esperado.
* Acceder a rutas superiores.
* Alcanzar recursos internos no expuestos.

### 📌 Descubrimiento de documentación interna

Mediante **path traversal** es común encontrar archivos como:

* `openapi.json`
* `swagger.json`
* Documentación de APIs internas o versiones antiguas.

Al combinar ambos vectores:

* Se descubre el **esquema interno de la API**.
* Se identifican **endpoints sensibles**.
* Se extraen **datos privilegiados** como tokens o flags internos.

Este fallo demuestra lo peligroso que resulta **construir rutas dinámicas utilizando entrada no validada**.

 

## 📝 Práctica

### 🎯 Objetivo

Autenticarse como **administrator** y eliminar la cuenta del usuario **carlos**.

 

### 1️⃣ Análisis inicial

Se accede a la funcionalidad **Forgot password** e introducimos:

```
administrator
```

Se interceptan las peticiones relevantes:

```http
POST /forgot-password
GET /static/js/forgotPassword.js
```

Ambas se envían al **Repeater** para analizarlas.

### 2️⃣ Pruebas iniciales de parameter pollution

Se prueba contaminación básica del parámetro `username`.

![](../images/65.png)
<br>
Resultado:

El backend **no acepta parámetros adicionales simples**.

Se prueban delimitadores como `#` y `;`.

![](../images/66.png)<br>

Resultado:

Tampoco funcionan directamente.

 

### 3️⃣ Introducción de path traversal

Se prueba **path traversal** en el parámetro `username`.

![](../images/67.png)<br>

Resultado:

El servidor devuelve una **respuesta diferente**, lo que indica que el traversal sí está siendo interpretado.

Esto confirma que el parámetro se **concatena directamente en una ruta REST interna**.

 

### 4️⃣ Descubrimiento de documentación interna

Se lanza un ataque con **Burp Intruder** utilizando una lista de rutas comunes de documentación de API, por ejemplo:

* `openapi.json`
* `swagger.json`
* `api-docs`
* `v1/openapi.json`

![](../images/68.png)
<br>
Resultado:

Se encuentra una **ruta válida** que devuelve el **esquema completo de una API interna**.

![](../images/69.png)

 

### 5️⃣ Identificación del endpoint sensible

En la documentación descubierta aparece el endpoint:

```http
/api/internal/v1/users/{username}/field/{field}
```

![](../images/70.png)<br>

Este endpoint permite:

* Consultar **campos internos arbitrarios**.
* Para **cualquier usuario**.

 

### 6️⃣ Explotación del endpoint interno

Se prueba acceder al endpoint manipulando el parámetro `username`.

![](../images/71.png)<br>

El endpoint funciona y devuelve **datos del usuario**.

Sabemos que en el flujo normal de reseteo de contraseña se utiliza el campo:

```
passwordResetToken
```

![](../images/72.png)<br>

Se intenta extraerlo directamente.

![](../images/73.png)<br>
Resultado:

El servidor responde:

```
This API version does not support this field
```

 

### 7️⃣ Cambio de versión de la API

La documentación revela la existencia de una **versión anterior de la API**.

Se retrocede un nivel en la ruta utilizando la versión anterior.

![](../images/74.png)<br>
Resultado:

El endpoint devuelve el **passwordResetToken del usuario administrator**.

 

### 8️⃣ Toma de control del administrador

Se construye la URL de reseteo utilizando el token obtenido:

```
/forgot-password/reset?passwordResetToken=VALOR_OBTENIDO
```

![](../images/75.png)<br>
Se cambia la contraseña del usuario **administrator**.

Posteriormente se inicia sesión con las nuevas credenciales.

 

### 9️⃣ Eliminación de carlos

Una vez autenticados como **administrator**:

* Se accede al **panel administrativo**.
* Se elimina la cuenta del usuario **carlos**.

![](../images/76.png)
 

### 🔟 Resultado final

Se explota una vulnerabilidad de **SSPP en rutas REST**.

Mediante **path traversal** se descubre documentación interna de la API y se identifican endpoints sensibles.

Se obtiene el **reset token del usuario administrator**, se toma control de su cuenta y finalmente se elimina el usuario **carlos**.

El laboratorio se resuelve correctamente.
