# 🌐 Explotación de Mass Assignment

## 📄 Descripción del laboratorio

El laboratorio presenta una vulnerabilidad de **Mass Assignment** en el endpoint de checkout de la API.

La aplicación asigna automáticamente las propiedades del **JSON recibido** a los campos internos del objeto pedido, incluyendo atributos sensibles que no están expuestos en la interfaz de usuario.

El objetivo es:

* Manipular un **parámetro interno del pedido**.
* Aplicar un **descuento del 100%**.
* Comprar la **Lightweight "l33t" Leather Jacket** por **0 €**.

Credenciales de prueba:

* **wiener : peter**

 

## 📚 Teoría

**Mass Assignment** ocurre cuando un framework o librería backend:

* Mapea automáticamente todos los campos de un **JSON**.
* Los asigna directamente a un **modelo interno**.

Esto ocurre sin utilizar una **lista blanca (allowlist) de campos permitidos**.

Frameworks comúnmente afectados:

* Express + body-parser
* Ruby on Rails
* Django REST Framework
* Laravel

### 📌 Por qué es peligroso

Si el backend:

* No filtra los **campos permitidos**.
* Confía en que el cliente solo enviará los **parámetros esperados**.

Un atacante puede **inyectar o sobrescribir atributos sensibles**, como por ejemplo:

* Descuentos
* Precios
* Roles
* Saldos
* Flags internos

En este laboratorio:

El endpoint:

```
/api/checkout
```

acepta un **JSON**, realiza **binding automático** y reconoce un campo oculto llamado **chosen\_discount**.

 

## 📝 Práctica

### 🎯 Objetivo

Aplicar un **descuento del 100%** inyectando el parámetro oculto **chosen\_discount**.

 

### 1️⃣ Análisis inicial

Se inicia sesión con **wiener:peter**.

Se añade la **Lightweight "l33t" Leather Jacket** al carrito y se procede al **checkout**.

Las peticiones se interceptan utilizando **Burp Suite**.

 

### 2️⃣ Análisis de las peticiones al checkout

Se observan dos peticiones relevantes hacia:

```
/api/checkout
```

* **GET** → devuelve el estado actual del pedido.
* **POST** → confirma la compra.

Ambas peticiones se envían al **Repeater** para analizarlas.

### 3️⃣ Identificación del campo oculto

En la respuesta del **GET** se observa una propiedad interesante:

```json
"chosen_discount": {
  "percentage": 0
}
```

<img width="587" height="409" alt="image" src="https://github.com/user-attachments/assets/b6695ba5-5cd0-426c-8230-019704bf7755" />
<br>
Observaciones:

* El campo **no aparece en la petición POST normal**.
* El backend **lo reconoce y lo devuelve**.

Esto es un indicador claro de **Mass Assignment**.

 

### 4️⃣ Inyección del parámetro sensible

Se modifica manualmente la petición **POST** añadiendo al cuerpo JSON:

```json
"chosen_discount": {
  "percentage": 100
}
```

<img width="768" height="355" alt="image" src="https://github.com/user-attachments/assets/cec823f2-1bf7-4946-8afa-0cadfb2d223b" />
<br>
Resultado:

El servidor responde con **200 OK** y el pedido se procesa con el descuento aplicado.

Esto confirma que el backend **asigna todos los campos recibidos** y que **no existe validación de atributos permitidos**.

{% hint style="info" %}
El ataque de **Mass Assignment** se refiere a la manipulación de parámetros de entrada en solicitudes HTTP para crear o modificar campos en un objeto de modelo de datos en una aplicación web. Los atacantes intentan explotar la funcionalidad de los parámetros existentes para modificar campos que no deberían ser accesibles para el usuario.
{% endhint %}

 

### 5️⃣ Confirmación de la compra

Se refresca la aplicación y se confirma que la compra se ha completado.

El **precio final es 0 €** gracias al **descuento del 100%**.

Se identifica una vulnerabilidad de **Mass Assignment**.

Se inyecta un **campo interno no expuesto** y se aplica un **descuento del 100%**, lo que permite completar la compra gratuitamente.
