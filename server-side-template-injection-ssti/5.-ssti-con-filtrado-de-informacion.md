# 5. SSTI con Filtrado de Información

## 💉 SSTI con filtrado de información vía objetos del usuario (Django)

### 📄 Descripción del laboratorio

La aplicación permite a usuarios con rol **content-manager** editar directamente la plantilla utilizada para visualizar posts.

El contenido introducido se procesa como una **plantilla Django real**, sin ningún tipo de sanitización ni escaping.

Esto introduce una vulnerabilidad de **Server-Side Template Injection (SSTI)** que no busca ejecución remota de código, sino **filtrado de información sensible desde objetos internos del framework**.

El objetivo es:

* Explotar **SSTI en el motor de plantillas Django**.
* Enumerar los **objetos disponibles en el contexto**.
* Extraer la **SECRET\_KEY** del objeto `settings`.

***

### 📚 Teoría

Django utiliza su propio **motor de plantillas**, cuya sintaxis se basa en:

Variables:

```python
{{ variable }}
```

Tags:

```python
{% tag %}
```

Aunque Django limita la ejecución de código arbitrario y no permite llegar fácilmente a **RCE**, sigue siendo peligroso evaluar **input del usuario sin control**, ya que muchas variables internas se exponen dentro del contexto de la plantilla.

Entre los objetos que pueden aparecer en el contexto se encuentran:

* `request`
* `user`
* `perms`
* `messages`
* `settings` (configuración global de Django)

#### 📌 El tag `{% debug %}`

El tag:

```python
{% debug %}
```

realiza un **volcado completo del contexto de la plantilla**.

Esto permite ver:

* Todos los **objetos disponibles**.
* Sus **atributos internos**.

Entre la información expuesta pueden aparecer:

* Variables de entorno
* Configuración del sistema
* **SECRET\_KEY**

#### 📌 Impacto de filtrar la SECRET\_KEY

La **SECRET\_KEY de Django** se utiliza para:

* Firmar **cookies de sesión**.
* Generar **tokens CSRF**.
* Crear **tokens de reseteo de contraseña**.
* Firmar **datos criptográficos internos**.

Si esta clave se filtra, un atacante puede:

* Forjar sesiones.
* Bypassear autenticación.
* Manipular tokens.
* Comprometer completamente la aplicación.

***

### 📝 Práctica

#### 🎯 Objetivo

Extraer la **SECRET\_KEY** de Django explotando **SSTI**.

***

#### 1️⃣ Acceso como content-manager

Se inicia sesión con las credenciales:

```
content-manager:C0nt3ntM4n4g3r
```

Se accede a la sección de **posts** y se localiza la opción para **editar la plantilla del post**.

El contenido introducido se refleja directamente al renderizar el post, lo que indica un posible punto de **SSTI**.

***

#### 2️⃣ Identificación del motor

Se elimina todo el contenido de la plantilla y se guarda.

Al cargar la vista del post aparece un **traceback detallado** con información como:

```
TemplateSyntaxError
Django Version: X.Y.Z
```

Esto confirma que el motor de plantillas utilizado es **Django**.

<figure><img src="../../../.gitbook/assets/image (438).png" alt=""><figcaption></figcaption></figure>

***

#### 3️⃣ Enumeración del contexto con `{% debug %}`

Se modifica la plantilla introduciendo:

```python
{% debug %}
```

Se guarda la plantilla y se recarga el post.

<figure><img src="../../../.gitbook/assets/image (439).png" alt=""><figcaption></figcaption></figure>

Resultado:

Se muestra un **volcado completo del contexto de la plantilla**, incluyendo objetos como:

* `request`
* `user`
* `perms`
* `messages`
* `settings`

Esto confirma que es posible acceder directamente a la **configuración global de Django**.

***

#### 4️⃣ Exfiltración de la SECRET\_KEY

Se sustituye el contenido de la plantilla por:

```python
{{ settings.SECRET_KEY }}
```

Se guarda la plantilla y se recarga la vista del post.

<figure><img src="../../../.gitbook/assets/image (440).png" alt=""><figcaption></figcaption></figure>

Resultado:

La página muestra directamente el valor de la **SECRET\_KEY de Django**.

***

#### 5️⃣ Resultado final

La **SECRET\_KEY** del framework Django se filtra correctamente mediante **Server-Side Template Injection**.

El laboratorio detecta la extracción de la clave secreta y se marca como **resuelto automáticamente**.
