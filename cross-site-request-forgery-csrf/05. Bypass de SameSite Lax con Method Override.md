# 🔑 Bypass de SameSite Lax con method override

## 📄 Descripción del laboratorio

La funcionalidad de cambio de correo electrónico es vulnerable a **CSRF**. La aplicación protege sus cookies de sesión utilizando **SameSite=Lax**, pero permite un bypass mediante **method override**.

El objetivo es:

* Realizar un **ataque CSRF** que cambie el correo de la víctima.
* Aprovechar una **navegación top-level GET**.
* Utilizar el **Exploit Server** para alojar el ataque.

Credenciales proporcionadas:

* **wiener : peter**



## 📚 Teoría

En este laboratorio, las cookies de sesión están protegidas con **SameSite=Lax**, que es el comportamiento por defecto en los navegadores modernos.

### 📌 ¿Qué permite SameSite=Lax?

* Envío de cookies en **navegaciones top-level GET**, como clics en enlaces o redirecciones.

### 📌 ¿Qué bloquea?

* Peticiones **cross-site POST**.
* **Subrequests** como `iframe`, `fetch`, `XHR`, entre otros.

### 📌 Fallo clave: method override

El endpoint de cambio de correo acepta **method override**, lo que permite realizar peticiones como:

```http
GET /endpoint?_method=POST
```

El servidor:

* Recibe una petición **GET**.
* Interpreta `_method=POST`.
* Ejecuta la acción como si fuera un **POST legítimo**.

### 📌 Bypass completo

El ataque funciona de la siguiente forma:

1. Se fuerza una **navegación top-level GET**.
2. El navegador envía la **cookie de sesión**, ya que SameSite=Lax lo permite.
3. El servidor interpreta la petición como **POST** gracias a `_method=POST`.
4. Se ejecuta una **acción sensible sin token CSRF**.

SameSite=Lax no protege frente a **navegaciones GET**, y el backend rompe la separación entre métodos.

Se inicia sesión con **wiener:peter** y se accede a **My account**, donde se encuentra la funcionalidad de cambio de correo electrónico.

### 2️⃣ Interceptación de la petición legítima

Se intercepta la solicitud **POST** generada al cambiar el correo electrónico.

* **URL:** `/my-account/change-email`
* **Método:** `POST`
* **Parámetros:** `email=nuevo@correo.com`

<br>
**Observaciones**

* No hay **token CSRF**.
* No hay **defensas adicionales visibles**.



### 3️⃣ Pruebas con GET y override

Primero se prueba una petición **GET** directa:

```
/my-account/change-email?email=test@ejemplo.com
```

<img width="1076" height="571" alt="image" src="https://github.com/user-attachments/assets/0e9d6442-89ad-413a-92d1-1e6efe725238" />
<br>
Resultado:

El servidor rechaza la petición, ya que el método **GET no está permitido**.

A continuación se añade el **method override**:

```
/my-account/change-email?email=test@ejemplo.com&_method=POST
```

<img width="988" height="608" alt="image" src="https://github.com/user-attachments/assets/c92a9cc6-0d0a-44ee-9a2b-2bc6f21da73e" />
<br>
Resultado:

La petición se acepta y el correo se cambia correctamente.

Esto confirma que:

* El backend **permite method override**.
* Una petición **GET puede convertirse en POST**.



### 4️⃣ Construcción del exploit

Se construye el exploit en el **Exploit Server**.

Pasos:

1. Forzar una **redirección top-level**.
2. Incluir el correo deseado.
3. Añadir el parámetro `_method=POST`.

El exploit final es el siguiente:

```javascript
<script>
    location = "https://ID-LABORATORIO.web-security-academy.net/my-account/change-email?email=hacked@hacked.es&_method=POST";
</script>
```

El código se pega en el **Exploit Server**, se pulsa **Store** y posteriormente **Deliver exploit to victim**.

### 5️⃣ Resultado final

El laboratorio se resuelve correctamente.

Cuando el administrador carga la página maliciosa:

* El navegador realiza una **navegación top-level GET**.
* La **cookie de sesión** se envía, ya que SameSite=Lax lo permite.
* El servidor interpreta `_method=POST`.
* El correo del administrador se cambia a **hacked@hacked.es**.

Todo ocurre **sin token CSRF ni interacción adicional del usuario**.
