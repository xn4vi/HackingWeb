# 🔑 CSRF con token no vinculado a la sesión

## 📄 Descripción del laboratorio

La funcionalidad de cambio de correo electrónico es vulnerable a **CSRF**. La aplicación utiliza tokens CSRF como mecanismo de defensa, pero estos no están integrados correctamente con el sistema de gestión de sesiones.

El objetivo del laboratorio es:

* Alojar una página maliciosa en el **Exploit Server**.
* Ejecutar un ataque CSRF reutilizando un token válido.
* Aprovechar que los tokens no están asociados a la sesión del usuario.

Cuentas proporcionadas:

* **wiener : peter**
* **carlos : montoya**



## 📚 Teoría

En este laboratorio la aplicación implementa un token CSRF, pero comete un error crítico: el token no está vinculado a la sesión del usuario.

Esto implica que cualquier token válido generado por el servidor puede ser reutilizado por cualquier usuario autenticado, sin importar quién lo haya recibido originalmente.

### 📌 Bypass del mecanismo

El ataque se basa en una lógica sencilla:

1. El atacante inicia sesión con su propia cuenta.
2. Captura un token CSRF válido desde la funcionalidad de cambio de correo.
3. Reutiliza ese mismo token en una petición **POST** desde otra cuenta (la víctima).
4. El servidor acepta el token porque no lo correlaciona con la sesión.

### 📌 Impacto

Esta defensa se convierte en una ilusión de seguridad:

* No protege realmente contra CSRF.
* Solo obliga al atacante a obtener cualquier token válido.

Esto resulta trivial si el atacante dispone de una cuenta propia. En un entorno real, este fallo suele derivar en **account takeover**, permitiendo cambiar el correo electrónico (y posteriormente la contraseña) de usuarios privilegiados.



## 📝 Práctica

### 1️⃣ Análisis inicial

Se inicia sesión con **wiener:peter** y se accede a **My account**, donde se encuentra la funcionalidad de cambio de correo electrónico.

### 2️⃣ Interceptación de la petición legítima

Al cambiar el correo electrónico, se intercepta la solicitud **POST**.

* **URL:** `/my-account/change-email`
* **Método:** `POST`
* **Parámetros:** `email=nuevo@correo.com` + `csrf=TOKEN`

<br>
**Observaciones**

* Existe un token CSRF.
* Los bypass anteriores (eliminar el token o cambiar el método a GET) no funcionan.
* La validación del token es obligatoria.



### 3️⃣ Reutilización del token entre sesiones

El laboratorio proporciona dos cuentas:

* **wiener:peter**
* **carlos:montoya**

La clave es que tenemos acceso legítimo a ambas.

Pasos del bypass:

1. Iniciar sesión con **wiener:peter**.
2. Generar una petición de cambio de correo y capturar el token CSRF.
3. Pulsar **Drop** en Burp para que el token:
   * No se consuma.
   * Siga siendo válido.
4. Iniciar sesión con **carlos:montoya** (víctima).
5. Modificar una petición **POST** de cambio de correo.
6. Sustituir el token CSRF por el capturado con **wiener**.

**Resultado**

La petición se acepta correctamente y el correo de Carlos se cambia utilizando un token que no le pertenecía.

<br>
Esto confirma que **los tokens CSRF no están vinculados a la sesión**.



### 4️⃣ Construcción del exploit

Para atacar a la víctima real (el administrador) se necesita un token CSRF válido y no consumido.

Como atacante con una cuenta propia, es posible generar tokens indefinidamente.

El exploit se construye en el **Exploit Server**:

1. Crear un formulario **POST**.
2. Indicar el correo deseado (`hacked@hacked.es`).
3. Incluir un token CSRF válido obtenido con nuestra cuenta.
4. Autoenviar el formulario.

El exploit final es el siguiente:

```html
<form action="https://ID-LABORATORIO.web-security-academy.net/my-account/change-email" method="POST">
    <input type="hidden" name="email" value="hacked@hacked.es">
    <input type="hidden" name="csrf" value="ycQbZSL3ocg0gOSh2pKfqoXLaBXN2hsJ">
</form>
<script>
    document.forms[0].submit();
</script>
```

El código se pega en el **Exploit Server**, se pulsa **Store** y posteriormente **Deliver exploit to victim**.

### 5️⃣ Resultado final

El laboratorio se resuelve correctamente.

Cuando el administrador carga la página maliciosa:

* El navegador envía sus cookies de sesión.
* Se envía la petición **POST** con un token CSRF válido.
* El servidor acepta el token sin comprobar la sesión.

Como resultado, el correo del administrador se cambia a **hacked@hacked.es**
