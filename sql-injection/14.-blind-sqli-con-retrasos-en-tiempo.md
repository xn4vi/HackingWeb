# 14. Blind SQLI con Retrasos en Tiempo

## 💉 Inyección SQL ciega con retrasos en el tiempo

### 📄 Descripción del laboratorio

Este laboratorio contiene una **inyección SQL ciega basada en tiempo (Time-based Blind SQL Injection)**.\
La aplicación utiliza una cookie llamada `TrackingId` para realizar análisis internos, y su valor se inserta directamente en una consulta SQL.

En este escenario:

* Los resultados de la consulta **no se devuelven en la respuesta**.
* La página **no cambia según el resultado de la consulta**.
* **No se muestran errores SQL**.
* La consulta se ejecuta **de forma síncrona**.

El objetivo es demostrar la vulnerabilidad provocando un **retraso de 10 segundos** en la respuesta del servidor mediante SQL Injection.

***

### 📚 Teoría

#### 📌 Blind SQL Injection basada en tiempo

En algunos escenarios de SQL Injection no existe ningún canal de información directo:

* No hay contenido reflejado.
* No aparecen errores del motor de base de datos.
* La respuesta HTTP es siempre idéntica.

Sin embargo, si la consulta SQL se ejecuta de forma **síncrona**, el servidor no enviará la respuesta hasta que la consulta haya terminado.

Esto permite utilizar **funciones de retardo del motor de base de datos** para crear un canal lateral basado en tiempo.

Las funciones más comunes son:

📌 **PostgreSQL**

```
pg_sleep(segundos)
```

📌 **MySQL**

```
SLEEP(segundos)
```

📌 **Microsoft SQL Server**

```
WAITFOR DELAY 'hh:mm:ss'
```

Si el código SQL inyectado ejecuta una de estas funciones, el retraso aparecerá directamente en el tiempo de respuesta HTTP.

Esto permite:

* Confirmar la existencia de SQL Injection.
* Construir condiciones booleanas basadas en tiempo.
* Extraer datos carácter por carácter midiendo retrasos.

***

### 📝 Práctica

#### 1️⃣ Confirmar comportamiento ciego

Interceptamos una petición HTTP y modificamos la cookie vulnerable:

```
TrackingId=abc'
```

Reenviamos la petición.

Resultado:

* La página carga normalmente.
* No aparece ningún error.
* El contenido de la respuesta no cambia.
* El tiempo de respuesta es normal.

Esto indica que estamos ante una **inyección SQL ciega sin errores ni respuestas condicionales visibles**.

***

#### 2️⃣ Probar un retardo genérico para MySQL

Probamos a introducir un retardo utilizando la función típica de MySQL.

```
TrackingId=abc'; SLEEP(10)--
```

Reenviamos la petición.

<figure><img src="../../../.gitbook/assets/image (335).png" alt=""><figcaption></figcaption></figure>

Resultado:

La respuesta se recibe inmediatamente, sin retraso.

Conclusión:

La función `SLEEP()` no es válida en este contexto, por lo que el motor de base de datos probablemente **no es MySQL**.

***

#### 3️⃣ Probar un retardo para PostgreSQL

Probamos ahora con la función de retardo propia de PostgreSQL.

```
TrackingId=abc'; SELECT pg_sleep(10)--
```

Reenviamos la petición.

<figure><img src="../../../.gitbook/assets/image (336).png" alt=""><figcaption></figcaption></figure>

Resultado:

* La respuesta tarda aproximadamente **10 segundos** en llegar.
* El contenido de la página sigue siendo el mismo.

Esto confirma que el código SQL inyectado se está ejecutando en la base de datos.

<figure><img src="../../../.gitbook/assets/image (337).png" alt=""><figcaption></figcaption></figure>

***

#### 4️⃣ Resultado

Mediante un ataque de **SQL Injection basada en tiempo** se ha conseguido:

* Confirmar la vulnerabilidad en la cookie `TrackingId`.
* Identificar que el backend utiliza **PostgreSQL**.
* Ejecutar código SQL controlado por el atacante.
* Provocar un retraso de **10 segundos** en la respuesta del servidor.

✅ El laboratorio detecta el retraso y se marca como completado.
